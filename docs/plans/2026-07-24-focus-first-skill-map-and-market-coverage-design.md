# 포커스 중심 스킬맵과 채용시장 커버리지 개선 설계

- 상태: 사용자 승인
- 승인일: 2026-07-24
- 대상: `/skills/graph`, `/market`, 기업 수집원 및 공개 수집 상태
- 디자인 시스템: 루트 `design.md`의 modern-minimal 제품 워크벤치

## 1. 배경

운영 중인 스킬맵은 상위 스킬 30개를 대상으로 공고 안에서 함께 등장한 모든 스킬 쌍을 연결한다. 2026-07-24 운영 API 응답은 스킬 노드 30개, 스킬 간 연결선 344개, 근거 공고 291개였다. 근거 공고가 가진 스킬을 모두 연결하면 기본 화면이 총 321개 노드와 3,286개 선을 그리게 된다.

스킬 간 가능한 연결은 435개이고 이 중 약 79%가 실제 선으로 표시된다. 평균 스킬 차수는 약 22.9이며 `LLM`은 다른 29개 스킬과 모두 연결된다. 이 데이터에서는 현재 기본값인 로컬 깊이 2가 사실상 전체 그래프와 같아진다.

채용시장 데이터는 같은 날 기준 196개 기업, 공개 공고 1,783건을 수집한다. 전체 범위는 충분히 넓지만, 토스 커뮤니티 141건·메가존클라우드 그룹 96건·쿠팡 77건처럼 한 기업의 공고량이 기술 순위를 크게 움직일 수 있다. 또한 공개 소스 상태는 실행 가능 여부만 `수집 중`으로 표현하여, 정상적으로 공고가 0건인 기업과 장기간 수집되지 않은 기업을 사용자가 구분할 수 없다.

## 2. 목표

1. 첫 화면에서 3초 안에 현재 시장의 핵심 기술과 강한 관계를 파악할 수 있게 한다.
2. 스킬을 선택하면 다음에 학습할 후보와 그 판단 근거 공고를 바로 확인할 수 있게 한다.
3. 저사양 PC와 모바일에서 불필요한 노드·선·계속되는 시뮬레이션을 제거한다.
4. 채용시장 순위가 대량 공고를 내는 소수 기업에만 좌우되지 않도록 기업 확산도를 함께 제공한다.
5. 공식 채용원이 분명한 고가치 기술 기업의 누락을 보완한다.
6. 공개 수집 상태를 `채용 없음`과 `점검 필요`까지 구분해 신뢰도를 높인다.

## 3. 비목표

- 모든 스킬 관계를 기본 화면에 동시에 노출하지 않는다.
- 그래프 안의 작은 공고 점을 정보의 주된 근거 표현으로 사용하지 않는다.
- `좋은 기업`을 하나의 주관적 점수로 단정하지 않는다.
- 공식 채용원과 반복 수집 가능성을 확인하지 못한 기업을 숫자를 늘리기 위해 추가하지 않는다.
- 이번 작업에서 임의의 분야별 군집 분류 모델을 새로 만들지 않는다.

## 4. 스킬맵 정보 구조

### 4.1 기본 상태: 시장 핵심 보기

- 명시적인 `seed`가 없는 첫 진입에서는 수요 상위 스킬 최대 12개만 표시한다.
- 표시된 노드 사이에서 최대 신장 트리를 먼저 선택해 고립을 막는다.
- 남은 관계를 점수순으로 추가하되 전체 선은 최대 18개로 제한한다.
- 근거 공고 노드는 표시하지 않는다.
- 기본 상태에서 임의의 첫 노드를 자동 선택하지 않는다. 사용자가 지도를 시장 개요로 먼저 읽을 수 있어야 한다.

### 4.2 선택 상태: 관련 기술 보기

- 사용자가 스킬을 선택하거나 URL에 `seed`가 있으면 선택 스킬과 직접 연결된 상위 8개 이웃만 표시한다.
- 우선순위는 관계 점수, 함께 등장한 공고 수, 이웃 스킬 수요, 이름 순의 안정적인 정렬로 결정한다.
- 선택 스킬에서 이웃으로 향하는 선을 먼저 표시하고, 문맥에 꼭 필요한 이웃 간 관계를 추가하되 총 12개를 넘지 않는다.
- 현재의 `2단계 관계` 기본값은 제거한다. 전체 탐색은 별도 보기에서만 제공한다.
- 다른 스킬을 선택하면 같은 캔버스 안에서 즉시 포커스가 전환되고 URL이 갱신된다.

### 4.3 전체 보기

- `전체 기술 보기`는 보조 동작으로 제공한다.
- 노드는 최대 30개를 유지하되 모든 관계를 복구하지 않는다.
- 최대 신장 트리와 각 노드의 최상위 관계를 합친 희소 백본만 표시하며 선은 최대 45개로 제한한다.
- 전체 보기는 구조 탐색용이고 정확한 관계 개수는 상세 패널이 담당한다.

### 4.4 시각 인코딩

한 시각 속성에는 한 가지 의미만 부여한다.

- 노드 크기: 현재 공개 공고에서의 수요. 최소·최대 반지름 범위를 압축해 큰 노드가 주변을 가리지 않게 한다.
- 중립 채움: 일반 시장 스킬.
- 강조 채움: 현재 선택한 스킬.
- 외곽 링: 사용자가 보유한 스킬.
- 추천 표시: 다음 학습 후보에만 얇은 강조 링과 `추천` 텍스트를 사용한다.
- 선 투명도: 관계 강도.
- 선 두께: 기본 0.6~1px, 선택된 직접 관계도 최대 1.6px. 원시 공고 수를 선 두께로 그대로 확대하지 않는다.
- 라벨: 개요에서 최대 8개, 선택 상태에서는 선택 스킬과 표시된 이웃만 노출한다. 충돌하는 라벨은 수요가 낮은 순서로 숨긴다.

캔버스 라벨은 디자인 시스템의 Pretendard를 사용하고, 색은 `design-tokens.ts`의 이름 있는 토큰으로만 전달한다. 현재 Geist/Arial 폴백과 렌더 함수 내부의 하드코딩된 보라색 RGBA는 제거한다.

## 5. 근거 공고 데이터 흐름

### 5.1 초기 요청

`GET /api/graph/skills`는 그래프 구조만 반환하는 가벼운 요청을 기본으로 한다. 응답 스키마의 `evidence` 배열은 하위 호환성을 위해 유지하되 기본값은 빈 배열로 둔다. 필요하면 `include_evidence=true`를 명시한 기존 내부 호출만 전체 근거를 받을 수 있다.

### 5.2 선택 후 요청

새로운 전용 요청은 선택된 한 스킬의 근거 공고만 반환한다.

```text
GET /api/graph/skills/evidence?skill=Python&career_type=experienced&limit=6
```

- 공개 상태이며 확인된 스킬 근거가 있는 공고만 대상으로 한다.
- `필수` 근거를 우선하고, 이후 최근 확인 시각과 공고 ID로 안정 정렬한다.
- 응답은 최대 6개 항목과 전체 근거 개수를 제공한다.
- 사용자가 스킬을 다시 선택하면 이전 요청 결과는 캐시하되, 늦게 도착한 이전 응답이 현재 선택을 덮지 못하게 한다.

### 5.3 화면 상태

- 초기: `기술을 선택하면 관련 공고를 확인할 수 있습니다.`
- 로딩: 패널 영역만 로딩 상태를 표시하고 그래프 조작은 막지 않는다.
- 성공: 필수·우대 구분, 기업명, 공고명, 공식 공고 링크를 제공한다.
- 결과 없음: `현재 공개된 근거 공고가 없습니다.`
- 실패: `근거 공고를 불러오지 못했습니다.`와 다시 시도 동작을 제공한다.

## 6. 성능 설계

- 데스크톱 기본 렌더 상한: 노드 12개, 선 18개.
- 모바일 기본 렌더 상한: 노드 8개, 선 10개.
- 공고 근거 노드는 캔버스에 넣지 않는다.
- force simulation은 안정화 임계값에 도달하면 정지하고 hover/tap만으로 재가열하지 않는다.
- 같은 입력의 초기 좌표와 희소 엣지 선택은 결정적이어야 하며, 렌더마다 순서가 흔들리지 않아야 한다.
- 캔버스 크기 변경은 `ResizeObserver`에서 합쳐 처리하고, 숨겨진 탭에서는 다시 계산하지 않는다.
- 근거 공고는 선택 후 지연 로딩하여 초기 네트워크 payload와 직렬화 비용을 줄인다.
- API에는 짧은 CDN 캐시와 `stale-while-revalidate`를 적용하되 개인 보유 스킬이 포함된 응답은 공유 캐시하지 않는다.

## 7. 모바일 상호작용

- 첫 탭은 노드를 선택하고 관련 이웃과 상세 패널을 갱신한다.
- 이미 선택된 노드를 다시 탭하면 상세 패널의 첫 행동으로 이동한다.
- 캔버스 드래그와 페이지 세로 스크롤을 구분한다. 한 손가락 세로 이동은 페이지 스크롤을 우선하며, 노드를 잡은 상태에서만 그래프 드래그가 시작된다.
- 확대·축소는 두 손가락 제스처와 명시적 `+`, `-`, `초기화` 버튼으로 제공한다.
- 그래프 밖 상세 패널의 버튼과 링크는 최소 44px 터치 영역을 유지한다.

## 8. 채용시장 기업 확산도

### 8.1 현재 통계 확장

`GET /api/skills/stats`의 각 기술에 다음 값을 추가한다.

```text
company_count: 해당 기술을 명시한 서로 다른 기업 수
```

집계 조건은 기존 공고 수와 동일하게 공개 공고 및 확인 임계값을 통과한 스킬 근거만 사용한다. `count`는 공고 수로 유지해 기존 계약을 깨지 않는다.

### 8.2 시장 화면

- 기본 정렬은 `기업 확산도`로 변경한다.
- 각 기술 행은 `요구 기업 N곳 · 공고 M건`을 함께 표시한다.
- 정렬 선택지는 `기업 확산도`, `명시 수요`, `전체 공고`, `필수`, `우대`, `이름`으로 제공한다.
- 설명 문구는 기업 확산도가 시장 전반의 폭, 공고 수가 채용 규모를 뜻한다는 차이를 명시한다.
- 주간 추세는 기존 공고 수 계열을 유지하되 `공고 기준`임을 제목과 툴팁에 분명히 표시한다. 기업 확산도의 과거 추세는 충분한 스냅샷이 쌓인 후 별도 작업으로 확장한다.

## 9. 공개 수집 상태

현재 `collection_status`는 수집기 실행 가능 여부이므로 그대로 유지한다. 사용자에게 실제 상태를 설명하는 파생 필드 `activity_status`를 추가한다.

- `active`: 실행 가능하고 최근 48시간 이내 성공했으며 공개 공고가 1건 이상.
- `quiet`: 실행 가능하고 최근 48시간 이내 성공했으며 공개 공고가 0건.
- `attention`: 실행 가능하지만 성공 시각이 없거나 48시간보다 오래됨.
- `preparing`: 정책 또는 커넥터 사유로 아직 실행 불가.

공개 문구는 각각 `공고 수집 정상`, `현재 공개 공고 없음`, `수집 상태 점검 필요`, 기존 준비 사유 문구를 사용한다. `quiet`는 장애처럼 붉게 표시하지 않는다.

## 10. 기업 수집원 보완

### 10.1 등록 기준

다음 네 조건 중 최소 두 개를 만족하고, 공식 채용원에서 반복 수집 가능한 구조를 확인한 기업만 추가한다.

1. 공식 한국 채용 페이지 또는 공개 채용 API가 있다.
2. 소프트웨어·데이터·AI 직군을 반복적으로 채용한다.
3. 독립적인 근무평판·산업 규모 근거가 있다.
4. 현재 데이터에서 부족한 시장 영역을 보완한다.

### 10.2 우선 검증 순서

1차 후보는 `SK AX`, `kt cloud`, `롯데이노베이트`, `한국신용데이터`다. AI/SI, 클라우드, 대기업 ICT, 핀테크 제품 개발이라는 서로 다른 영역을 보완한다.

2차 후보는 `현대카드·현대커머셜`, `LIG넥스원`, `한화비전`이다. 금융 기술과 임베디드·로보틱스·컴퓨터비전 수요를 보완한다.

각 후보는 실제 구현 시 다음 순서로 검증한다.

1. 공식 채용 URL과 robots/이용 정책을 확인한다.
2. 기존 커넥터로 수집 가능한지 작은 fixture로 검증한다.
3. 공고가 있을 때 기술 직군 필터가 정확한지 확인한다.
4. 공고가 없을 때 정상 빈 목록과 파싱 실패를 구분하는 테스트를 만든다.
5. 위 조건을 통과한 기업만 seed catalog에 추가한다. 차단된 후보는 `준비 중`으로 부풀려 공개하지 않고 검증 대기 목록에 남긴다.

## 11. 오류와 하위 호환성

- 그래프 구조 요청이 실패하면 기존 재시도 화면을 유지한다.
- 근거 공고 요청 실패는 그래프 전체 실패로 승격하지 않는다.
- `SkillGraphResponse.evidence` 필드는 제거하지 않는다.
- `SkillStatsResponse`의 기존 필드는 유지하고 `company_count`만 추가한다.
- `SourceDirectoryItem.collection_status`를 유지하고 `activity_status`를 추가한다.
- 새 필드를 모르는 이전 웹 클라이언트도 정상 동작해야 한다.

## 12. 검증 기준

### 스킬맵

- 기본 진입에서 데스크톱 노드 12개 이하, 선 18개 이하.
- 모바일 노드 8개 이하, 선 10개 이하.
- 선택 상태에서 선택 스킬과 상위 8개 이웃만 표시.
- 기본 화면에 공고 노드가 0개.
- 390px와 1,440px 화면에서 주요 라벨이 서로 겹치지 않음.
- 노드 크기·선 투명도·보유/추천 상태의 의미가 범례와 일치.
- 키보드 대체 목록과 모바일 탭으로 같은 선택 상태에 도달 가능.

### 시장 통계

- 한 기업이 같은 기술 공고를 여러 개 올려도 `company_count`는 1만 증가.
- 경력·카테고리 필터가 공고 수와 기업 수에 동일하게 적용.
- 기본 정렬이 기업 수, 동률이면 명시 공고 수, 이름 순으로 안정적.

### 수집원

- 최근 성공·공고 있음, 최근 성공·0건, 성공 지연, 준비 중 네 상태 테스트.
- 새 기업마다 실제 응답 fixture와 빈 목록 fixture를 모두 검증.
- 전체 백엔드·웹 단위 테스트, 타입 검사, 프로덕션 빌드 통과.
- 배포 후 운영 API에서 노드/선 상한, 기업 수, 새 상태 분포를 다시 측정.

## 13. Hallmark 준수 사항

- macrostructure는 기존 `Map / Diagram`을 유지하며 그래프를 주인공으로 둔다.
- 그래프 제어와 근거 공고는 보조 계층으로 낮춘다.
- 무지개 범주색, 장식용 글로우, 굵은 관계선, 과도한 pill을 추가하지 않는다.
- Pretendard와 기존 spacing/color token을 사용한다.
- 구현 후 실제 데스크톱·모바일 스크린샷으로 정보 계층과 터치 동작을 다시 감사한다.
