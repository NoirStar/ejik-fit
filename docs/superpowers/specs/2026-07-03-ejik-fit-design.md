- 프로젝트: 이직핏
- 저장소: `NoirStar/ejik-fit`
- 저장소명: `ejik-fit`
- 작성일: 2026-07-03
- 상태: 사용자 승인 완료

## 1. 제품 정의

이직핏은 한국 기업의 공식 채용 페이지에서 기술 직무 공고를 수집하고, 공고의 요구 역량을 원문 근거와 함께 구조화하여 공고 탐색, 시장 통계, 관계 그래프, 개인별 역량 차이 분석을 제공하는 IT 커리어 인텔리전스 서비스다.

브랜드 문구는 다음을 기본안으로 사용한다.

> 한국 IT 채용시장을 읽고, 내 기술에 맞는 다음 자리를 찾다.

이직핏의 `핏`은 합격 확률이 아니다. 공고에 명시된 조건과 사용자가 입력한 역량을 투명하게 비교한 결과다. 모든 비교 결과는 `충족`, `부족`, `판단 불가`로 구분하고 가능한 경우 공고 원문 근거를 함께 표시한다.

## 2. 해결하려는 문제

한국 기업의 채용 공고는 잡코리아, 사람인 같은 포털뿐 아니라 기업별 공식 채용 사이트, 국내 ATS, 자체 제작 사이트에 분산되어 있다. 사용자는 관심 기업을 일일이 방문해야 하고, 잘 알려지지 않은 우수 기업의 공고를 발견하기도 어렵다.

공고마다 직무명과 조건 표현도 달라 다음 질문에 답하기 어렵다.

- 어떤 기업이 지금 내가 원하는 기술 직무를 채용하는가?
- 신입과 경력직에게 실제로 많이 요구되는 기술은 무엇인가?
- 특정 기술과 함께 자주 등장하는 기술과 직무는 무엇인가?
- 내가 가진 기술로 지원할 수 있는 인접 직무는 무엇인가?
- 다음 커리어를 위해 무엇을 우선 학습해야 하는가?

이직핏은 공식 출처의 공고를 모으고, 근거 중심으로 구조화하여 이 질문에 답한다.

## 3. 대상 사용자와 직무 범위

### 3.1 핵심 사용자

현직 IT 종사자와 이직 준비자가 핵심 사용자다. 관심 기업과 직무를 추적하고, 자신의 기술과 공고 조건의 차이를 확인하며, 시장 수요를 근거로 커리어 방향을 정할 수 있어야 한다.

### 3.2 신입 사용자

신입도 공고 검색, 기업 탐색, 관계 그래프, 시장 통계를 모두 사용할 수 있다. 경력이 없다는 이유만으로 낮은 점수를 부여하지 않는다. 신입·경력무관·인턴 공고를 별도로 탐색하고, 해당 표본에서 요구되는 기술과 준비 조건을 확인하게 한다.

### 3.3 포함 직무

직접 기술을 다루는 다음 직무를 포함한다.

- 소프트웨어: 백엔드, 프론트엔드, 모바일, 데스크톱, 클라이언트
- 인프라: 클라우드, DevOps, SRE, 시스템, 네트워크, 데이터베이스
- 보안: 침해 대응, 모의해킹, 보안 관제, 제품 보안, 클라우드 보안, 취약점 연구
- 데이터·AI: 데이터 엔지니어링, 데이터 분석, 머신러닝, AI 엔지니어링
- 게임: 게임 클라이언트, 서버, 엔진, 테크니컬 아트
- 로보틱스·임베디드: ROS, 자율주행, 제어 소프트웨어, 시뮬레이션, 펌웨어

### 3.4 초기 제외 범위

- 자동 입사 지원
- 합격 확률 예측
- 이력서 첨삭과 자동 생성
- 기업 리뷰와 익명 커뮤니티
- 비기술 직무
- 로그인, 인증, CAPTCHA 또는 접근 통제 우회

## 4. 핵심 사용자 기능

### 4.1 공고 탐색

사용자는 직무, 기술, 기업, 지역, 경력, 고용 형태로 공고를 검색하고 필터링한다. 신입, 경력무관, 인턴 필터를 제공한다. 각 공고에는 신규, 수정, 마감 예정, 확인 지연 상태와 공식 원문 링크, 마지막 확인 시간을 표시한다.

### 4.2 공고 분석

공고 상세 화면은 담당 업무, 필수 조건, 우대 조건을 분리한다. 기술, 요구 경력, 학력, 자격증, 근무 조건을 구조화하여 표시한다. 추출한 항목에는 가능한 경우 해당 원문 문장을 근거로 제공한다. 비슷한 공고와 인접 직무도 연결한다.

### 4.3 기업 탐색

기업별로 공식 채용 페이지, 현재 공고, 최근 채용 직무와 기술 수요 변화를 보여준다. 잘 알려진 기업뿐 아니라 공공 데이터의 강소기업, 청년친화강소기업, 공시 대상 기업 등을 후보군으로 활용한다. 사용자는 관심 기업을 저장할 수 있다.

### 4.4 시장 통계

- 직무별로 자주 요구되는 기술
- 신입과 경력직의 조건 차이
- 필수 기술과 우대 기술의 비율
- 기간별 기술 수요 변화
- 함께 등장하는 기술 조합
- 통계의 기간, 표본 공고 수, 데이터 갱신 시점

표본 수가 부족한 통계는 숨기거나 낮은 신뢰도로 표시한다.

### 4.5 내 이직핏

사용자는 관심 직무와 보유 기술을 직접 등록한다. 공고별 조건을 `충족`, `부족`, `판단 불가`로 비교한다. 부족한 기술이 시장에서 얼마나 자주 요구되는지, 어떤 인접 직무에 활용되는지 보여준다. 관심 기업, 직무, 검색 조건에 맞는 신규 공고 알림은 개인화 단계에서 제공한다.

## 5. 관계 그래프

관계 그래프는 이직핏의 핵심 탐색 방식이다. 옵시디언과 유사하게 검색 키워드를 중심으로 연관 기술, 직무, 공고, 기업을 거미줄 형태로 보여준다.

### 5.1 노드 종류

- 스킬
- 직무
- 공고
- 기업

### 5.2 관계 종류

- 공고가 스킬을 필수로 요구함
- 공고가 스킬을 우대함
- 공고가 직무로 분류됨
- 기업이 공고를 게시함
- 스킬들이 같은 공고에 함께 등장함
- 직무에서 스킬이 반복적으로 요구됨

### 5.3 화면 동작

예를 들어 `ROS2`를 검색하면 ROS2가 중심 노드가 되고 C++, Linux, Python, Docker 같은 관련 스킬, 로봇 소프트웨어·자율주행·시뮬레이션 직무, 실제 공고와 기업이 주변에 표시된다.

선 굵기는 공고에서 함께 등장한 횟수나 가중치를 나타낸다. 노드 색상은 종류를 구분한다. 기간, 신입·경력, 필수·우대 필터를 그래프에도 동일하게 적용한다.

초기에는 관련도 높은 노드만 표시하고 클릭 시 단계적으로 확장한다. 과도한 노드로 화면이 읽기 어려워지는 것을 막기 위해 기본 노드 수를 제한한다. 오른쪽에는 동일 결과를 목록으로 제공하여 그래프가 유일한 탐색 수단이 되지 않게 한다.

관계는 AI가 임의로 생성하지 않는다. 실제 공고의 동시 등장, 분류 결과, 원문 근거에서 계산한다.

## 6. 데이터 수집 전략

### 6.1 기본 접근

ATS 커넥터와 범용 수집기를 결합한다. 검색엔진과 공공 데이터는 신규 기업과 채용 페이지를 발견하는 보조 수단으로 사용한다.

수집 우선순위는 다음과 같다.

1. 공개 API 또는 구조화된 공개 데이터
2. Schema.org `JobPosting` JSON-LD
3. 정적 HTML
4. 공개 네트워크 응답
5. JavaScript 렌더링이 필요한 경우에만 일반 Playwright

CloakBrowser는 기본 의존성과 수집 경로에서 제외한다. 접근 통제 우회 수단으로 사용하지 않는다.

### 6.2 초기 커넥터

- 그리팅 계열
- `recruiter.co.kr` 계열
- 나인하이어 계열
- JSON-LD 범용 수집기
- 사이트별 범용 HTML 어댑터

각 커넥터는 목록 발견과 상세 공고 파싱을 분리한다. 기업별 설정은 코드가 아니라 출처 레지스트리에 저장한다.

### 6.3 기업·출처 레지스트리

기업명, 별칭, 홈페이지, 공식 채용 URL, ATS 유형, 수집 정책 상태, 마지막 검증 시각, 활성 상태를 관리한다. 출처 상태는 `허용`, `검토 필요`, `중단`으로 구분한다.

## 7. 데이터 흐름

```text
기업 후보 목록
→ 공식 채용 출처 발견·검증
→ 예약된 수집 작업
→ ATS/범용 커넥터
→ 원문 스냅샷과 콘텐츠 해시 저장
→ 공고 구조화
→ 중복 제거와 변경 감지
→ 직무·스킬·조건 추출
→ 검색 색인·통계·관계 그래프 갱신
→ 사용자 화면 공개
```

원문 스냅샷은 재현과 파서 회귀 테스트를 위해 보관한다. 구조화 결과는 파서 버전과 함께 저장하여 추출 규칙 변경 시 다시 처리할 수 있게 한다.

## 8. 핵심 데이터 모델

- `Company`: 기업 기본 정보와 별칭
- `CareerSource`: 공식 채용 출처, ATS 유형, 수집 정책
- `CrawlRun`: 수집 시작·종료, 상태, 오류, 응답 메타데이터
- `RawSnapshot`: 원문 위치, 콘텐츠 해시, 수집 시각
- `JobPosting`: 정규화된 공고와 상태
- `JobRevision`: 공고 변경 이력
- `Role`: 표준 직무 분류
- `Skill`: 표준 기술명과 별칭
- `PostingSkill`: 공고·스킬 관계, 필수·우대, 원문 근거, 신뢰도
- `PostingRole`: 공고·직무 관계와 신뢰도
- `UserProfile`: 사용자 관심 직무
- `UserSkill`: 사용자 보유 기술과 숙련도
- `SavedSearch`: 저장한 검색 조건과 알림 설정

초기 관계 그래프는 PostgreSQL 관계 테이블과 집계 결과로 구성한다. 전용 그래프 데이터베이스는 실제 질의 복잡도와 데이터 규모가 필요성을 증명한 뒤 도입한다.

## 9. 시스템 구조

```text
Next.js Web
    ↓
FastAPI API ─ PostgreSQL
    ├─ Meilisearch
    └─ Redis 작업 큐
           ↓
      Python Worker
           ↓
기업 채용사이트 → 원문 저장 → 정규화 → 색인
```

### 9.1 기술 스택

- 웹: Next.js 16.2, TypeScript
- API: Python 3.12, FastAPI, Pydantic, SQLAlchemy, Alembic
- 수집: HTTP 클라이언트와 HTML 파서 우선, 필요한 경우 Playwright
- 작업 예약과 재시도: Redis, Celery
- 핵심 데이터: PostgreSQL
- 한국어 검색: Meilisearch
- 원문 저장: 개발 환경의 S3 호환 로컬 저장소, 운영 환경의 S3 호환 객체 저장소
- 관계 그래프: Sigma.js 안정 버전, Graphology
- 로컬 실행: Docker Compose

### 9.2 저장소 구조

```text
ejik-fit/
├── apps/
│   ├── web/
│   ├── api/
│   └── worker/
├── packages/
│   └── contracts/
├── infra/
├── docs/
└── tests/
    └── fixtures/
```

웹, API, 워커를 분리하여 수집 장애가 사용자 검색을 중단시키지 않게 한다. 초기에는 Kubernetes나 불필요한 마이크로서비스를 도입하지 않는다.

## 10. 수집 원칙과 법적 경계

- 로그인, 인증, CAPTCHA, 접근 통제를 우회하지 않는다.
- 출처별 이용약관과 robots 정책 검토 상태를 기록한다.
- 호스트별 요청 간격과 동시 실행 수를 제한한다.
- ETag와 Last-Modified를 지원하는 출처에는 조건부 요청을 사용한다.
- 서비스 식별과 연락 가능한 수집기 User-Agent를 운영 단계에 적용한다.
- 원문 전체의 대체 서비스가 되지 않도록 검색과 분석에 필요한 정보만 제공하고 공식 원문 링크를 유지한다.
- 채용 담당자 이름, 개인 이메일, 전화번호 등 개인정보는 분석 대상에서 제외한다.
- 기업 또는 권리자의 삭제·수집 중단 요청을 처리할 관리 절차를 마련한다.

공개된 정보라도 반복적·대량 복제가 데이터베이스 제작자의 권리나 사이트 약관과 충돌할 수 있으므로 출처별 허용 범위를 확인한다. 서비스 공개 전에는 국내 법률 전문가의 검토를 별도로 받는다.

## 11. 실패 처리

- 한 번의 수집 실패로 공고를 마감 처리하지 않는다.
- 재시도 후에도 실패하면 출처와 공고를 `확인 지연` 상태로 표시한다.
- 공식 마감 표시, 명시된 마감일 경과, 여러 차례의 정상 목록 수집에서 연속 부재 중 하나가 충분히 확인될 때만 마감한다.
- 구조 변경으로 추출 신뢰도가 임계값보다 낮아지면 자동 공개하지 않고 검토 대기로 보낸다.
- 이전 정상 데이터를 유지하고 실패한 새 결과가 이를 덮어쓰지 못하게 한다.
- URL, 출처 공고 ID, 콘텐츠 해시, 기업·제목·기간 조합으로 중복을 판별한다.
- 모든 변경은 공고 수정 이력으로 남긴다.

## 12. 테스트와 품질 지표

### 12.1 테스트

- ATS별 고정 HTML을 사용하는 파서 단위 테스트
- 필수·우대 조건과 기술 추출 골든 테스트
- 중복 제거, 변경 감지, 마감 판정 테스트
- 검색과 필터 API 계약 테스트
- 통계와 그래프 관계 계산 테스트
- 웹의 핵심 사용자 흐름 E2E 테스트
- 실제 사이트에는 낮은 빈도의 별도 상태 점검

테스트용 HTML은 재현 가능한 fixture로 보관하고 필요하면 개인정보와 불필요한 콘텐츠를 제거한다.

### 12.2 운영 지표

- 출처별 수집 성공률
- 마지막 정상 수집 시각과 데이터 신선도
- 공개 공고 수와 확인 지연 공고 수
- 중복률
- 추출 검토 대기 건수
- 수동 검증 표본의 필수·우대·스킬 추출 정확도
- 검색 결과가 없는 주요 질의

많은 공고 수보다 출처와 근거가 명확한 신뢰 가능한 데이터를 우선한다.

## 13. 구현 로드맵

### 13.1 0단계: 원격 저장소와 개발 환경

- GitHub 원격 저장소 `NoirStar/ejik-fit`을 유일한 소스 오브 트루스로 사용
- 웹, API, 워커 기본 구조
- PostgreSQL, Redis, Meilisearch, 객체 저장소의 Docker Compose 구성
- 환경 변수 예제, 코드 검사, 테스트, 개발 명령 문서화

완료 기준은 새 체크아웃에서 문서화된 명령으로 전체 개발 서비스를 실행하고 상태 점검을 통과하는 것이다.

### 13.2 1단계: 작동하는 세로 조각

- 기업 10곳 등록
- 공통 ATS 커넥터 1개와 JSON-LD 범용 수집기
- 원문 저장, 중복 제거, 공고 목록·상세 화면
- 출처와 마지막 확인 시간 표시

완료 기준은 실제 공식 공고가 끝까지 수집되어 검색되고, 화면의 구조화 결과를 원문과 대조할 수 있는 것이다.

### 13.3 2단계: 수집 범위 확대

- 그리팅, `recruiter.co.kr`, 나인하이어 커넥터
- 기업 100곳 이상
- 예약 수집, 재시도, 변경·마감 감지
- 수집 상태 관리 화면

### 13.4 3단계: 직무 인텔리전스

- 기술·직무 표준 사전
- 필수·우대 조건과 근거 문장 추출
- 신입·경력별 기술 수요 통계
- 관계 그래프

### 13.5 4단계: 내 이직핏

- 사용자 계정과 보유 스킬
- 관심 직무·기업·검색 조건 저장
- 공고별 조건 비교
- 인접 직무와 학습 우선순위
- 신규 공고 알림

### 13.6 5단계: 공개 서비스 확장

- 기업 발견 자동화와 출처 확장
- 삭제·수집 중단 요청 처리
- 품질 모니터링과 운영 비용 최적화
- 필요성이 입증된 경우에만 그래프 DB와 분산 워커 도입

## 14. 첫 구현 계획의 범위

첫 구현 계획은 0단계와 1단계만 다룬다. 저장소 기반과 실제 공고 하나의 전체 데이터 흐름을 먼저 검증한다. 2단계 이후는 앞 단계의 데이터 품질, 운영 비용, 실패 패턴을 확인한 뒤 별도 설계와 구현 계획으로 진행한다.

첫 구현에서 다음은 하지 않는다.

- AI 기반 자유 형식 추출의 전면 적용
- 사용자 계정과 개인화
- 관계 그래프의 완성형 UI
- 대규모 기업 자동 발견
- CloakBrowser 도입

## 15. 첫 구현 성공 기준

- GitHub 원격 저장소에서 코드와 문서를 일관되게 관리한다.
- Docker 기반 로컬 실행 절차가 재현된다.
- API, 워커, 웹의 기본 상태 점검이 자동화된다.
- 최소 10개 기업과 공식 채용 출처를 등록할 수 있다.
- 최소 하나의 공통 커넥터와 JSON-LD 수집기가 fixture 테스트를 통과한다.
- 실제 공고가 원문 스냅샷, 정규화, 중복 제거, 검색 색인, 웹 표시까지 흐른다.
- 공고 화면에 공식 출처와 마지막 확인 시간을 표시한다.
- 수집 실패가 기존 정상 공고를 잘못 마감하거나 삭제하지 않는다.
