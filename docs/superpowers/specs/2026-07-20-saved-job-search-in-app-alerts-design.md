# 저장 검색과 인앱 새 공고 알림 설계

- 작성일: 2026-07-20
- 상태: 사용자 승인 후 문서화
- 대상: 이직핏 공고 탐색, 내 정보, 알림 메뉴

## 1. 목적

사용자가 공고 탐색에서 사용한 검색 조건을 계정에 저장하고, 이후 이직핏이 새로
확인한 공식 공고 중 해당 조건에 맞는 항목을 서비스 안에서 다시 확인할 수 있게
한다.

첫 버전의 핵심 흐름은 다음과 같다.

```text
공고 조건 설정
→ 검색 저장
→ 이직핏 재방문
→ 알림 메뉴 또는 알림 관리 페이지 열기
→ 저장 시점 이후 새로 확인된 공고 확인
→ 실제 공고 상세 또는 같은 검색 결과로 이동
```

이 기능의 `새 공고`는 기업의 실제 게시 시각을 추정한 값이 아니라 이직핏의
`first_seen_at`이 마지막 정상 확인 시각보다 늦은 공고를 뜻한다. 화면에서는
`새로 등록됨`보다 `이직핏이 새로 확인`이라는 표현을 사용한다.

## 2. 확정 범위

### 포함

- 로그인 사용자의 검색 조건 저장
- 검색 이름 자동 생성과 수정
- 저장 검색 최대 10개 관리
- 저장 검색 활성화와 일시 중지
- 저장 검색 삭제
- 조건에 맞는 현재 공고로 이동
- 저장 이후 이직핏이 새로 확인한 공고의 인앱 표시
- 알림별 마지막 정상 확인 시각 기록
- 기존 관심 기업·저장·지원 현황 알림과의 결합

### 제외

- 이메일 발송
- 브라우저 푸시
- 모바일 OS 푸시
- 백그라운드 알림 워커
- 로그인하지 않은 사용자의 브라우저 저장 검색
- 외부 채용 사이트의 게시 시각 추정
- 저장 검색을 다른 사용자와 공유하는 기능
- AI가 검색 조건이나 알림 이름을 생성하는 기능

## 3. 제품 원칙

1. 저장 검색은 로그인 기능의 명확한 이점으로 제공한다.
2. 검색 저장과 알림 판단은 기존 공고 API의 실제 결과만 사용한다.
3. API 오류를 새 공고 없음으로 바꾸지 않는다.
4. 결과 조회에 성공한 검색만 마지막 확인 시각을 갱신한다.
5. 동일 조건을 여러 번 저장하지 않는다.
6. 첫 버전은 설정 마법사나 별도 발송 인프라 없이 짧은 사용자 흐름으로 만든다.

## 4. 데이터 모델

Alembic migration으로 `user_saved_job_searches` 테이블을 추가한다. 운영
PostgreSQL에서는 Supabase Auth 사용자와 연결하고 RLS로 소유자만 접근하게 한다.
SQLite 테스트에서도 같은 핵심 컬럼과 제약을 생성할 수 있어야 한다.

| 컬럼 | 의미 |
| --- | --- |
| `id` | 저장 검색 UUID |
| `user_id` | Supabase Auth 사용자 UUID |
| `name` | 사용자가 보는 검색 이름, 최대 60자 |
| `query_text` | 공고 검색어, 최대 200자 |
| `query_key` | 중복 비교용 정규화 검색어 |
| `category` | 현재 공고 화면이 지원하는 기술 분야 값 |
| `career_type` | `new_comer`, `experienced`, `mixed` 또는 빈 값 |
| `is_enabled` | 인앱 알림 확인 대상 여부 |
| `last_checked_at` | 마지막으로 결과 조회에 성공한 기준 시각 |
| `created_at` | 생성 시각 |
| `updated_at` | 마지막 수정 시각 |

데이터베이스 제약은 다음을 보장한다.

- `user_id`, `query_key`, `category`, `career_type` 조합은 유일하다.
- 검색어, 기술 분야, 경력 조건 중 하나 이상이 있어야 한다.
- 이름과 검색어의 최대 길이를 제한한다.
- 기술 분야와 경력 조건은 현재 공고 API가 지원하는 값만 허용한다.
- PostgreSQL에서는 `auth.users(id)` 삭제 시 함께 삭제한다.
- PostgreSQL RLS 정책은 `auth.uid() = user_id`인 행만 조회·삽입·수정·삭제하게 한다.

최대 10개 규칙은 UI와 저장 서비스에서 확인한다. 알림 조회 API도 한 요청에 최대
10개만 받아, 클라이언트 변조가 공고 API 부하 증가로 이어지지 않게 한다.

## 5. 저장 검색 정규화

웹의 별도 모델 계층에서 다음을 담당한다.

- 검색어 앞뒤 공백 제거와 연속 공백 축약
- 중복 비교용 소문자 `query_key` 생성
- 지원하지 않는 분야와 경력 값 제거
- 같은 필터 조합의 fingerprint 생성
- 화면용 자동 이름 생성

자동 이름은 활성 조건을 짧게 연결한다.

```text
Python · 백엔드 · 경력
Kubernetes · 인프라
신입 프론트엔드
```

필터가 하나도 없으면 저장하지 않고 `검색어·기술 분야·경력 조건 중 하나를
선택해 주세요.`라고 안내한다.

## 6. 저장과 인증 흐름

공고 탐색 필터 영역의 검색·초기화 동작 가까이에 `이 검색 저장` 버튼을 배치한다.

### 로그인 사용자

1. 버튼을 누르면 필터 영역 안에 작은 이름 입력 폼을 연다.
2. 현재 조건으로 자동 이름을 채운다.
3. 저장 성공 시 폼을 닫고 성공 안내와 알림 관리 링크를 표시한다.
4. 같은 조건이 이미 있으면 새 행을 만들지 않고 기존 검색 이름을 안내한다.

### 비로그인 사용자

1. 현재 공고 URL을 `next`에 보존해 로그인 화면으로 이동한다.
2. URL에는 `save_search=1`을 추가해 로그인 후 저장 흐름을 다시 열 수 있게 한다.
3. 인증 callback은 기존 안전한 내부 경로 검증을 그대로 사용한다.
4. 로그인 후 원래 검색어·기술 분야·경력 조건이 유지된 공고 화면으로 돌아온다.

비로그인 상태에서 localStorage에 임시 저장 검색을 만들거나 로그인 시 병합하는
경로는 추가하지 않는다.

## 7. 저장 검색 관리 화면

`/career/alerts`를 추가하고 다음 경로에서 연결한다.

- 사용자 메뉴의 `공고 알림`
- `/career/account`의 커리어 데이터 요약
- 검색 저장 성공 안내
- 알림 메뉴의 `전체 알림 관리`

페이지 제목은 다른 내 정보 화면과 같은 32px 안팎의 밀도를 유지한다. 규칙마다
거대한 카드를 만들지 않고 하나의 목록 패널 안에서 행으로 구분한다.

각 행은 다음 정보를 제공한다.

- 검색 이름
- 검색어·기술 분야·경력 조건 칩
- 활성 또는 일시 중지 상태
- 현재 조건에 맞는 공식 공고 수
- 새로 확인된 공고 수
- 마지막 정상 확인 시각
- `공고 보기`, 이름 수정, 일시 중지·재개, 삭제

삭제는 실수 방지를 위해 같은 행 안에서 한 번 더 확인한다. 활성화와 이름 변경은
낙관적으로 반영하되 Supabase 쓰기가 실패하면 이전 값으로 되돌리고 오류를
표시한다.

## 8. 인앱 알림 판단

저장 검색 행은 인증된 브라우저 Supabase client가 RLS 범위 안에서 읽는다. 공고
매칭은 공개 공고 API를 감싸는 Next route에서 수행한다.

권장 route:

```text
POST /notifications/saved-search-jobs
```

요청은 최대 10개의 정규화된 검색 규칙을 받는다. 각 규칙에 대해 기존
`getPostings()`를 사용하고, `Promise.allSettled` 방식으로 일부 실패가 다른 규칙의
성공 결과를 막지 않게 한다.

응답 그룹에는 다음을 포함한다.

- 저장 검색 ID
- 현재 조건의 전체 공고 수
- `first_seen_at > last_checked_at`인 새 공고
- 평가 시각
- 해당 규칙의 성공 또는 오류 상태

새 공고는 전체 규칙을 합쳐 최신 확인순으로 정렬하고 알림 메뉴에는 최대 5개만
노출한다. 같은 공고가 여러 저장 검색에 걸리면 한 번만 표시하되, 어떤 저장
검색에서 발견됐는지 이름을 함께 표시한다.

알림 메뉴나 `/career/alerts`가 결과를 정상적으로 렌더링한 뒤 성공한 규칙만
`last_checked_at`을 응답의 평가 시각으로 갱신한다. 확인 시각 업데이트가 실패하면
다음 방문에 같은 공고가 다시 보일 수 있지만, 알림을 잃는 것보다 안전한 동작으로
본다.

첫 저장 시 `last_checked_at`은 현재 시각으로 시작한다. 따라서 이미 존재하던
공고를 신규 알림처럼 쏟아내지 않는다.

일시 중지된 검색은 인앱 알림 batch에서 제외하고 확인 시각도 갱신하지 않는다.
사용자가 다시 활성화하면 재개 시각을 새 `last_checked_at`으로 저장한다. 따라서
알림을 꺼둔 기간의 공고가 재개 직후 한꺼번에 신규 알림으로 나타나지 않는다.
관리 화면은 일시 중지 상태에서도 사용자가 `공고 보기`를 눌러 현재 검색 결과를
직접 확인할 수 있게 한다.

## 9. 알림 메뉴 구성

기존 `ActivityNotificationCenter`의 관심 기업·저장 공고·지원 기록·내 기술 안내는
유지한다. 저장 검색 신규 공고는 책임이 다른 별도 컴포넌트로 분리해 위쪽에
표시한다.

예시:

```text
저장 검색 · Python 백엔드
Google Korea · Backend Engineer
이직핏이 새로 확인
```

목록 항목 전체는 공고 상세로 이동한다. 신규 항목이 5개보다 많으면
`공고 알림에서 더 보기`를 제공한다. 백그라운드 작업이 없으므로 사이트에 들어오기
전부터 정확한 미확인 개수 배지를 약속하지 않는다.

## 10. 컴포넌트와 책임

이름은 저장소 구조에 맞게 조정할 수 있지만 책임은 다음처럼 분리한다.

- `saved-job-searches.ts`
  - 타입, 정규화, fingerprint, 자동 이름
- `use-saved-job-searches.ts`
  - 인증 사용자 RLS CRUD, 최대 개수와 낙관적 업데이트
- `SavedSearchComposer`
  - 공고 필터에서 현재 조건 저장
- `SavedSearchManager`
  - `/career/alerts` 목록과 관리 동작
- `SavedSearchNotificationList`
  - 저장 검색 신규 공고 로딩과 표시
- `/notifications/saved-search-jobs`
  - 최대 10개 규칙 검증, 공고 API batch 조회, 부분 성공 응답

공고 필터, Supabase client, 공고 응답 정규화와 회사 로고 컴포넌트는 기존 구현을
재사용한다.

## 11. 상태와 오류 처리

### Loading

- 공고 필터 전체를 막지 않고 저장 버튼 또는 관리 목록 행만 로딩 처리한다.
- 알림 창은 기존 메뉴 크기를 유지하는 짧은 상태 문구를 사용한다.

### Empty

- 저장 검색 없음: 공고 탐색으로 이동하는 CTA를 제공한다.
- 저장 검색은 있으나 새 공고 없음: `마지막 확인 이후 새로 확인된 공고가 없습니다.`
- 현재 조건의 공고 없음: 저장 검색은 유지하고 조건 수정 또는 공고 탐색 링크를
  제공한다.

### Error

- 저장 목록 로딩 실패: 재시도 버튼을 표시한다.
- 공고 매칭 전체 실패: 확인 시각을 갱신하지 않는다.
- 일부 규칙 실패: 성공한 결과를 표시하고 실패한 규칙은 별도 안내한다.
- 저장·수정·삭제 실패: 낙관적 상태를 되돌리고 행 가까이에 오류를 표시한다.

## 12. 접근성과 반응형

- 저장, 일시 중지, 삭제 버튼에 명확한 이름과 `focus-visible` 상태를 둔다.
- 상태를 색상만으로 표현하지 않고 `활성`, `일시 중지` 텍스트를 함께 표시한다.
- 인라인 저장 폼의 오류는 입력과 연결하고 `role="alert"`로 읽히게 한다.
- 모바일에서는 관리 행의 조건과 메타 정보를 세로로 재배치하고 작업 버튼은
  최소 터치 크기를 지킨다.
- 긴 검색 이름과 기업·공고 제목은 의미가 남도록 말줄임 처리한다.

## 13. 성능과 보안

- 저장 검색 조회는 로그인 사용자에게만 수행한다.
- batch route는 규칙 수, 문자열 길이, enum 값을 다시 검증한다.
- 한 규칙당 반환하는 신규 공고 수를 제한한다.
- 병렬 공고 조회는 제한된 10개 범위에서만 실행한다.
- 저장 검색 쓰기는 Supabase publishable client와 RLS만 사용하며 service role key를
  브라우저에 노출하지 않는다.
- 외부 URL이나 공고 본문을 저장 검색 테이블에 저장하지 않는다.

## 14. 테스트 전략

사용자 요청에 따라 모든 표시 코드를 TDD로 만들지 않고, 오류 비용이 큰 경계만
선별한다.

### 단위 테스트

- 필터 정규화와 자동 이름
- 지원하지 않는 enum 제거
- 동일 조건 fingerprint와 중복 방지
- 최대 10개 제한
- 신규 공고 dedupe와 정렬

### 데이터베이스 보안 테스트

- RLS 활성화
- 본인 행만 접근하는 owner policy
- Auth 사용자 외래 키
- 유일 제약과 필터 최소 한 개 제약

### 중요 UI 동작 테스트

- 비로그인 저장 시 현재 검색 조건을 보존한 로그인 링크
- 로그인 저장 성공과 중복 안내
- 일부 알림 조회 실패 시 성공 결과 유지
- 오류 규칙의 `last_checked_at` 미갱신

### 완료 검증

- 관련 Vitest와 backend migration 보안 테스트
- TypeScript 검사
- production build
- 공고 저장 → 알림 관리 → 같은 조건의 공고 이동 브라우저 확인
- 데스크톱과 모바일 핵심 화면 확인

CSS 픽셀 값, 단순 아이콘 표시, 정적인 설명 문구를 위한 별도 테스트는 만들지
않는다.

## 15. 완료 기준

- 로그인 사용자가 현재 공고 검색 조건을 저장할 수 있다.
- 로그인 전 저장 요청은 검색 조건을 잃지 않고 인증 후 이어진다.
- 동일 조건과 10개 초과 저장이 차단된다.
- `/career/alerts`에서 수정·중지·재개·삭제·공고 이동이 동작한다.
- 저장 이후 이직핏이 새로 확인한 실제 공고만 알림에 나타난다.
- API 오류는 새 공고 없음으로 표시되지 않고 확인 시각도 갱신하지 않는다.
- 사용자 간 저장 검색 데이터가 RLS로 격리된다.
- 기존 관심 기업·저장·지원 알림이 깨지지 않는다.
- 모바일과 데스크톱에서 레이아웃과 키보드 조작이 정상이다.
- TypeScript, 관련 테스트와 production build가 통과한다.
