# 이직핏 커뮤니티 피드 홈 설계

## 상태

- 작성일: 2026-07-13
- 범위: 글로벌 앱 셸과 홈 화면 개편, 연결 라우트의 최소 셸
- 기준 자료: `docs/handoff/2026-07-13-remake-ui.md`, `docs/handoff/image.png`
- 결정: 레퍼런스의 정보 구조와 밀도를 따르되 현재 데이터 계약과 신뢰 원칙에 맞지 않는 세부 요구는 수정한다.

## 목표

기존의 좌측 내비게이션 기반 채용 대시보드 홈을 커뮤니티 피드 중심의 커리어 네트워크 홈으로 교체한다. 사용자는 실제 경험을 읽고, 같은 흐름 안에서 실제 공고 데이터로 확인한 시장 신호와 관련 공고를 탐색할 수 있어야 한다.

이번 변경은 다음을 동시에 만족해야 한다.

1. 중앙 커뮤니티 피드가 홈의 명확한 주인공이어야 한다.
2. 공고, 스킬 수요, 시장 수치는 기존 API 응답에서만 만들어야 한다.
3. 커뮤니티 글과 면접 후기는 자연스러운 한국어 mock 데이터로 제공해도 된다.
4. 기존 공고, 스킬 그래프, 신뢰성 페이지와 백엔드 계약을 보존해야 한다.
5. 데스크톱, 태블릿, 모바일에서 탐색과 핵심 상호작용이 실제로 작동해야 한다.

## 현재 코드 감사 결과

- 프레임워크: Next.js 16 App Router, React 19, TypeScript
- 스타일: CSS Modules와 전역 semantic token, Pretendard Variable
- 아이콘: Phosphor Icons
- 상태: React 로컬 상태와 URL 검색 파라미터, 보유 기술은 기존 localStorage 기반 시트 사용
- 데이터: 서버 컴포넌트에서 공고, 스킬 통계, 스킬 그래프 API를 병렬 호출하고 부분 실패를 구분
- 현재 앱 셸: 데스크톱 좌측 고정 레일, 모바일 하단 4개 내비게이션
- 기존 주요 라우트: `/`, `/jobs`, `/jobs/[id]`, `/skills/graph`, 신뢰성 페이지
- 기준 검증: 테스트 66개, TypeScript 검사, 프로덕션 빌드가 모두 통과

현재의 API 호출, 실패 상태 모델, 공고 상세, 스킬 그래프, 보유 기술 저장 구조는 유지한다. 홈 전용 대시보드 표현과 좌측 고정 전역 셸은 새 구조로 교체한다.

## 데이터 신뢰 경계

### mock 사용 가능

- 커뮤니티 질문
- 면접 후기
- 댓글 미리보기
- 인기 커뮤니티 주제
- 작성자 표시명과 아바타 대체 표현

mock 콘텐츠는 컴포넌트 내부에 흩뿌리지 않고 `features/home-feed/data` 계층에 typed fixture로 둔다. 외부 회사의 실제 면접을 사실처럼 단정하지 않도록 기업명 대신 회사 유형이나 명시적인 가상 조직명을 사용한다.

### 실제 API만 사용

- 공고명, 회사명, 지역, 경력, 고용 형태, 출처, 확인 시각
- 스킬 수요 건수
- 필수 및 우대 요구 건수
- 공고와 기술의 연결 관계
- 사용자가 선택한 기술과 실제 공고 요구 기술의 겹침
- 분석 표본 수와 출처 수

실제 응답이 실패했을 때 mock 수치로 대체하지 않는다. 해당 피드 아이템만 오류 또는 비어 있음 상태로 바꾸고 커뮤니티 피드는 계속 표시한다.

### 이번 API로 표현하지 않는 수치

현재 API에는 역사적 시계열과 추천 전후 시뮬레이션이 없다. 따라서 문서 예시의 `최근 8주 +14%`, sparkline, `42개에서 65개로 증가`, 합격 가능성, 정밀 매칭 점수는 표시하지 않는다. 대신 현재 API로 검증 가능한 표본 수, 필수/우대 건수, 함께 요구된 횟수, 선택 기술과 겹치는 공고 수를 사용한다.

## 정보 구조와 시각 방향

### 글로벌 헤더

레퍼런스처럼 데스크톱에서 2단 구조를 사용한다.

- 첫 줄: 이직핏 로고, 통합 검색, 글쓰기, 알림, 사용자 메뉴
- 둘째 줄: 홈, 시장, 스킬맵, 공고, 내 커리어
- 현재 위치는 보라색 텍스트와 얇은 하단 선으로만 강조
- 헤더는 sticky이되 작은 노트북에서 첫 피드 카드가 충분히 보이도록 레퍼런스보다 높이를 절제
- 모바일은 로고, 검색, 알림만 남기고 주요 탐색은 하단 고정 내비게이션으로 이동

### 홈 레이아웃

- 최대 폭: 약 1,420px
- 데스크톱: 왼쪽 232px / 중앙 minmax(0, 744px) / 오른쪽 300px
- 태블릿: 중앙과 오른쪽의 2열, 왼쪽 프로필 기능은 내 커리어와 보유 기술 시트로 이동
- 모바일: 단일 피드, 오른쪽 인사이트는 피드 사이의 보조 모듈로 재배치

중앙 열은 너비, 대비, 콘텐츠 밀도 모두에서 가장 강해야 한다. 좌우 레일은 큰 외곽 카드 하나로 묶지 않고 섹션 간 여백과 약한 구분선으로 구성한다.

### 디자인 토큰

- 밝은 회청색 앱 배경과 흰 표면
- 차콜 기본 텍스트, 청회색 보조 텍스트
- 보라색은 선택 상태, 핵심 링크, 글쓰기 버튼에만 제한
- 카드 테두리는 얇고 그림자는 거의 사용하지 않음
- 외부 카드 반경은 약 14px, 내부 컨트롤은 8~10px
- 한국어 본문은 Pretendard 14~16px, 행간 1.6 전후
- 화려한 그라데이션, 글로우, 과도한 glass 효과, 장식용 차트는 사용하지 않음

## 컴포넌트 구조

### 공통 앱 셸

- `AppHeader`: 데스크톱 2단 헤더와 모바일 상단 바
- `GlobalSearch`: `/` 키 포커스와 `/jobs?q=` 제출
- `PrimaryNavigation`: 다섯 개 글로벌 목적지와 현재 위치 표시
- `MobileBottomNavigation`: 홈, 시장, 스킬맵, 공고, 내 정보
- `ComposerDialog`: 글쓰기 버튼이 여는 접근 가능한 로컬 작성 폼
- `NotificationMenu`, `UserMenu`: 키보드로 열고 닫을 수 있는 메뉴

실제 인증 세션이 없으므로 사용자 메뉴에 가짜 이름이나 직무를 넣지 않는다. 헤더와 왼쪽 레일은 `내 커리어 설정`과 사용자가 로컬에서 선택한 기술만 보여준다.

### 홈

- `HomeFeedPage`: 서버 데이터 조합과 전체 레이아웃
- `HomeFeedClient`: 탭, 공감, 저장, 작성 글의 클라이언트 상태
- `LeftCareerRail`: 게스트/로컬 기술 기반 프로필 요약, 관심 기술, 빠른 링크
- `FeedContextBar`: 현재 선택 기술과 API 확인 기준
- `FeedTabs`: 추천, 팔로잉, 최신, 인기
- `FeedList`: discriminated union 렌더러
- `CommunityPostCard`
- `InterviewReviewCard`
- `MarketInsightCard`
- `RecommendedJobCard`
- `SocialActions`
- `RightInsightRail`: 인기 주제, 실제 스킬 수요, 데이터 기준 요약

클라이언트 컴포넌트의 경계는 상호작용이 필요한 홈 피드와 메뉴에 한정한다. API 호출과 실제 데이터 변환은 서버에 둔다.

## 데이터 모델과 흐름

홈 피드는 `FeedItem` discriminated union을 사용한다.

- `community_post`: mock 질문과 로컬 작성 글
- `interview_review`: mock 후기
- `market_insight`: `SkillStatsResponse`와 `SkillGraphResponse`에서 변환
- `recommended_job`: `PostingListResponse`와 graph evidence에서 변환

흐름은 다음과 같다.

1. 홈 서버 컴포넌트가 기존 세 API를 병렬 호출한다.
2. 기존 `ResourceState`로 성공, 부분 실패, 전체 실패를 보존한다.
3. 순수 함수가 API 응답을 실제 데이터 피드 아이템으로 바꾼다.
4. mock 커뮤니티 fixture와 실제 데이터 아이템을 명시적인 순서 규칙으로 합친다.
5. 클라이언트는 합쳐진 초기 목록을 받아 탭 필터, 공감, 저장, 로컬 작성만 처리한다.
6. 공고와 기술 링크는 기존 실제 라우트로 이동한다.

필드가 없는 값은 추정하지 않고 `미기재`, `확인 필요`, 또는 UI 생략으로 처리한다.

## 기업 로고

초기 구현은 런타임 외부 로고 API에 의존하지 않는다.

1. 실제 API에 등장하는 주요 기업에 대해 공식 홈페이지, 뉴스룸, 브랜드 자료실에서 제공하는 로고를 선별한다.
2. 허용된 파일을 `public/company-logos`에 저장하고 회사명/도메인 매핑을 둔다.
3. `source_url` 호스트와 정규화된 회사명을 사용해 로고를 찾는다.
4. 로고가 없거나 출처가 불명확하면 회사명 이니셜을 사용한다.
5. mock 면접 후기는 실제 기업 로고를 사용하지 않는다.

resolver 인터페이스는 나중에 Logo.dev 같은 서비스로 교체할 수 있게 유지한다. 모든 로고 프레임은 고정 크기이며 로딩 실패가 레이아웃 이동을 만들지 않아야 한다.

## 상호작용

- 피드 탭은 목록 유형과 정렬을 바꾸며 활성 상태를 텍스트로도 표시한다.
- 공감과 저장은 버튼으로 즉시 토글되고 `aria-pressed`를 제공한다.
- 로컬 작성 글은 현재 세션의 피드 상단에 추가한다. 서버 저장처럼 표시하지 않는다.
- 보유 기술 추가는 기존 `OwnedSkillsSheet`를 재사용한다.
- 기술 태그는 `/skill-map?skill=...`로 이동한다.
- 검색은 실제 공고 검색 경로로 제출한다.
- 메뉴와 dialog는 Escape, 바깥 클릭, 포커스 복귀를 지원한다.
- 모션은 150~200ms의 opacity, transform, border/background 전환으로 제한한다.

## 라우팅

- `/`: 새 홈 피드
- `/jobs`, `/jobs/[id]`: 기존 실제 공고 경험 유지
- `/skills/graph`: 기존 스킬 그래프 유지
- `/skill-map`: `skill` 파라미터를 기존 그래프의 `seed` 파라미터로 변환하고 나머지 검색 조건을 보존해 `/skills/graph`로 연결하는 사용자 친화 경로
- `/market`: 실제 시장 API가 아직 제공하지 않는 기능은 과장하지 않는 준비 중 셸
- `/career`: 보유 기술 시트와 연결되는 준비 중 셸
- `/posts/[id]`: mock 콘텐츠용 최소 상세 셸

기존 신뢰성 페이지와 `/skills/graph` 링크는 깨뜨리지 않는다. 준비 중 셸은 공통 헤더, 제목, 현재 가능한 기능, 홈 복귀 링크를 제공한다.
기존 스킬 그래프 페이지도 `seed` 검색 파라미터를 받아 실제 API 요청의 시작 기술로 사용하도록 확장한다.

## 상태와 오류 처리

- 커뮤니티 mock은 실제 API 장애와 무관하게 표시한다.
- 공고 API 실패: 추천 공고 위치에 짧은 오류 상태와 다시 시도 링크 표시
- 스킬 통계/그래프 부분 실패: 확인 가능한 실제 영역만 표시하고 누락된 지표는 숨김
- 실제 데이터가 비어 있음: 기술 설정 또는 공고 탐색으로 이어지는 빈 상태 표시
- 로고 실패: 이니셜 fallback
- 로컬 작성 폼: 필수 제목/본문 검증과 인라인 오류
- 로딩: 카드의 실제 구조를 닮은 skeleton 사용

## 접근성

- `header`, `nav`, `main`, `aside`, `article`을 용도에 맞게 사용
- 링크와 버튼을 역할에 맞게 구분
- 모든 아이콘 버튼에 접근 가능한 이름 제공
- 현재 페이지와 현재 탭을 `aria-current`, `aria-selected`로 표현
- 공감과 저장은 `aria-pressed` 사용
- 최소 44px 터치 영역과 명확한 `focus-visible` 제공
- 모바일 하단 내비게이션이 본문을 가리지 않도록 safe area 반영
- `prefers-reduced-motion`에서 비필수 전환 제거

- 필수/우대 기술 구분과 누락 값 처리
- 기업 로고 resolver와 이니셜 fallback
- 피드 탭 필터/정렬 규칙

### 컴포넌트 테스트

- 앱 셸의 다섯 목적지와 현재 위치
- 공감/저장 토글과 접근성 속성
- 글쓰기 dialog 검증과 로컬 글 추가
- 부분 API 실패에서도 커뮤니티 피드 유지
- 실제 데이터 영역에 mock 수치가 섞이지 않음
- 모바일 내비게이션 목적지

### 통합 검증

- 기존 공고, 공고 상세, 스킬 그래프, 신뢰성 페이지 테스트 유지
- TypeScript 검사, 전체 Vitest, Next.js 프로덕션 빌드
- 데스크톱, 태블릿, 모바일 viewport의 스크린샷과 주요 클릭 흐름 확인
- 콘솔 오류, 수평 오버플로, 키보드 포커스 이동 확인

## 정리 범위

- 홈 교체 후 참조되지 않는 `features/dashboard` 홈 표현과 테스트는 제거한다.
- 기존 API 변환 로직 중 유용한 순수 함수는 새 `home-feed` 모델로 이동한다.
- `components/dashboard`와 거대한 전역 CSS는 전체 참조를 확인한 뒤 홈/과거 대시보드 전용 코드만 제거한다.
- 다른 페이지에서 사용하는 그래프, 공고, 정책, 보유 기술 코드는 보존한다.
- 새 UI 라이브러리나 상태 관리 라이브러리는 추가하지 않는다.

## 완료 기준

1. 홈의 첫 인상이 커뮤니티 피드이며 데이터 대시보드처럼 보이지 않는다.
2. 실제 공고/스킬/시장 정보와 mock 커뮤니티 데이터의 경계가 코드에서 분명하다.
3. 근거 없는 추세, 매칭률, 예상 합격 가능성이 없다.
4. 실제 공고에 가능한 경우 공식 기업 로고가 보이고 항상 fallback이 있다.
5. 데스크톱 3열, 태블릿 2열, 모바일 단일 열과 하단 내비게이션이 자연스럽다.
6. 핵심 상호작용, 키보드 탐색, 오류/빈/로딩 상태가 작동한다.
7. 기존 실제 데이터 페이지와 API 계약이 깨지지 않는다.
8. 전체 테스트, 타입 검사, 빌드가 통과한다.
