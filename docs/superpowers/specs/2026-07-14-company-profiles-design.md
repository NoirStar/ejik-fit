# 실제 공고 기반 기업 채용 상세 설계

## 목표

공고 목록과 상세에서 본 기업을 `/companies/[companyId]`에서 다시 확인할 수 있게 한다. 사용자는 기업 홍보 문구나 추정 수치가 아니라, 공식 채용페이지에서 현재 확인된 공개 공고·요구 기술·경력·고용 형태·근무지만으로 이 기업의 채용 범위를 판단할 수 있어야 한다.

## 선택한 방향

기업 일반 소개가 아닌 **채용 근거 프로필**을 만든다. 기업 규모, 문화, 평균 연봉, 직원 수, 성장률처럼 현재 API에 없는 값은 만들거나 추정하지 않는다. `companyId`는 백엔드의 안정적인 기업 slug를 사용하고, 페이지의 모든 집계는 해당 slug로 조회한 공식 공고 API 응답에서 계산한다.

사용자가 세부 설정 판단과 순차 구현을 위임했으므로 이 방향을 승인된 작업 기준으로 적용한다.

## 정보 구조

1. 공고 탐색으로 돌아가는 경로
2. 검증된 기업 마크, 기업명, “공식 채용 공고 기준” 설명
3. 현재 응답 범위 요약
   - 공개 공고 수
   - 확정 기술 수
   - 근무 지역 수
   - 가장 최근 확인일
4. 현재 공개 공고 목록
   - 공고명, 경력, 고용 형태, 근무지, 접수 정보
   - 필수·우대·언급 기술
   - 공고 분석과 공식 원문 경로
5. 채용 근거 패널
   - 공고별 중복을 제거한 요구 기술 집계
   - 필수·우대·언급 횟수
   - 경력, 고용 형태, 근무지 분포
   - 집계 방법과 데이터 정책 경로

## 데이터 계약과 흐름

1. 백엔드 `PostingSummary`에 `company_slug`를 추가한다.
2. 데이터베이스와 검색 인덱스 양쪽 응답이 동일한 slug를 반환하게 한다.
3. 프론트엔드 런타임 계약은 배포 순서에 안전하도록 `company_slug`를 선택 필드로 먼저 수용한다.
4. 기존 공고 목록과 상세는 slug가 있을 때만 기업 상세 링크를 노출한다.
5. 기업 페이지는 `GET /api/postings?company={slug}&limit=100`으로 조회한다.
6. 순수 모델이 공고별 기술 중복을 제거하고, 동일 기술에 여러 요구 유형이 있으면 `required > preferred > unspecified` 우선순위로 한 번 집계한다.
7. 서버 페이지가 검증된 데이터만 컴포넌트에 전달한다.

검색 결과의 `total`은 현재 API가 반환한 범위와 같으므로 “현재 확인된 공개 공고”로 표현하고 기업의 전체 채용 규모라고 주장하지 않는다.

## 신뢰 경계

- 기업명, 공고명, 날짜, 근무지, 경력, 고용 형태, 요구 기술은 API 값만 사용한다.
- 공식 링크는 기존 HTTP(S) 런타임 검증을 통과한 `source_url`만 사용한다.
- 로고는 기업명 별칭과 신뢰된 공식 출처 호스트가 모두 맞는 기존 레지스트리만 사용한다.
- 로고 확인이 안 되면 기업명 이니셜 마크를 사용한다.
- 회사 소개, 직원 수, 연봉, 평점, 투자 단계, 성장률은 표시하지 않는다.
- 기술 수요는 공고 단위 확인 횟수로 명명하고 시장 전체 수요로 표현하지 않는다.

## 상태와 오류 처리

- slug는 소문자 영문·숫자·하이픈만 허용하며 잘못된 경로는 404로 처리한다.
- API 오류 또는 계약 오류는 “공고 데이터를 불러오지 못했습니다”로 표시하고 0건이라고 주장하지 않는다.
- 정상 응답이 0건이면 “현재 확인되는 공개 공고가 없습니다”로 표시한다. 삭제된 기업인지 일시적 공고 공백인지 추정하지 않는다.
- 기술 근거가 없으면 확정 임계값을 통과한 기술이 없다고 명시한다.
- 날짜·지역·경력·고용 형태가 없으면 각각 “미기재”로 표시한다.
- API 오류 상태에는 현재 기업 경로 재시도와 전체 공고 탐색 링크를 제공한다.

## 반응형 및 접근성

- 데스크톱은 공고 목록과 약 22rem 근거 패널의 2열을 사용한다.
- 960px 이하에서는 단일 열로 전환하고 근거 패널을 공고 목록 아래에 둔다.
- 680px 이하에서는 기업 요약, 공고 사실, 분포 행을 1열로 바꾼다.
- 공통 모바일 하단 내비게이션과 겹치지 않도록 고정 하단 액션을 추가하지 않는다.
- 버튼과 링크는 최소 44px 터치 영역과 명확한 `focus-visible`을 제공한다.
- 집계는 `dl`, 공고와 기술은 의미 있는 제목 및 목록 구조를 사용한다.
- 기업 상세에서는 전역 “공고” 내비게이션을 현재 위치로 표시한다.

## 시각 방향

기존 공고 탐색과 같은 평면적이고 근거 중심인 디자인 시스템을 이어간다. 밝은 중립 배경, 얇은 경계, 절제된 보라색 강조, 작은 상태 라벨을 사용한다. 기업 로고를 히어로의 시각 앵커로 두되 과장된 배너, 그라디언트, 유리 효과, 허위 그래프는 사용하지 않는다.

## 테스트 기준

- 백엔드: 목록·상세·데이터베이스 요약이 `company_slug`를 반환한다.
- 계약: 선택 slug를 보존하고 기존 slug 없는 응답도 안전하게 정규화한다.
- 순수 모델: 기술 우선순위, 공고별 중복 제거, 분포, 최근 확인일을 검증한다.
- 서버 페이지: 유효 slug, 오류, 0건, 실제 공고, 메타데이터를 검증한다.
- 연결: 공고 목록·상세의 기업명 링크와 전역 공고 내비게이션 활성 상태를 검증한다.
- 브라우저: fixture API의 company 필터, 기업 진입, 공고 상세 이동, 공식 원문, 1440/820/390px 오버플로 0, 44px 액션을 검증한다.
- 전체 단위 테스트, TypeScript, lint, Playwright, Vercel 운영 환경 빌드를 통과한다.

- 기업 팔로우와 알림
