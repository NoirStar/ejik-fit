# 이메일 자격 증명과 계정 프로필 설계

## 1. 목표

이 단계는 현재 이메일 매직링크 인증을 이메일·비밀번호 인증으로 확장하고, 이메일
확인과 공개 닉네임 프로필을 추가한다. 비로그인 상태에서 만든 커리어 데이터는 첫
로그인 때 계정 데이터와 병합하고, 로그인 이후에는 서버 저장값을 기기 간 원본으로
사용한다.

같은 배포 단위에서 커뮤니티 상세 화면의 제목 폭과 중복 미리보기 문제도 수정한다.
상세 화면은 홈 카드의 요약을 본문처럼 다시 보여주지 않고 실제 본문만 표시한다.

## 2. 현재 상태와 확인된 문제

- `/login`은 `signInWithOtp`를 사용해 이메일 매직링크만 발송한다.
- `user_career_states`, `user_saved_job_searches`, `user_notifications`는 Supabase Auth의
  사용자 UUID를 소유자로 사용하며 RLS가 적용돼 있다.
- 내 기술, 커리어 조건, 저장 공고, 지원 단계와 관심 기업은 로그인 시
  `useAccountStateSync`가 브라우저 값과 서버 값을 병합한다.
- 커뮤니티 글, 댓글, 반응과 작성자 팔로우는 아직 브라우저 전용이다. 이 데이터의
  서버 영속화는 다음 독립 단계에서 구현한다.
- 커뮤니티 상세 제목은 데스크톱에서도 `34px`, `max-width: 22ch`로 제한돼 506px
  너비에서 두 줄로 나뉜다.
- 커뮤니티 상세 헤더의 회색 문장은 홈 피드의 `body` 또는 `summary`를 그대로 다시
  출력한다. 실제 상세 본문은 별도의 `detail.paragraphs`에 있어 정보가 중복된다.

## 3. 사용자 흐름

### 3.1 회원가입

1. 사용자는 로그인 화면에서 `회원가입` 탭을 선택한다.
2. 이메일, 비밀번호, 비밀번호 확인과 닉네임을 입력한다.
3. 클라이언트는 입력을 정규화·검증한 뒤 Supabase `signUp`을 호출한다.
4. 닉네임은 `raw_user_meta_data.nickname`에 전달한다.
5. Supabase 확인 메일의 링크는 `/auth/callback`으로 돌아온다.
6. 콜백은 인증 코드를 세션으로 교환하고 안전하게 검증된 `next` 경로로 이동한다.
7. 로그인된 앱 셸은 기존 브라우저 커리어 데이터와 빈 계정 데이터를 병합해 서버에
   저장한다.

확인 메일이 늦거나 만료됐을 때는 같은 화면에서 확인 메일을 다시 보낼 수 있다.
가입 성공 여부를 이용한 이메일 계정 존재 여부 추측을 줄이기 위해 성공·중복 안내는
개인정보를 과도하게 드러내지 않는 문구를 사용한다.

### 3.2 로그인

1. 사용자는 이메일과 비밀번호로 로그인한다.
2. `signInWithPassword` 성공 후 `next` 경로로 이동한다.
3. 이메일이 아직 확인되지 않았으면 확인 메일 재전송 행동을 제공한다.
4. 인증 실패는 이메일 존재 여부를 구분하지 않고 `이메일 또는 비밀번호를 확인해
   주세요.`로 표시한다.

### 3.3 기존 매직링크 계정

기존 사용자를 삭제하거나 새 계정으로 분리하지 않는다. 기존 이메일을 비밀번호 재설정
흐름에 입력하면 확인 링크를 거쳐 새 비밀번호를 설정할 수 있다. 프로필이 없는 기존
사용자는 계정 화면에서 닉네임을 설정한다.

### 3.4 비밀번호 재설정

1. 로그인 화면의 `비밀번호를 잊으셨나요?`에서 이메일을 입력한다.
2. `resetPasswordForEmail`이 `/auth/callback?next=/login?mode=update-password`로 연결되는
   메일을 보낸다.
3. 콜백이 세션을 만든 뒤 새 비밀번호 입력 화면으로 이동한다.
4. 새 비밀번호는 `updateUser({ password })`로 저장하고 계정 화면으로 이동한다.

## 4. 인증과 프로필 데이터 모델

로그인 식별자는 이메일이다. 별도의 로그인용 아이디는 만들지 않는다. 이메일은
Supabase Auth에만 보관하고 공개 프로필 테이블에 복제하지 않는다.

새 `user_profiles` 테이블은 다음 필드를 가진다.

- `user_id uuid primary key`: `auth.users(id)`를 `ON DELETE CASCADE`로 참조
- `nickname varchar(20) null`: 공개 닉네임. 앞뒤 공백 제거 후 2~20자
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

닉네임은 표시 이름이며 중복을 허용한다. 이번 단계에서는 `@아이디`, 사용자 검색,
멘션 또는 닉네임 기반 로그인을 만들지 않는다. 사용자 소유권은 항상 UUID로 판단한다.

`auth.users` 추가 시 실행되는 `SECURITY DEFINER` 트리거는
`raw_user_meta_data.nickname`이 길이 규칙을 통과할 때만 프로필에 저장한다. 기존
사용자에는 닉네임이 비어 있는 프로필을 역으로 생성한다. 프로필이 없거나 닉네임이
비어 있으면 계정 화면에 설정 행동을 표시한다.

RLS 정책은 다음과 같다.

- 공개·로그인 사용자는 `user_profiles`의 `user_id`, `nickname`을 읽을 수 있다.
- 로그인 사용자는 `auth.uid() = user_id`인 자기 행만 수정할 수 있다.
- 클라이언트는 프로필 행을 임의 사용자 UUID로 삽입하거나 삭제할 수 없다.
- 계정 삭제 시 `auth.users` 삭제가 프로필과 기존 개인 데이터에 연쇄 적용된다.

## 5. 입력 규칙

- 이메일: 브라우저 이메일 형식, 앞뒤 공백 제거, 최대 254자
- 비밀번호: 10~72자, 영문자와 숫자를 각각 1자 이상 포함
- 비밀번호 확인: 비밀번호와 완전히 일치
- 닉네임: 앞뒤 공백 제거 후 2~20자, 제어문자와 줄바꿈 금지

클라이언트 검증은 빠른 피드백용이다. 닉네임 길이와 형태는 DB 제약 및 프로필 저장
함수에서도 다시 검증한다. 비밀번호 원문은 앱 DB, 로그, 분석 이벤트 또는 브라우저
저장소에 기록하지 않는다.

## 6. 화면 구조

### 6.1 로그인 페이지

기존 한 장짜리 매직링크 폼을 다음 상태를 소유하는 단일 인증 패널로 교체한다.

- `로그인`: 이메일, 비밀번호, 로그인 버튼, 비밀번호 재설정 링크
- `회원가입`: 이메일, 비밀번호, 비밀번호 확인, 닉네임, 가입 버튼
- `이메일 확인`: 전송 주소, 재전송 버튼, 다른 이메일 사용
- `비밀번호 재설정 요청`: 이메일, 재설정 메일 발송
- `새 비밀번호 설정`: 비밀번호, 비밀번호 확인, 저장

탭은 URL의 `mode=signin|signup|reset|update-password`와 동기화한다. 새로고침과 뒤로
가기를 지원하고 `next`는 기존 `safeAuthNextPath` 검증을 그대로 사용한다. 제출 중에는
해당 폼만 잠그며 페이지 전체를 로딩 화면으로 바꾸지 않는다.

### 6.2 계정 페이지

로그인 사용자는 이메일을 읽기 전용으로 확인하고 닉네임을 수정할 수 있다. 동기화
대상 안내는 현재 실제 범위와 일치하게 유지한다. 닉네임 저장 실패, 세션 만료와
마이그레이션 미적용 상태를 각각 구분해 다음 행동을 제공한다.

### 6.3 커뮤니티 상세

- 목업 상세의 홈 미리보기 `lead`를 제거한다.
- 로컬 글 상세의 `이 브라우저에서 직접 작성...` 회색 문장도 제거한다.
- 제목 다음에는 작성자 정보가 오고 그 다음 실제 본문이 온다.
- 데스크톱 작업 영역은 본문 열이 약 856px, 보조 열이 최대 256px이 되도록 조정한다.
- 제목은 데스크톱 최대 30px, `max-width` 제한 없이 `word-break: keep-all`과
  `text-wrap: pretty`를 사용한다.
- 1440px에서 현재 기본 목업 제목은 한 줄이어야 한다. 좁은 화면과 실제 긴 제목은
  단어 경계에서 자연스럽게 줄바꿈한다.

## 7. 브라우저 저장과 서버 병합

비로그인 상태에서는 내 기술, 조건, 저장 공고, 지원 단계와 관심 기업을 계속 브라우저에
임시 저장한다. 가입 전에 기능을 체험할 수 있어야 하기 때문이다.

로그인 시 현재 `mergeAccountCareerState`의 합집합 병합을 유지한다.

- 배열 데이터는 중복을 제거해 합친다.
- 지원 단계처럼 동일 키가 충돌하면 타임스탬프가 없는 현재 계약에 맞춰 로그인하는
  브라우저의 값을 우선하는 기존 규칙을 사용한다.
- 서버 쓰기가 성공하기 전에는 브라우저 값을 지우지 않는다.
- 로그아웃 시 이전 계정의 데이터가 다음 사용자에게 보이지 않도록 현재와 같이 계정
  캐시를 비운다.
- 로그인 이후 브라우저 저장은 즉시 화면 반영을 위한 캐시이며 서버 동기화 실패가
  발생하면 계정 화면과 앱 셸에서 오류 상태를 표시한다.

커뮤니티 글·댓글·반응·저장·팔로우를 서버로 옮기는 작업은 다음 단계에서 별도 테이블과
소유권 정책으로 구현한다. 이번 단계에서 로컬 커뮤니티 데이터를 삭제하거나 실제 서버
게시물처럼 표시하지 않는다.

## 8. 오류와 보안 처리

- 인증 오류의 원문 객체와 Supabase 내부 메시지를 사용자 또는 로그에 노출하지 않는다.
- 로그인, 가입, 재전송과 재설정은 중복 제출을 막는다.
- `next`는 내부 절대 경로만 허용한다.
- 가입 확인과 비밀번호 재설정 콜백은 `Cache-Control: private, no-store`를 유지한다.
- RLS가 없는 상태에서 프로필 데이터를 공개하지 않는다.
- DB 마이그레이션은 PostgreSQL에서 `auth.users`, `auth.uid()`와 역할 존재를 확인한 뒤
  외래키·권한·정책·트리거를 생성한다. SQLite 테스트에서도 테이블과 제약은 생성된다.
- 새 비밀번호 설정 화면은 유효한 복구 세션이 없으면 재설정 요청 화면으로 안내한다.

## 9. 테스트 범위

모든 UI 세부를 TDD로 만들지 않는다. 보안과 데이터 손실 위험이 있는 경계만 자동화한다.

- 순수 검증 테스트: 이메일, 비밀번호, 비밀번호 확인, 닉네임, 안전한 mode와 next 처리
- 인증 폼 컴포넌트 테스트: `signUp`, `signInWithPassword`, `resend`,
  `resetPasswordForEmail`, `updateUser` 호출과 오류 상태
- 프로필 저장 테스트: 자기 프로필 조회·수정, 잘못된 닉네임 거부
- 마이그레이션 테스트: 전체 Alembic upgrade SQL 생성과 SQLite upgrade
- 기존 계정 병합 테스트: 비로그인 데이터가 로그인 뒤 사라지지 않는 기존 회귀 테스트
- 커뮤니티 상세 컴포넌트 테스트: 중복 lead 부재, 실제 본문 존재
- 브라우저 테스트: 1440px 제목 한 줄, 390px 가로 넘침 없음, 가입·로그인 탭 URL 상태
  유지

최종 검증은 웹 전체 단위 테스트, TypeScript, 프로덕션 빌드와 백엔드 마이그레이션 테스트를
실행한다. 전체 E2E는 배포 직전 한 번만 실행한다.

## 10. 배포 순서

1. 커뮤니티 상세 표시 수정과 집중 테스트를 커밋하고 `main`에 푸시한다.
2. `user_profiles` 모델·마이그레이션·RLS·트리거를 배포한다.
3. 가입·로그인·재설정·닉네임 UI를 배포한다.
4. 운영 Supabase Auth의 이메일 확인과 허용 redirect URL을 확인한다.
5. 신규 가입, 이메일 확인, 비밀번호 로그인, 기존 계정 재설정, 로컬 데이터 병합과 계정
   삭제를 운영 환경에서 검증한다.
6. 다음 설계 단위인 서버 커뮤니티 영속화를 시작한다.

각 단계는 독립적으로 롤백 가능한 커밋을 사용하고 Vercel 및 GitHub CI 상태를 확인한다.

## 11. 이번 단계에서 제외

- 별도 로그인 아이디와 `@handle`
- 소셜 로그인
- 서버 커뮤니티 글·댓글·반응·팔로우
- 이메일 또는 웹 푸시 알림
- 프로필 이미지 업로드
- 프로필 기반 공고 매칭
- 관리자·신고·콘텐츠 제재 도구

제외 항목 중 서버 커뮤니티와 운영 도구는 이어지는 독립 설계·배포 단계에서 구현한다.
