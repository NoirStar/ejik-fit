# 로컬 커리어 비교 조건 설계

## 목표

`내 커리어`에서 선택한 경력 조건과 희망 기술 분야를 화면 이동이나 새로고침 뒤에도 유지한다. 조건은 브라우저에만 저장하고, 비교 결과는 계속 현재 공식 공고 API와 현재 스킬 그래프의 분야 메타데이터로 계산한다.

## 데이터 경계

- 사용자 선택: 경력 조건, 희망 기술 분야만 localStorage에 저장한다.
- 기술 분야 선택지: `/api/skills/graph`의 현재 노드 `domains`에서 파생한 값만 정상 선택지로 인정한다.
- 비교 수치와 다음 기술: `/api/fit/analyze` 현재 응답만 표시한다.
- 저장된 분야가 현재 그래프 목록에 없으면 해당 값을 실제 비교 요청에 보내지 않는다.
- 그래프 요청이 일시 실패한 경우 저장된 희망 분야를 삭제하지 않고, 그 요청만 전체 분야로 비교한다.
- 합격 가능성, 전망, 연봉 또는 지원 결과를 저장하거나 추정하지 않는다.

## 저장 계약

- 키: `ejik-fit:career-preferences`
- 값: `{ careerCondition, targetDomain }`
- 허용 경력 조건: 전체(`""`), 신입, 경력, 신입·경력
- 분야 ID는 공백 제거 후 안전한 영문·숫자·밑줄·하이픈 형식, 최대 80자로 제한한다.
- 손상된 JSON, 알 수 없는 경력 조건, 잘못된 분야 ID는 기본값으로 복구한다.
- 같은 탭 변경과 다른 탭 `storage` 이벤트를 모두 구독한다.
- 저장소 쓰기가 실패하면 UI는 이전 실제 저장값으로 돌아가고 성공 안내를 하지 않는다.

## 재검증 상태

- 현재 목록에 존재: 저장된 분야를 비교 API의 `domains`에 보낸다.
- 현재 목록에 없음: 저장값을 정리하고 전체 분야로 다시 비교한다.
- 분야 목록 API 실패: 저장값을 보존하고 `현재 요청은 전체 분야`라고 안내한다.
- API 실패 중 사용자가 전체 분야를 선택하면 저장된 분야를 명시적으로 지울 수 있다.

## UI

- 비교 조건 패널에 `이 브라우저에 저장` 경계를 명확히 표시한다.
- 저장·정리 실패와 임시 API fallback을 보이는 상태 문구 및 스크린리더 안내로 제공한다.
- 저장된 선택은 새로고침, 동일 탭 다른 컴포넌트 변경, 다른 탭 변경에 동기화된다.
- 기존 44px select와 390/768/1440px 반응형 구조를 유지한다.

## 개인정보 삭제

- 개인정보 페이지에 새 키와 저장 내용을 설명한다.
- `이 브라우저의 저장 데이터 삭제`는 커리어 비교 조건도 함께 삭제하고 실제 키 삭제 여부를 검증한다.

## 검증

- 정규화, 손상 복구, 저장 실패, clear, 동일·다른 탭 동기화 단위 테스트
- 저장값 선행 hydration, 변경 유지, 현재 목록 재검증, 임시 API 실패 보존 컴포넌트 테스트
- 모바일에서 조건 선택→비교 요청→새로고침 유지 E2E
- 전체 Vitest, E2E, lint, build, audit 및 운영 실제 API 확인
