# 검증된 공고 상세 설계

## 목표

공고 목록에서 확인한 공식 공고 근거, 저장 상태, 보유 기술 비교가 상세 화면에서도 끊기지 않게 한다. 사용자는 이 화면에서 무엇을 요구하는 공고인지, 데이터가 언제 어디서 확인됐는지, 내 기술과 실제로 무엇이 겹치는지를 판단한 뒤 공식 채용페이지로 이동할 수 있어야 한다.

## 선택한 방향

근거 중심 본문과 결정 패널을 나눈 2열 상세를 사용한다. 단일 칼럼 편집형은 구현은 단순하지만 긴 원문에서 주요 근거와 행동이 멀어지고, 다중 탭 대시보드형은 데이터 규모에 비해 탐색 비용이 크다. 2열 구조는 본문 읽기 흐름을 유지하면서 저장, 지원, 검증 시각, 내 기술 교집합을 가까이 둔다.

사용자가 앞서 세부 설정 판단과 중단 없는 순차 구현을 명시적으로 위임했으므로, 이 방향을 승인된 작업 기준으로 적용한다.

## 정보 구조

1. 공고 탐색으로 돌아가는 경로
2. 기업 마크, 기업명, 공고명, 공개 상태
3. 경력, 고용 형태, 근무지, 접수 기간
4. 본문 칼럼
   - 요구 기술 요약: 필수, 우대, 공고 언급을 구분
   - 각 기술의 실제 원문 근거
   - 안전하게 구조화한 공고 원문 텍스트
5. 결정 패널
   - 공식 채용페이지 이동
   - 브라우저 로컬 저장/해제
   - 보유 기술과 공고 기술의 대소문자 비구분 정확 일치 교집합
   - 공식 출처와 마지막 검증 시각
   - 분석 방법 및 정보 정정 경로

예측 점수, 합격 가능성, 근거 없는 추천 문구, 생성 요약은 표시하지 않는다.

## 컴포넌트와 경계

- 서버 페이지는 API 조회, 런타임 계약 검증, 메타데이터, JSON-LD, 정적 공고 사실을 담당한다.
- `JobDetailActions` 클라이언트 컴포넌트는 저장 공고와 보유 기술 로컬 상태만 읽고 구독한다. API 사실을 수정하거나 재해석하지 않는다.
- `buildJobDetailEvidence` 순수 모델은 기술 그룹, 추출 기술 수, 보유 기술 교집합을 계산한다.
- `parsePostingDescription` 순수 모델은 원문의 `#` 제목, 글머리표, 줄바꿈만 구조로 보존한다. 텍스트를 요약하거나 새로운 제목을 만들지 않는다.
- `PostingDescription`은 파싱 결과를 시맨틱 제목, 문단, 목록으로 렌더링하며 HTML 문자열을 주입하지 않는다.
- 기업 마크는 목록과 같은 `CompanyMark`를 재사용하고 공식 로고가 없는 기업은 기존 이니셜 폴백을 사용한다.

각 클라이언트 단위는 공고 ID, 공고 기술명, 공식 URL처럼 필요한 최소 데이터만 props로 받는다.

## 데이터 흐름과 신뢰 경계

1. 서버가 `/api/postings/{id}`를 조회한다.
2. `normalizePostingDetail`이 날짜, URL, 배열, 기술 근거를 검증한다.
3. 검증된 데이터만 페이지와 구조화 데이터에 사용한다.
4. 클라이언트가 `ejik-fit:saved-job-ids`와 `ejik-fit:owned-skills`를 읽는다.
5. 저장/보유 기술 변경 이벤트를 구독해 목록, 홈, 커리어 화면과 같은 탭에서도 즉시 동기화한다.
6. 저장소 쓰기가 거부되면 유틸리티가 반환한 실제 저장 상태만 화면에 반영한다.

공식 지원 링크는 검증된 HTTP(S) URL만 사용한다. 공고 본문은 `description_html`을 직접 렌더링하지 않고 검증된 `description_text`만 텍스트 노드로 출력한다.

## 상태와 오류 처리

- 404는 기존 not-found 흐름을 유지한다.
- API 또는 계약 오류는 상세 전용 오류 화면에서 재시도, 공고 탐색, 데이터 정책 경로를 제공한다.
- 기술 근거가 없으면 확정 임계값을 통과한 기술이 없다는 사실을 명시한다.
- 보유 기술이 없으면 내 커리어에서 기술을 저장하는 경로를 제공한다.
- 보유 기술이 있지만 교집합이 없으면 0개라는 사실만 표시한다.
- 마감일이 없으면 `마감일 미기재`, 시작일도 없으면 `접수 기간 미기재`로 표현한다.
- 원문 텍스트가 비어 있으면 공식 원문 확인 링크와 함께 본문 미제공 상태를 표시한다.

## 반응형 및 접근성

- 넓은 화면은 최소 폭이 보장된 본문과 약 20rem 결정 패널의 2열을 사용한다.
- 패널은 데스크톱에서만 화면 안에 머무르는 sticky 요소로 사용한다.
- 900px 이하에서는 단일 열로 전환하고 패널을 제목 다음에 배치한다.
- 모바일에서는 고정 하단 버튼을 만들지 않아 공통 하단 내비게이션과 겹치지 않게 한다.
- 680px 이하의 사실 정보와 액션은 1열 또는 자연스러운 줄바꿈으로 전환한다.
- 모든 버튼과 링크는 최소 44px 터치 영역, 명확한 `focus-visible`, 텍스트를 포함한 상태 변화를 제공한다.
- 저장 버튼은 `aria-pressed`, 상태 알림은 `aria-live`, 기술 근거는 제목 구조와 목록을 사용한다.
- 장식 기업 마크는 기업명이 이미 텍스트로 제공되므로 중복 낭독하지 않는다.

## 시각 방향

목록 화면의 평면적이고 근거 중심인 시스템을 이어간다. 밝은 중립 배경, 얇은 경계, 절제된 보라색 강조, 과도하지 않은 라운드와 그림자를 사용한다. 제목은 현재 화면보다 작게 제한해 사실 정보와 기술 근거가 첫 화면에 들어오게 한다. 공식 지원은 브랜드 주요 버튼으로, 저장은 보조 버튼으로 구분한다.

## 테스트 기준

- 순수 모델: 기술 그룹, 중복 제거, 대소문자 비구분 보유 기술 교집합, 원문 블록 파싱
- 컴포넌트: 저장/해제 지속성, 같은 탭 변경 구독, 저장소 실패, 보유 기술 없음/0개/교집합 상태
- 서버 페이지: 메타데이터, JSON-LD, 안전한 공식 URL, 사실 정보, 신뢰 경로, 빈 본문
- 브라우저: 실제 fixture API를 이용한 상세 진입, 1440/820/600/390px 오버플로 0, 44px 액션, 저장 후 목록과 공유, 모바일 하단 내비게이션 비겹침
- 전체 단위 테스트, TypeScript, lint, Playwright, Vercel 운영 환경 빌드
