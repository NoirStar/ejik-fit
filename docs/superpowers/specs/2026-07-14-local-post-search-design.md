# 브라우저 전용 글 통합 검색 설계

## 문제

통합 검색의 커뮤니티 결과 타입은 이미 `mock | local` 출처를 표현하지만,
`/search`의 서버 스냅샷에는 mock 항목만 들어간다. 사용자가 방금 작성한 로컬 글은
홈과 상세, 저장 보관함에서 볼 수 있어도 제목·본문·태그로 다시 찾을 수 없다.

## 목표

- 현재 브라우저에 저장된 내 글을 통합 검색의 커뮤니티 결과에 합친다.
- 공고·기업·기술 결과와 수치는 계속 실제 API 응답만 사용한다.
- 내 글과 화면 검증용 mock 결과를 카드별 출처 라벨로 구분한다.
- 같은 탭과 다른 탭에서 글이 생성·삭제되면 검색 결과와 개수를 갱신한다.
- 서버 HTML이나 metadata에 로컬 글 제목·본문을 포함하지 않는다.

## 데이터 흐름

1. 서버는 기존처럼 실제 공고/기술 API와 mock 커뮤니티로 검색 스냅샷을 만든다.
2. 클라이언트는 첫 hydration 뒤 검증된 `ejik-fit:local-community-posts`만 읽는다.
3. 순수 모델 함수가 현재 query로 로컬 글의 제목·본문·태그를 검색한다.
4. 로컬 결과를 mock 결과 앞에 합치고 커뮤니티 개수와 전체 결과 유무만 갱신한다.
5. 공고·기업·기술 결과, 오류 상태, API 표본 수치는 변경하지 않는다.

로컬 글은 `local-*` ID, 길이 제한, 날짜·태그 검증을 통과한 기존 저장 계층의
값만 사용한다. 손상된 저장 값이나 임의 ID를 결과로 만들지 않는다.

## 검색과 정렬

- 검색은 대소문자를 구분하지 않고 제목, 본문, 태그, 카테고리, 로컬 작성자
  레이블을 대상으로 한다.
- 로컬 결과 내부는 정확한 제목/태그, 제목 접두사, 제목 포함, 태그 포함,
  본문 포함 순으로 정렬한다.
- 사용자가 소유한 로컬 결과를 mock 결과보다 먼저 표시한다.
- 같은 ID는 한 번만 표시한다.
- 전체 범위의 기존 결과 노출 제한과 커뮤니티 단독 범위의 전체 노출 계약은
  유지한다.

## 화면과 문구

- 커뮤니티 카드는 `내 로컬 글` 또는 `예시 콘텐츠` 라벨을 표시한다.
- 커뮤니티 섹션은 내 글은 현재 브라우저 전용이고, 예시는 실제 사용자 글이
  아님을 함께 설명한다.
- 검색 소개와 빈 상태는 커뮤니티 전체를 가리키도록 수정한다.
- API 오류가 있어도 일치하는 로컬 글은 표시하고 실제 데이터 오류 안내는
  그대로 유지한다.

## 개인정보와 SEO

- 로컬 글은 브라우저에서만 검색하며 서버 요청 payload로 전송하지 않는다.
- 서버 metadata는 query만 사용하고 로컬 글 내용은 사용하지 않는다.
- 검색 페이지의 기존 `noindex` 정책을 유지한다.

## 접근성과 반응형

- hydration 전후 결과 개수는 기존 탐색 링크의 접근 가능한 이름에 반영된다.
- 출처는 색뿐 아니라 텍스트로 표시한다.
- 카드 제목·태그 링크는 기존 44px 터치 영역을 유지한다.
- 1440px, 820px, 390px에서 가로 넘침이 없어야 한다.

## 검증 기준

- 모델이 제목·본문·태그 일치 로컬 글을 찾고 mock 앞에 합친다.
- 원본 서버 스냅샷과 실제 API 결과는 변경하지 않는다.
- 작성한 글을 검색하고 상세로 이동하며 새로고침 뒤에도 다시 찾는다.
- 삭제·다른 탭 변경 뒤 결과와 개수가 갱신된다.
- mock 결과는 계속 명시적으로 예시 라벨을 유지한다.
