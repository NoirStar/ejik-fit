# 내 커리어 현황 설계

## 목표

`/career`의 준비 중 셸을 브라우저에 저장한 보유 기술과 실제 공개 공고 분석을 연결하는 개인화 화면으로 교체한다.

사용자는 이 화면에서 다음 질문에 답할 수 있어야 한다.

1. 이 브라우저에 어떤 기술을 저장했는가?
2. 저장한 기술과 한 개 이상 겹치는 현재 공개 공고가 몇 개인가?
3. 그중 명시된 필수 기술의 절반 이상을 보유한 공고가 몇 개인가?
4. 실제 공고에서 반복해 부족 요구사항으로 확인된 다음 기술은 무엇인가?
5. 어느 기술 분야에서 보유 기술과 부족 기술이 함께 나타나는가?

## 확인된 기존 계약

- 보유 기술은 `ejik-fit:owned-skills` 키로 현재 브라우저의 `localStorage`에만 저장된다.
- `POST /api/fit/analyze`는 `owned_skills`, 선택적 `career_type`, 선택적 `domains`를 받는다.
- `matching_posting_count`는 보유 기술과 한 개 이상 겹치는 공개 공고 수다.
- `strong_fit_posting_count`는 필수 기술이 명시된 공고 중 그 절반 이상을 보유한 공고 수다.
- 추천 기술은 부족 필수·우대 기술의 실제 공고 등장 횟수를 제공한다.
- 분야 분기는 보유 기술, 부족 필수·우대 기술, 근거 공고 수를 제공한다.
- `GET /api/skills/stats`는 현재 공고 수 상위 기술을 제공하므로 기술 입력 제안에 사용할 수 있다.

## 데이터 정직성

- `strong_fit_posting_count`를 합격 가능성, 적합도 점수, 지원 가능 공고로 부르지 않는다.
- 전체 공고 수나 전체 시장 비율로 과장하지 않는다.
- 추천 기술은 학습 순서나 선수 과목이 아니라 현재 겹치는 공고의 부족 요구사항이라고 설명한다.
- 공고 비교 기간 데이터가 없으므로 상승·하락·예측을 표시하지 않는다.
- 사용자가 입력하지 않은 이름, 직무, 연차, 회사, 연봉, 지원 상태를 만들지 않는다.
- API가 없는 희망 연봉, 저장 공고, 지원 관리 기능은 이번 범위에 포함하지 않는다.

## 화면 구조

### 상단 소개

- 제목: `내 커리어`
- 설명: 저장한 기술과 공식 공고 요구사항을 비교한다고 명시한다.
- 배지: `이 브라우저에만 저장`

### 기술 관리 패널

- 기술 직접 입력과 추가 버튼
- 현재 저장 기술과 개별 삭제, 전체 삭제
- 실제 기술 통계 API에서 받은 상위 기술을 빠른 추가 제안으로 표시
- 저장 기술이 없으면 첫 기술 추가를 안내
- 헤더의 `내 스택` 시트에서 바꾼 기술도 즉시 동기화

### 분석 조건

- 전체, 신입, 경력, 신입·경력 중 한 경력 조건을 선택한다.
- 선택은 현재 화면의 적합도 API 요청에만 사용한다.
- `mixed`는 `신입·경력`이며 `경력 무관`으로 표현하지 않는다.

### 현재 비교 요약

- `겹치는 공개 공고`: `matching_posting_count`
- `필수 기술 절반 이상`: `strong_fit_posting_count`
- `다음 준비 후보`: 추천 기술 개수
- 각 지표의 정의를 화면에 바로 설명한다.

### 다음 준비 기술

- 최대 6개를 표시한다.
- 기술명, 근거 공고 수, 부족 필수·우대 등장 횟수를 제공한다.
- 기술명은 스킬맵, 관련 공고는 공고 검색으로 연결한다.
- 보유 기술로 추가하는 버튼은 제공하지 않는다. 학습 완료를 서비스가 추측하지 않는다.

### 분야별 근거

- 최대 4개 분야를 표시한다.
- 근거 공고 수, 보유 기술, 부족 필수 기술, 부족 우대 기술을 구분한다.
- 알려진 분야 키는 한국어로 표시하고 알 수 없는 값은 원문을 보존한다.

### 기준 안내

- 공식 공개 공고와 확정된 기술 추출만 사용한다고 설명한다.
- 합격 가능성이나 장기 예측이 아니라고 명시한다.
- 스킬맵, 공고, 분석 방법으로 이동할 수 있다.

## 상태 처리

- 초기 저장소 읽기: 짧은 로딩 상태를 제공한다.
- 기술 없음: 기술 입력과 실제 상위 기술 제안을 유지한다.
- 분석 중: 이전 결과를 지우고 진행 상태를 알린다.
- 분석 실패: 내부 URL이나 원시 오류를 노출하지 않고 다시 시도 버튼을 제공한다.
- 결과 없음: 현재 조건에서 확인된 겹침이 없다고 설명하고 전체 조건 및 스킬맵 이동을 제안한다.
- 기술 제안 API만 실패해도 직접 입력과 적합도 분석은 계속 동작한다.

## 동기화

`writeOwnedSkills`와 `clearOwnedSkills`는 같은 탭의 변경을 알리는 전용 브라우저 이벤트를 보낸다. `/career`는 전용 이벤트와 다른 탭의 `storage` 이벤트를 모두 구독한다. 가짜 서버 저장이나 로그인 동기화를 만들지 않는다.

## 반응형과 시각 원칙

- 데스크톱은 20rem 기술 관리 열과 유연한 분석 열을 사용한다.
- 960px 이하에서는 단일 열로 전환한다.
- 640px 이하에서 지표와 추천 기술 행을 한 열로 배치한다.
- 필터, 버튼, 링크의 실제 터치 영역은 최소 44px이다.
- 홈과 시장 화면의 semantic token, 평평한 흰 표면, 약한 테두리, 제한된 보라색을 사용한다.
- 그라데이션, 글로우, 장식용 차트, 가짜 프로필 이미지는 사용하지 않는다.

## 접근성

- 입력에는 영구 label과 오류 연결을 제공한다.
- 분석 상태는 `aria-live`로 알린다.
- 현재 경력 조건은 select의 값으로 명확히 표시한다.
- 기술·추천·분야 목록은 목록 의미를 유지한다.
- 오류와 빈 상태는 제목과 다음 행동을 제공한다.
- 키보드 포커스와 reduced motion을 지원한다.

## 파일 경계

- `app/career/page.tsx`: metadata, 실제 상위 기술 요청, 클라이언트 화면에 제안 상태 전달
- `features/career/model.ts`: 경력 조건, 분야 레이블, 링크, 적합도 응답을 화면 모델로 변환
- `features/career/career-overview.tsx`: 저장 기술 관리, 적합도 요청, 상태와 의미 구조
- `features/career/career-overview.module.css`: 전용 반응형 레이아웃
- `lib/owned-skills.ts`: 같은 탭과 다른 탭을 위한 저장 변경 구독

## 검증 기준

- 저장 기술 추가·삭제·전체 삭제와 헤더 시트 동기화
- 경력 조건별 적합도 요청 payload
- 실제 응답 지표, 추천, 분야 링크와 레이블
- 로딩, 오류, 겹침 없음, 기술 없음 상태
- 1440px, 820px, 390px에서 가로 넘침 없음
- 최소 44px 상호작용 영역
- 콘솔 오류 없음
- 전체 테스트, TypeScript, Playwright, Vercel production build 통과
