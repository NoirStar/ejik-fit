# 이직핏 서비스 UI 밀도 리디자인 설계

## 상태

- 작성일: 2026-07-14
- 범위: 전역 브랜드 워드마크, 타이포그래피, 앱 셸, 홈, 탐색형 페이지, 상세 페이지, 반응형 표현
- 승인된 방향: 정보 추가 없이 현재 기능과 데이터의 위계·밀도·가독성을 실제 커뮤니티 및 채용 서비스 수준으로 재정리
- 제품 성격: 개발자와 경력직 사용자를 위한 데이터 기반 커리어 커뮤니티
- 디자인 다이얼: 시각 변주 4 / 모션 2 / 정보 밀도 7

이 문서는 기존 기능 설계와 데이터 신뢰 경계를 유지하면서 시각 체계와 화면 배치를 갱신한다. 이전 문서와 충돌할 경우 브랜드 표현, 글자 크기, 여백, 카드 표현, 반응형 배치에는 이 문서를 우선 적용한다. API 계약, mock 허용 범위, 라우팅 및 기존 사용자 저장 상태는 기존 기능 설계를 우선한다.

## 배경과 문제

현재 홈은 커뮤니티 서비스에 가까워졌지만 다른 주요 페이지에는 다음 문제가 남아 있다.

1. 일반 페이지 제목이 56px에서 76px까지 커져 실제 서비스 화면보다 랜딩 페이지처럼 보인다.
2. 보라색 eyebrow, 대형 제목, 소개 문장, 지표 카드가 여러 라우트에서 반복되어 생성형 AI 템플릿 같은 인상을 준다.
3. 공고, 검색 결과, 시장 데이터처럼 사용자가 찾는 실제 내용이 첫 화면 아래로 밀린다.
4. 흰색 둥근 카드가 모든 정보 단위에 반복되어 정보 위계보다 박스가 먼저 보인다.
5. Pretendard 선언 뒤에 전역 body가 Geist와 시스템 글꼴로 다시 덮여 한글과 영문 인상이 일관되지 않다.
6. 페이지별 최대 폭, 상단 여백, 제목 크기, 패널 반경이 달라 하나의 제품처럼 정렬되지 않는다.
7. 모바일 공고 제목이 한글 음절 단위로 끊기고, 그래프나 소개 영역이 첫 화면을 과도하게 점유한다.
8. 알림 메뉴가 상단 탭과 시각적으로 겹쳐 레이어의 소속과 클릭 영역이 불분명하다.
9. 현재 기하형 심볼과 영문 보조 표기가 사용자가 원하는 친근한 한글 브랜드와 맞지 않는다.

## 외부 밀도 조사

2026-07-14에 1440x900 데스크톱과 390x844 모바일 뷰포트에서 공식 서비스의 실제 계산 스타일을 확인했다.

| 서비스 | 주요 글 제목 | 본문 또는 요약 | 메타 정보 |
| --- | ---: | ---: | ---: |
| [OKKY](https://okky.kr/) | 14px | 기본 16px | 12~14px |
| [GeekNews](https://news.hada.io/) | 약 16.5~19px | 약 14.8px | 약 13.2px |
| [Disquiet](https://disquiet.io/) | 16px | 16px | 14px |
| [Velog](https://velog.io/) | 16px | 14px | 12px |

조사의 결론은 본문을 무조건 작게 만드는 것이 아니다. 일반 커뮤니티는 14~16px의 읽을 수 있는 본문을 유지하면서 페이지 상단의 거대한 제목과 과도한 소개 여백을 거의 사용하지 않는다. 이직핏도 실제 콘텐츠 제목, 본문, 메타 정보 사이의 작은 단계 차이로 위계를 만든다.

Reddit, Stack Overflow, Wanted Community, Blind는 조사 환경의 네트워크 또는 보안 차단으로 실제 화면을 확인할 수 없어 수치 근거에서 제외했다.

## 목표

1. 모든 주요 라우트가 같은 브랜드, 글꼴, 폭, 간격 체계를 사용하는 실제 운영 서비스처럼 보여야 한다.
2. 정보나 기능을 늘리지 않고 중복 소개, 중복 수치, 과도한 여백만 줄인다.
3. 홈은 커뮤니티 읽기 흐름을 유지하고, 공고·시장·스킬맵·커리어는 업무형 데이터 화면으로 정리한다.
4. 데스크톱 첫 화면에는 실제 게시물, 공고, 시장 데이터 중 해당 페이지의 주 콘텐츠가 보여야 한다.
5. 모바일 390px에서도 제목이 낱글자로 끊기거나 가로 스크롤이 생기지 않아야 한다.
6. 공고와 시장 수치는 계속 실제 API 데이터만 사용하고 커뮤니티 mock 경계를 유지한다.
7. 상호작용, URL 상태, 로컬 저장 상태, 오류 상태와 접근성을 보존한다.

## 비목표

- 새로운 지표, 추천 알고리즘, API 또는 커뮤니티 기능 추가
- 공고·스킬 시장 수치를 mock 데이터로 보강
- 새로운 다크 모드 설계
- 이번 작업에서 GitHub Octocat처럼 독립적인 마스코트 또는 favicon 제작
- 모든 회사 로고를 새로 수집하거나 외부 로고 API를 런타임에 추가
- 장식 목적의 차트, 3D 그래픽, 그라데이션 hero 또는 큰 마케팅 문구 추가

## 브랜드 방향

### 한글 워드마크

헤더 브랜드는 심볼과 영문 부제를 제거하고 한글 이직핏 워드마크를 사용한다.

- 이직: 짙은 잉크색
- 핏: 절제된 단일 포인트 바이올렛
- 전체 형태: 획 끝이 부드럽고 글자별 높이와 기준선에 아주 작은 변주가 있는 말랑한 인상
- 읽기성: 작은 헤더에서도 즉시 이직핏으로 읽혀야 함
- 금지: 무지개색, 강한 그라데이션, 과도한 기울임, 캐릭터 같은 표정, 기하형 AI 심볼, 영문 EJIK FIT 부제

워드마크는 일반 본문 글꼴에 색만 입힌 표현이 아니라 헤더 전용 SVG 또는 동등한 고정형 자산으로 만든다. 다만 장식 때문에 한글 자형이 무너지지 않게 한다. 헤더에서 워드마크의 시각 높이는 데스크톱 약 26px, 모바일 약 24px를 목표로 한다.

### favicon

독립 마스코트 favicon은 별도 브랜드 작업으로 미룬다. 이번 리디자인에서는 기존 favicon을 헤더 로고로 재사용하지 않으며, favicon 교체 때문에 리디자인 배포를 지연하지 않는다.

### 회사 로고

기존 공식 로고 자산과 resolver를 유지한다.

- 로고가 확인된 회사: 고정 프레임 안에 공식 로고 표시
- 로고가 없거나 불명확한 회사: 회사명 기반 이니셜 대체
- mock 커뮤니티 콘텐츠: 실제 기업 로고를 사실처럼 사용하지 않음
- 런타임 Google 이미지 검색이나 출처 불명 이미지를 추가하지 않음

## 시각 시스템

### 글꼴

전역 제품 글꼴은 Pretendard Variable로 통일한다. body에서 Geist 또는 별도 시스템 글꼴이 Pretendard를 덮지 않게 정리한다.

- 기본 가중치: 400
- 보조 강조: 500
- 항목 제목: 600
- 페이지 제목: 700
- 800과 900은 워드마크 자산을 제외한 일반 UI에서 사용하지 않음
- 숫자 표와 시장 지표에는 tabular numbers 적용
- 한글 제목은 word-break: keep-all을 기본으로 하고 긴 영문·URL만 안전하게 줄바꿈

### 타이포그래피 규격

#### 데스크톱

| 용도 | 크기 | 행간 | 가중치 |
| --- | ---: | ---: | ---: |
| 일반 페이지 제목 | 30~32px | 1.25 | 700 |
| 공고·게시글 상세 제목 | 최대 34px | 1.3 | 700 |
| 섹션 제목 | 19~21px | 1.35 | 600~700 |
| 피드·공고 항목 제목 | 16~18px | 1.45 | 600 |
| 일반 본문 | 15px | 1.6~1.65 | 400 |
| 설명·요약 | 14px | 1.55 | 400 |
| 메타·날짜·태그 | 12~13px | 1.5 | 400~500 |
| 핵심 수치 | 최대 24~28px | 1.2 | 600~700 |

#### 모바일

| 용도 | 크기 | 행간 |
| --- | ---: | ---: |
| 일반 페이지 제목 | 26~28px | 1.3 |
| 상세 제목 | 최대 28px | 1.35 |
| 섹션 제목 | 18~19px | 1.4 |
| 항목 제목 | 16~17px | 1.45 |
| 일반 본문 | 15px | 1.65 |
| 긴 게시글 본문 | 16px | 1.7 |
| 메타 정보 | 12~13px | 1.5 |

설명 문장은 넓은 화면에서도 지나치게 길게 늘이지 않는다. 긴 본문이 아닌 UI 설명은 두 줄을 넘기지 않는 것을 기본으로 한다.

### 색상

라이트 UI를 기준으로 다음 역할 토큰을 사용한다.

- 앱 배경: #F7F7FA
- 기본 표면: #FFFFFF
- 강한 텍스트: #17171C
- 보조 텍스트: #62626D
- 약한 텍스트: #8B8B96
- 기본 선: #E7E7EC
- 약한 선: #EFEFF3
- 포인트 바이올렛: #6D4BE8
- 포인트 배경: #F1EDFF
- 성공 또는 적합 신호: 기존 의미 토큰을 유지하되 장식용으로 사용하지 않음
- 위험 또는 하락 신호: 실제 상태와 수치에만 사용

포인트 색은 워드마크의 핏, 활성 탭, 핵심 링크, 선택 상태, 주요 CTA에만 쓴다. 보라색 그라데이션, glow, 넓은 색면 hero는 사용하지 않는다.

### 표면과 형태

- 일반 카드 반경: 10~12px
- 큰 독립 패널 반경: 최대 16px
- 버튼과 입력 반경: 8~10px
- pill 형태: 상태, 기술 태그, 범주처럼 의미가 있는 짧은 값에만 사용
- 일반 콘텐츠 목록: 카드 반복보다 하나의 표면과 구분선 사용
- 그림자: 알림·메뉴·dialog 같은 떠 있는 레이어에만 사용
- 테두리: 1px 중립선, 중첩 테두리 최소화

### 간격

4px 단위를 기본으로 한다.

- 데스크톱 페이지 상단: 28~32px
- 모바일 페이지 상단: 20~24px
- 주요 섹션 사이: 24~32px
- 목록 행 패딩: 세로 16~20px
- 일반 패널 패딩: 16~20px
- 모바일 좌우 여백: 16px
- 컨트롤 최소 높이: 40px
- 터치 대상 최소 높이와 너비: 44px

## 공통 앱 셸

### 데스크톱 헤더

한 줄 64px 헤더를 기본으로 한다.

1. 한글 이직핏 워드마크
2. 통합 검색
3. 홈, 시장, 스킬맵, 공고, 내 커리어
4. 글쓰기, 알림, 사용자 메뉴

현재 위치는 텍스트 색과 얇은 하단 표시만 사용한다. 메뉴마다 배경 pill을 만들지 않는다. 헤더 내부 최대 폭과 본문 최대 폭은 같은 기준선에 맞춘다.

### 모바일 셸

- 상단 바: 56px, 워드마크·검색 진입·알림
- 주요 탐색: 기존 모바일 하단 내비게이션 유지
- 하단 safe area와 본문 패딩 보장
- 작은 화면에서 데스크톱 메뉴를 억지로 축소하지 않음

### 폭과 정렬

- 전역 셸 최대 폭: 1280px
- 전역 데스크톱 좌우 여백: 24px
- 넓은 표·그래프: 셸 폭 안에서 사용
- 읽기 본문: 680~720px
- 공고 상세: 본문 약 720~760px + 보조 패널

## 홈

홈은 hero가 없는 커뮤니티 피드 첫 화면이다.

### 데스크톱 구조

- 최대 폭 1280px
- 왼쪽 레일 약 216px
- 중앙 피드 minmax(0, 1fr)
- 오른쪽 레일 약 280px
- 열 사이 간격 20px

왼쪽에는 현재 프로필·관심 기술·기존 바로가기만 유지한다. 중앙 피드가 가장 강한 대비와 폭을 가진다. 오른쪽에는 현재 인기 주제·기술 트렌드·커리어 인사이트·팔로우 글을 유지하되 카드 수와 내부 여백을 줄인다.

### 중앙 피드

- 추천, 팔로잉, 최신, 인기 탭 높이 42~44px
- 하나의 흰 피드 표면 안에서 게시물을 얇은 구분선으로 분리
- 항목 제목 17~18px
- 요약 14px
- 작성자·시간·반응 12~13px
- 기본 행 세로 패딩 18~20px
- 이미지·시장 인사이트·공고처럼 구조가 다른 항목만 부분 강조
- 모든 게시물에 동일한 외곽 카드와 그림자를 반복하지 않음
- 첫 900px 높이에서 실제 게시물 2~3개가 보여야 함

### 모바일

- 중앙 피드를 가장 먼저 표시
- 좌우 레일의 보조 정보는 피드 뒤로 이동하거나 현재 접기 동작을 사용
- 피드 좌우 여백 16px
- 핵심 게시물 이미지가 화면 폭을 밀어내지 않음
- 작성·저장·공감 버튼은 최소 44px 터치 영역 유지

## 탐색형 페이지

탐색형 페이지는 큰 마케팅 hero 대신 제목, 조작, 결과의 순서로 구성한다.

### 채용 공고

1. 30~32px 채용 공고 제목과 짧은 보조 설명
2. 검색·경력·기술 필터
3. 결과 수와 정렬
4. 실제 공고 목록

대형 지표 카드와 소개 영역을 결과 위에 반복하지 않는다. 공고 행은 회사 로고 40px, 공고명 17px, 회사·지역·경력·기술 13~14px, 저장과 상태 액션으로 구성한다. fit 정보가 존재할 때만 작고 명확한 상태로 표시한다.

데스크톱 900px 높이에서 필터와 최소 2개 공고가 보이고, 모바일 첫 화면에도 최소 1개 공고가 보여야 한다.

### 검색 결과

- 헤더의 통합 검색을 주 검색 수단으로 사용
- 본문에는 검색어가 포함된 26~30px 결과 제목과 결과 수 표시
- 같은 거대한 검색 입력을 본문에 중복하지 않음
- 기존 공고·게시글·기업 분류와 링크 동작 유지
- 결과가 없는 경우 짧은 안내와 기존 다음 행동만 표시

### 기술 시장

- 30~32px 제목과 데이터 확인 기준
- 핵심 수치는 개별 대형 카드가 아니라 구분선이 있는 한 줄 요약
- 실제 API 차트와 기술 데이터가 첫 화면에 진입
- 차트 제목 18~20px, 축·범례·표 메타 12~13px
- 상승과 하락 색은 실제 수치 의미에만 사용
- API 실패 시 mock 차트나 수치로 대체하지 않음

### 스킬맵

- 30~32px 제목과 검색·선택 조작
- 데스크톱: 그래프 작업 공간과 선택 기술 상세를 나란히 배치
- 그래프 설정은 필요할 때 펼치는 보조 조작으로 유지
- 패널마다 중첩 카드를 만들지 않고 그래프·상세의 큰 두 영역으로 정리
- 모바일: 그래프 높이 약 300~340px로 제한하고 상세정보를 바로 아래 배치
- 그래프가 첫 화면 전체를 점유해 설명과 다음 행동을 밀어내지 않음
- 기존 실제 API 그래프 근거와 선택 상태 유지

### 내 커리어

- 30~32px 제목
- 현재 경력 상태, 관심 기술, 저장한 시장 조건, 추천 공고를 기존 데이터 범위에서 표시
- 대형 환영 hero와 홍보성 문구 제거
- 설정과 결과를 하나의 개인 작업 화면처럼 정렬
- 저장 상태와 URL 또는 localStorage 계약을 변경하지 않음

## 상세 페이지

### 공고 상세

- 회사 로고 48~56px
- 회사명과 확인 기준 13~14px
- 공고 제목 데스크톱 최대 34px, 모바일 최대 28px
- 본문 약 720~760px
- 데스크톱 지원·저장 영역은 우측 보조 패널
- 모바일 지원·저장 영역은 safe area를 고려한 하단 고정 바
- 제목에 word-break: keep-all과 안전한 overflow-wrap 적용
- 채용 같은 단어가 채 / 용처럼 음절 단위로 분리되지 않아야 함

### 기업 상세

- 로고 48~56px
- 기업명 약 30px
- 기본 정보, 기술, 채용 중인 공고를 바로 이어서 표시
- 로고가 없을 때 동일 크기 이니셜 대체
- 대형 브랜드 hero나 장식 배경을 추가하지 않음

### 게시글 상세

- 읽기 폭 680~720px
- 제목 30~32px, 모바일 최대 28px
- 작성자·작성일·반응 12~13px
- 본문 16px, 행간 약 1.7
- 댓글은 본문 아래 자연스럽게 이어지는 구분선 목록
- 공유·저장·공감은 의미가 분명한 아이콘과 접근 가능한 이름 제공

## 알림과 오버레이

알림 메뉴는 헤더의 알림 버튼에 기준을 둔 별도 레이어다.

- 데스크톱: 헤더 아래에 정렬된 우측 popover
- 모바일: 화면 좌우 여백을 지키는 폭 넓은 패널
- top 위치는 실제 헤더 높이에서 계산
- z-index는 본문 탭보다 높고 dialog보다 낮은 명시적 레이어 토큰 사용
- 최대 높이와 내부 스크롤 제공
- 상단 메뉴나 탭의 글자가 알림 패널 위로 비치거나 클릭 영역이 겹치지 않음
- Escape, 바깥 클릭, 트리거 포커스 복귀 유지
- 그림자는 이 레이어에서만 절제해 사용

## 상태 표현

- 로딩: 실제 목록 행과 패널 구조를 닮은 skeleton
- 빈 결과: 18~20px 이하 제목과 짧은 안내, 기존 다음 행동
- 오류: 실패한 데이터 영역 안에서 설명하고 나머지 화면은 유지
- 부분 API 실패: 확인 가능한 실제 데이터만 표시
- 긴 텍스트: 두 줄 말줄임 또는 읽기 본문 흐름 중 해당 맥락에 맞는 방식 사용
- 가짜 성공률, 가짜 증가율, 가짜 공고 수를 시각 장식으로 추가하지 않음

## 반응형 규칙

### 1280px 이상

- 전역 최대 폭 1280px
- 홈 3열
- 공고·시장·스킬맵은 넓은 작업 화면

### 약 840~1279px

- 홈 오른쪽 레일을 아래로 이동하거나 현재 우선순위 규칙에 따라 숨김
- 왼쪽 프로필은 축약
- 상세 페이지 보조 패널 폭 축소
- 검색 폭을 먼저 줄이고 주요 기능을 제거하지 않음

### 839px 이하

- 한 열 본문
- 16px 좌우 여백
- 56px 상단 바와 모바일 하단 내비게이션
- 표는 의미를 보존한 행 또는 가로 스크롤 영역으로 전환
- 고정 액션은 하단 내비게이션과 safe area를 침범하지 않음
- 어떤 라우트도 문서 전체 가로 스크롤을 만들지 않음

## 모션

- 기본 전환 시간: 120~180ms
- 허용: opacity, background-color, border-color, 작은 translate
- 금지: hero 등장 연출, 반복 pulse, 카드 부유 효과, 스크롤 hijack
- prefers-reduced-motion에서는 비필수 전환 제거

## 데이터와 기능 보존

이번 변경은 표현 계층의 리디자인이다.

- 커뮤니티 fixture와 로컬 작성 데이터는 기존 mock 경계 유지
- 공고, 기술 시장 수치, 스킬 그래프, 회사 정보는 기존 실제 API 경계 유지
- 필터와 검색 파라미터 유지
- 저장, 공감, 팔로우, 알림, 로컬 커리어 설정 동작 유지
- API가 제공하지 않는 정보는 새 UI를 채우기 위해 추정하지 않음
- 기존 링크와 라우트 호환성 유지

## 구현 구조 원칙

1. 전역 typography, color, spacing, radius, layer 토큰을 먼저 정리한다.
2. 앱 셸과 공통 PageHeader, SectionHeader, 목록 행, 상태 표현을 공용화한다.
3. 페이지 전체를 하나의 범용 카드 컴포넌트에 억지로 맞추지 않는다.
4. 홈 피드, 공고 행, 시장 표, 상세 본문처럼 의미가 다른 구조는 각 도메인 컴포넌트가 소유한다.
5. 기존 도메인 데이터 변환과 상호작용 로직은 시각 컴포넌트에서 분리한 상태를 유지한다.
6. CSS 우선순위를 정리해 뒤쪽 전역 선언이 Pretendard와 semantic token을 덮지 않게 한다.
7. 사용되지 않는 legacy dark token과 충돌 스타일은 현재 노출 기능을 확인한 뒤 제거한다.

## 접근성

- landmark와 제목 수준을 페이지 의미에 맞게 유지
- 현재 내비게이션과 탭 상태를 aria-current, aria-selected로 표현
- 아이콘 전용 버튼에 접근 가능한 이름 제공
- focus-visible을 포인트 색과 충분한 대비로 표시
- 최소 44px 터치 대상
- 색만으로 상승·하락·선택 상태를 표현하지 않음
- dialog와 popover의 포커스 이동·복귀 유지
- 본문 확대 200%에서도 콘텐츠와 기능 손실이 없어야 함

검수 항목:

1. 일반 페이지 제목의 계산 크기가 32px을 넘지 않는다.
2. 상세 제목은 데스크톱 34px, 모바일 28px을 넘지 않는다.
3. body의 계산 글꼴이 Pretendard 계열이다.
4. 1440x900 첫 화면에서 홈 게시물 2개 이상, 공고 목록 2개 이상이 보인다.
5. 390x844 공고 화면에서 실제 공고 1개 이상이 보인다.

7. 문서 전체 가로 스크롤이 없다.
8. 알림 패널이 헤더 탭과 겹치거나 그 아래로 비치지 않는다.
9. 실제 API 데이터의 출처·확인 시각·오류 상태가 유지된다.
10. 카드, pill, 보라색 강조가 의미 없는 장식으로 반복되지 않는다.

## 배포 방식

리디자인은 기능 회귀를 줄이기 위해 다음 단위로 구현하고 각 단위마다 테스트한다.

1. 전역 토큰, Pretendard 충돌, 워드마크, 앱 셸
2. 홈 피드와 좌우 레일
3. 공고·검색·시장·스킬맵·커리어
4. 공고·기업·게시글 상세와 알림 레이어
5. 전체 반응형·접근성·시각 회귀 검수

각 단계는 작은 의도적 커밋으로 남긴다. 전체 검증이 끝난 변경만 main에 병합하고 원격 main에 push해 Vercel 배포를 시작한다. 문서 커밋은 구현 코드와 분리한다.
