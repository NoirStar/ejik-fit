# 저장 보관함 디자인

## 목적

홈, 공고 목록, 공고 상세에서 저장한 항목은 이미 브라우저 `localStorage`에 남지만, 다시 모아 볼 수 있는 화면이 없다. `/career/saved`를 추가해 저장 행동을 완결된 탐색 흐름으로 만든다.

이 화면은 로그인 계정이나 서버 보관함을 가장하지 않는다. 저장 여부는 이 브라우저에만 남고, 저장한 공고의 내용은 화면을 열 때 현재 공식 공고 API에서 다시 확인한다. 커뮤니티는 실제 사용자 백엔드가 없으므로 기존 mock 글만 사용하고 이를 명시한다.

## 제품 원칙

- 저장 ID는 브라우저 로컬 상태다.
- 공고 제목, 기업, 조건, 기술, 출처, 확인 시각은 실제 `GET /api/postings/{id}` 응답만 사용한다.
- API에서 더 이상 확인되지 않는 저장 ID를 임의의 공고 카드로 복원하지 않는다.
- API 장애와 404를 구분한다. 404는 현재 확인 불가, 그 밖의 실패는 재확인 실패로 표시한다.
- 커뮤니티 글은 `예시 콘텐츠`와 `실제 사용자가 작성한 글이 아님`을 화면에서 숨기지 않는다.
- 저장 해제는 즉시 로컬 상태에 반영되고 다른 화면의 저장 상태와 같은 계약을 사용한다.
- 개인화 페이지이므로 검색엔진에 색인하지 않는다.

## 정보 구조

### 상단

- `브라우저 저장함` eyebrow
- `저장 보관함` 제목
- 로컬 저장과 API 재확인 방식을 설명하는 짧은 문장
- `내 기술 비교` 링크와 로컬 저장 배지

### 범위 전환

- 전체
- 공식 공고
- 커뮤니티 예시

각 범위에는 브라우저에서 확인한 현재 저장 개수만 표시한다. 서버 전체 수치처럼 표현하지 않는다.

### 공식 공고 영역

- 저장한 ID를 내부 POST route로 전달한다.
- route는 ID를 제한·정규화하고 실제 공고 상세 API를 조회한다.
- 성공한 항목만 카드로 표시한다.
- 카드에는 검증된 기업 로고 또는 안전한 이니셜, 기업명, 제목, 경력·고용·지역, 현재 상태, 마지막 확인 시각, 필수·우대·미분류 기술, 공고 분석 링크, 공식 원문 링크를 표시한다.
- 404 항목 수와 일시적 조회 실패 수를 별도로 고지한다.
- 일시적 실패에는 재시도 동작을 제공한다.

### 커뮤니티 영역

- 기존 mock 커뮤니티·면접 후기 중 저장 ID와 일치하는 항목만 표시한다.
- 작성자, 분류, 제목, 요약, 태그를 표시한다.
- 모든 카드와 섹션에 예시 콘텐츠임을 인지할 수 있는 라벨을 둔다.
- 현재 mock 목록에 없는 저장 ID가 있으면 개수만 알리고 가짜 내용을 만들지 않는다.

## 데이터 route

`POST /career/saved/data`

요청:

```json
{ "job_ids": ["posting-id"] }
```

제약:

- 객체 body만 허용
- `job_ids`는 문자열 배열
- 공백 제거 후 비어 있지 않고 200자 이하
- 중복 제거
- 최대 24개

응답:

```json
{
  "items": [],
  "unavailable_ids": [],
  "failed_ids": []
}
```

- `items`: 실제 API 응답을 표시 전용 필드로 축약한 값
- `unavailable_ids`: 실제 API 404
- `failed_ids`: 404가 아닌 조회 실패
- 응답은 `no-store`
- 공고 설명 HTML은 보관함 응답에 포함하지 않는다.

## 반응형과 접근성

- 데스크톱은 공고 목록을 주축으로 하고 커뮤니티 목록을 보조 열로 배치한다.
- 태블릿과 모바일은 단일 열로 전환한다.
- 모바일 하단 내비게이션과 겹치지 않도록 페이지 하단 여백을 유지한다.
- 모든 버튼·링크의 최소 클릭 영역은 44px이다.
- 로딩, 빈 상태, 부분 실패, 전체 실패를 텍스트와 `role=status` 또는 `role=alert`로 구분한다.
- 저장 해제 후 스크린리더 알림을 제공한다.
- 탭은 URL 이동이 아닌 동일 화면 범위 전환이며 키보드 화살표를 지원한다.
