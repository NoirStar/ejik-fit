# 출처 신뢰성과 공고 공개 상태 설계

## 목적

공식 채용 출처가 일시적으로 접근 불가해져도 공고를 곧바로 마감하지 않되,
오랫동안 재검증하지 못한 공고가 일반 검색·시장 통계·알림에 무기한 남지 않게
한다. 넥슨 그룹처럼 공개 브라우저 렌더링이 필요한 출처도 다음 정기 실행에서
자동으로 다시 검증할 수 있어야 한다.

## 현재 문제

운영 DB에는 191개 기업이 등록되어 있지만 넥슨 그룹 13개 출처가 비실행 상태로
남아 최근 전체 수집은 178개 출처만 실행한다. 넥슨 브라우저 커넥터 자체는
구현되어 있으나, 과거 `REVIEW/REVIEW` 전환 뒤 seed가 `ALLOWED/REVIEW` 상태를
만들었고 현재 복구 조건은 이 조합을 인식하지 못한다.

공고 API와 기술 통계는 `job_postings.status = OPEN`만 검사한다. 출처가 비실행
상태가 되어도 기존 공고 상태는 계속 `OPEN`이므로, 현재 민트로켓 2건과
딜로퀘스트 1건이 출처 디렉터리 집계에서는 제외되지만 일반 공고 API에는
남아 있다.

## 선택한 접근

새 PostgreSQL enum이나 별도 공개 상태를 추가하지 않는다. 초기 스키마부터
존재하지만 아직 사용하지 않는 `PostingStatus.DELAYED`를 검증 지연 상태로
사용한다.

1. 출처 수집이 실패해도 기존 공고는 즉시 닫지 않는다.
2. 비실행 출처의 마지막 성공 시각이 72시간보다 오래되면 해당 출처의
   `OPEN` 공고를 `DELAYED`로 전환한다.
3. `DELAYED` 공고는 일반 목록, 검색, 기술 통계, 스킬 그래프, 핏 분석,
   새 공고 알림에서 제외된다. 이 영역들은 이미 `OPEN`만 읽으므로 기존
   공개 쿼리 계약을 유지한다.
4. 직접 공고 상세 조회는 `DELAYED`를 계속 반환한다. 화면은 이를 마감으로
   오해하지 않고 `공식 출처 확인 지연`으로 표시한다.
5. 수집이 복구되면 실제 목록에서 다시 확인된 공고만 ingestion 과정에서
   `OPEN`으로 돌아온다.
6. 복구된 목록에서 보이지 않은 `DELAYED` 공고는 다시 `OPEN`으로 만들지
   않고, 정상 목록에서 세 번 연속 누락되면 `CLOSED`로 전환한다.

이 정책은 일시 장애에 72시간의 유예를 주면서도 오래된 공고가 실제 수요와
알림에 섞이는 것을 막는다.

## 출처 상태 복구

넥슨 seed 복구 조건은 다음 조건을 모두 만족하는 과거 자동 차단 상태를
복구한다.

- connector family가 넥슨 공개 브라우저 커넥터다.
- 마지막 오류 코드가 `blocked`다.
- 마지막 오류 사유가 넥슨 응답에서 발생한 것으로 식별된다.
- 현재 정책 상태가 `REVIEW` 또는 `BLOCKED`다.
- 운영자가 명시적으로 `STOPPED`로 바꾼 출처가 아니다.

복구 시 source와 policy를 seed의 `ALLOWED`로 맞추고 자동 오류를 지운다.
이후 정기 workflow의 일반 Playwright Chromium 수집이 다시 실행된다.
다시 접근 제한이 발생하면 기존 crawler 정책에 따라
`NEEDS_BROWSER/ALLOWED`로 기록하며, 다음 seed가 재검증 기회를 제공한다.
로그인, 프록시, stealth 또는 CAPTCHA 우회는 추가하지 않는다.

## 지연 전환 시점

지연 전환은 전체 수집의 모든 출처 실행이 끝난 뒤, 시장 스냅샷을 만들기 전에
한 번 수행한다. 기준 시각은 해당 실행이 사용하는 UTC 시각이며 cutoff는
정확히 72시간 전이다.

- `source.is_runnable`이 참이면 전환하지 않는다.
- `last_success_at`이 cutoff보다 최신이면 전환하지 않는다.
- 한 번도 성공하지 않은 출처는 기존 공개 공고가 있을 때만 지연 처리한다.
- `STOPPED`, `BLOCKED`, `REVIEW`, `NEEDS_BROWSER`, `NEEDS_CONNECTOR`처럼 현재
  실행할 수 없는 출처가 대상이다.
- 이미 `DELAYED` 또는 `CLOSED`인 공고는 변경하지 않는다.

함수는 전환한 공고 수를 반환하고 전체 수집 결과 및 GitHub Actions summary에
그 수를 포함한다.

## 상세 화면

공고 상세 API에는 기존 `status` 필드가 있으므로 계약을 바꾸지 않는다.
웹 상세 화면은 `status === "delayed"`일 때 다음 의미를 표시한다.

> 공식 출처 확인이 지연되고 있습니다.
> 마지막 확인 이후 채용 상태가 바뀌었을 수 있으니 지원 전 공식 원문을
> 확인해주세요.

저장과 지원 단계 기록은 유지하지만 새 공고 알림 대상으로 취급하지 않는다.

## 오류 처리와 운영 관측

- 지연 sweep 자체가 실패하면 전체 수집을 실패시켜 시장 스냅샷이 잘못된
  상태로 저장되지 않게 한다.
- 전체 결과 JSON과 step summary에 `delayed` 건수를 출력한다.
- 넥슨 재검증이 다시 실패해도 기존 공고를 한 번의 실패로 닫지 않는다.
- 72시간 기준은 상수 하나로 관리하고 시간 주입이 가능한 순수 함수 중심으로
  테스트한다.

## 검증 범위

전 UI를 TDD로 만들지 않는다. 회귀 위험이 높은 다음 동작만 자동화한다.

- `ALLOWED/REVIEW`로 남은 넥슨 자동 차단 상태를 seed가 복구한다.
- 비실행·오래된 출처의 `OPEN` 공고만 `DELAYED`가 된다.
- 최신 성공 출처와 실행 가능한 출처는 유지된다.
- `DELAYED` 공고가 성공 목록에서 다시 확인되면 `OPEN`으로 복구된다.
- `DELAYED` 공고가 성공 목록에서 누락되면 세 번 뒤 `CLOSED`가 된다.
- 전체 수집 결과가 지연 건수를 포함한다.
- 상세 화면이 지연 상태를 마감으로 표시하지 않는다.

그 외 레이아웃은 TypeScript, 기존 컴포넌트 테스트와 실제 화면 확인으로
검증한다.
