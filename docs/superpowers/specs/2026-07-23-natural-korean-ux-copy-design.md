# 자연스러운 한국어 UX 카피 전면 교정 설계

## 배경

현재 이직핏에는 기능을 설명하려다 문장이 길어진 곳, 보안·저장 구조 같은 내부 사정을
사용자에게 그대로 말하는 곳, 같은 개념을 여러 이름으로 부르는 곳이 함께 존재한다.
대표적으로 로그인 화면의 다음 문구는 정보보다 구현 의도를 먼저 드러낸다.

- `가입 여부와 관계없이 ... 주소로 안내를 요청했습니다.`
- `이메일 확인으로 계정을 보호하고 내 커리어 정보를 여러 기기에서 이어보세요.`

코드에서 예시·데이터 콘텐츠를 제외하고 사용자에게 노출될 수 있는 한국어 문자열 조각
2,188개를 112개 파일에서 확인했다. `확인`은 296회, `현재`는 112회, `공식`은
109회 등장하며, `해주세요`와 `해 주세요`, `미분류`와 `구분 안 됨`, `기술 맵`과
`스킬맵`, `내 스택`과 `보유 기술`이 혼용된다.

## 목표

모든 제품 화면을 처음 읽는 사용자가 문장을 해석하지 않고도 상태와 다음 행동을 바로
알 수 있게 한다.

1. 번역투와 추상적인 명사 나열을 자연스러운 한국어로 바꾼다.
2. 제목·라벨·버튼으로 이미 알 수 있는 설명은 삭제한다.
3. 기능 이름과 데이터 용어를 전 화면에서 통일한다.
4. 오류, 빈 상태, 인증 안내는 발생한 일과 다음 행동을 분명히 말한다.
5. 데이터 범위와 개인정보에 필요한 사실은 줄이지 않되 읽기 쉬운 문장으로 나눈다.

## 제외 범위

- 사용자가 작성한 글과 댓글
- 기업이 작성한 채용공고 제목과 원문
- 기업명, 기술명, 직무명 같은 원천 데이터
- 데이터 수집·인증·저장 동작 자체의 변경
- 별도 번역 프레임워크나 대규모 카피 사전 도입
- 카피 교정과 관계없는 화면 재설계

## 선택한 접근

맞춤법만 고치는 방식이나 친근한 대화체로 바꾸는 방식 대신, 전체 UX 카피를 사용자의
과업에 맞게 다시 쓴다. 정보 구조와 기능은 유지하되 필요 없는 문장을 삭제하고, 남은
문장은 짧고 직접적인 존댓말로 통일한다.

## 말투 원칙

### 문장 종결

- 사실과 상태: `~합니다`, `~했습니다`, `~없습니다`
- 사용자가 해야 하는 일: `~해 주세요`
- 버튼과 메뉴: 존댓말 없이 구체적인 동사 또는 목적지 (`저장`, `다시 시도`, `공고 보기`)
- `있어요`, `보여드려요`, `해보세요`처럼 화면마다 달라지는 친근한 말투는 쓰지 않는다.
- 보조 용언은 `확인해 주세요`, `입력해 주세요`처럼 띄어 쓴다.

### 문장 구성

- 한 문장은 한 가지 사실만 말한다.
- 제목이나 버튼이 같은 내용을 이미 말하면 설명을 삭제한다.
- `현재`, `실제`, `공식`, `확인`은 의미가 달라질 때만 쓴다.
- `~할 수 있습니다` 대신 결과나 행동을 직접 말한다.
- `다음 탐색 근거`, `공고 기반 다음 신호` 같은 추상 표현을 쓰지 않는다.
- 사용자에게 필요 없는 구현 용어를 쓰지 않는다. 예: `복구 세션`, `그래프 응답`,
  `운영 DB`, `브라우저 원본`, `계정 데이터`.
- 진행 상태의 말줄임표는 `...`가 아니라 `…`를 쓴다.

## 용어 체계

| 개념 | 기본 표현 | 적용 규칙 |
| --- | --- | --- |
| 사용자가 등록한 기술 | `내 기술` | `내 스택`, `보유 기술`, `저장한 기술`을 대체한다. |
| 기술 관계 화면 | `스킬맵` | `기술 맵`, `스킬 맵`을 사용하지 않는다. |
| 채용공고 | `공고` | 화면 첫 범위 안내에서만 `기업 공식 채용공고`를 쓸 수 있다. |
| 필수·우대 구분이 없는 기술 | `필수·우대 미표기` | 좁은 표나 그래프에서는 `미표기`로 줄이고 범례에서 뜻을 설명한다. |
| 함께 자주 등장하는 기술 | `함께 요구되는 기술` | `가까운 기술`, `공고 동시 등장`은 필요한 통계 문맥에서만 쓴다. |
| 추가 학습 후보 | `다음에 배울 기술` | `다음 준비`, `다음 신호`, `다음 경로`를 대체한다. |
| 저장한 항목 모음 | `저장 목록` | `저장 보관함`, `저장 라이브러리`를 대체한다. |
| 데이터가 마지막으로 검증된 시각 | `최근 확인` | `마지막 확인`, `가장 최근 확인`을 대체한다. |

`필수·우대 미표기`의 설명은 다음 의미를 유지한다.

> 공고에 기술은 나오지만 필수 또는 우대로 구분되어 있지 않은 경우입니다.

## 화면별 적용

### 전역 탐색과 공통 상태

- 검색 입력의 예시 문구는 `회사, 직무, 기술, 주제 검색`으로 줄인다.
- 메뉴는 좁은 공간에서 뜻이 분명한 가장 짧은 이름을 쓰고, 페이지 제목은 같은 개념의
  자연스러운 전체 이름을 쓴다. 예: 메뉴 `공고`, 페이지 제목 `채용공고`.
- 결과가 화면에 바로 반영되는 저장·공감·팔로우 동작은 성공 메시지를 띄우지 않는다.
  접근성에 필요한 상태는 버튼의 `aria-pressed`, 라벨 또는 화면 낭독용 상태로 전달한다.
- 전역 오류의 `안전한 경로` 같은 모호한 표현은 실제 목적지인 `홈` 또는 `공고`로 바꾼다.

### 로그인과 회원가입

- 페이지 제목은 `로그인`으로 바꾸고, 상단 소개 문장과 `로그인하면 달라지는 점` 영역은
  삭제한다.
- 로그인 폼 안의 중복 제목과 설명은 삭제한다.
- 회원가입 화면에는 이메일 확인이 필요하다는 사실만 한 번 표시한다.
- 계정 존재 여부를 숨기는 보안 동작은 유지하되 내부 정책을 설명하지 않는다.
- 비밀번호 재설정 완료 문구는 다음으로 통일한다.

  ```text
  비밀번호 재설정 요청을 완료했습니다.
  {email}의 메일함을 확인해 주세요.
  ```

- 만료된 링크는 `이 링크는 만료되었거나 사용할 수 없습니다.`라고 설명하고
  `재설정 메일 다시 받기`를 제공한다.

### 홈과 커뮤니티

- `커리어 이야기 둘러보기`는 `커리어 이야기`로 줄인다.
- `채용 시장 인사이트`는 `채용 시장`, `내 커리어 인사이트`는
  `내 기술과 맞는 공고`처럼 내용이 드러나는 제목으로 바꾼다.
- 작성·댓글 입력 안내는 개인정보와 회사 기밀을 적지 말라는 필수 주의만 남긴다.
- 수정·삭제·신고의 실패 문구는 작성 내용을 보존했는지 함께 알린다.
- 화면에서 결과가 보이는 댓글 등록, 공감, 저장, 팔로우 성공 문구는 제거한다.
- `정말 삭제`는 `글 삭제`로 바꾸고, 삭제 범위를 확인 문장으로 설명한다.

### 채용 시장

- 페이지 제목은 `채용 시장 기술 동향`으로 통일한다.
- 도입 문장은 `기업 채용공고에 많이 나온 기술과 최근 변화를 보여줍니다.`로 줄인다.
- 데이터 범위 안내는 페이지 상단에 한 번만 둔다. 이후 카드마다 `현재`, `공식`,
  `확인 가능한 범위`를 반복하지 않는다.
- `미분류`와 `구분 안 됨`은 `필수·우대 미표기`로 통일한다.
- 차트 제목, 범례, 표 머리글은 같은 단어를 사용한다.
- 추세 데이터가 부족하면 필요한 주차와 현재 쌓인 주차를 한 문장으로 안내한다.

### 스킬맵

- 제목은 `스킬맵`으로 통일한다.
- 도입 문장은 `내 기술과 함께 자주 요구되는 기술을 보여줍니다.`로 줄인다.
- `내 스택과 필터`, `그래프 필터`처럼 중복되는 제목은 하나만 남긴다.
- 추천 영역은 `다음에 배울 기술`, 관계 영역은 `함께 요구되는 기술`로 이름 붙인다.
- 사용법은 데스크톱과 모바일 입력 방식에 맞춰 한 줄로 표시한다.
- 오류 문구에서 `임의 데이터로 채우지 않았습니다`, `그래프 응답` 같은 내부 설명을
  제거한다. 실제 데이터만 표시하는 원칙은 데이터 정책에서 설명한다.

### 공고

- 제목은 `채용공고`로 통일한다.
- 도입 문장은 `기술·직무·기업으로 공고를 찾고 내 기술과 비교합니다.`로 줄인다.
- `이번 페이지 기업`은 `현재 목록`처럼 사용자가 보고 있는 범위를 직접 가리킨다.
- `분석 보기`는 `기술 요건 보기`로 바꾼다.
- 지원 전 원문 확인 안내는 검색 패널마다 반복하지 않고 결과 영역 하단에 한 번 둔다.
- 저장 검색을 만들 수 없는 이유는 `검색어, 기술 분야, 경력 조건 중 하나를 선택해
  주세요.`라고 버튼 가까이에 표시한다.

### 내 커리어와 계정 기능

- 메뉴와 페이지 제목은 `내 커리어`를 유지하고, 첫 번째 주요 영역은 `내 기술`로
  통일한다. 도입 문장에서는 내 기술과 공고를 비교하는 화면임을 바로 설명한다.
- `브라우저 저장 · 로그인 시 동기화` 같은 구현 중심 배지는 로그인 상태에 따라
  `이 기기에 저장됨` 또는 `계정에 저장됨`으로 바꾼다.
- 비어 있을 때는 `먼저 내 기술을 추가해 주세요.`와 기술 추가 행동만 남긴다.
- `숫자를 읽는 방법` 영역은 반복되는 설명을 삭제하고 분석의 한계만 짧게 남긴다.
- 저장 목록, 알림, 일정, 관심 기업, 내 글 화면도 같은 용어와 문장 종결을 사용한다.

### 검색, 빈 상태, 오류

- 빈 상태는 `무엇이 비었는지`와 `다음 행동`만 최대 두 문장으로 말한다.
- 오류는 `무엇을 하지 못했는지`와 `어떻게 다시 시도하는지`를 말한다.
- 알 수 없는 원인을 네트워크 문제로 단정하지 않는다.
- 입력 내용이 유지되거나 일부 결과를 계속 볼 수 있으면 그 사실을 함께 알린다.
- `설정을 불러오지 못했습니다`처럼 사용자가 알 수 없는 대상을 말하지 않는다.

### 개인정보, 데이터 정책, 분석 방법

- 저장 범위, 공개 범위, 삭제 결과, 분석 한계는 사실을 그대로 유지한다.
- 한 문단에 여러 저장소와 예외가 섞이면 목록이나 짧은 문단으로 나눈다.
- `URL query`는 `주소에 포함된 검색 조건`, `운영 DB`는 `서비스에 반영된 데이터`처럼
  바꾼다.
- Supabase처럼 개인정보 처리 방식을 이해하는 데 필요한 서비스명은 유지한다.
- 정책 문구를 제품 화면마다 반복하지 않고 관련 정책 페이지로 연결한다.

## 코드 구성

모든 문구를 하나의 거대한 사전으로 옮기지 않는다. 화면에만 쓰이는 문구는 해당 컴포넌트
가까이에 두고, 여러 화면이 공유하는 용어와 상태 라벨만 기존 `src/lib/labels.ts` 또는
작은 전용 모듈에서 관리한다. 이렇게 하면 문맥을 잃지 않으면서 용어의 재발산을 막을 수
있다.

구현은 다음 순서로 나눈다.

1. 공통 용어와 금지 표현 기준 확정
2. 로그인, 전역 탐색, 공통 오류 교정
3. 홈, 시장, 스킬맵, 공고 교정
4. 내 커리어, 저장 목록, 알림, 일정, 관심 기업, 내 글 교정
5. 검색, 커뮤니티 상세와 작성 흐름 교정
6. 개인정보, 데이터 정책, 분석 방법 교정
7. 테스트 기대값과 접근성 라벨 교정

## 검증

### 자동 검증

- 기존 단위 테스트를 새 문구에 맞춰 갱신한다.
- 사용자 노출 소스를 검사하는 카피 계약 테스트를 추가한다. 사용자 작성 예시,
  외부 공고 원문, 기업명·기술명 데이터와 테스트 설명은 검사 대상에서 제외한다.
- 아래 표현은 허용 목록 없이 제품 UI에 남지 않아야 한다.

  - `가입 여부와 관계없이`
  - `이메일 확인으로 계정을 보호하고`
  - `해주세요`
  - `있어요`, `보여드려요`, `해보세요`, `이어보세요`
  - `복구 세션`, `그래프 응답`, `운영 DB`, `브라우저 원본`
  - `미분류`, `구분 안 됨`, `기술 맵`, `내 스택`
  - 사용자에게 보이는 `...`

- 인증, 검색, 시장, 스킬맵, 공고, 내 커리어, 커뮤니티 핵심 흐름의 E2E 테스트를 실행한다.
- lint와 프로덕션 빌드를 통과해야 한다.

### 화면 검증

- 390px 모바일과 1440px 데스크톱에서 핵심 화면을 비교한다.
- 삭제한 문구 때문에 빈 공간이나 불필요한 구분선이 남지 않아야 한다.
- 버튼과 탭 라벨은 한 줄로 유지한다.
- 긴 이메일, 오류, 빈 상태가 화면을 밀어내거나 잘리지 않아야 한다.
- 스크린 리더용 라벨도 화면에 보이는 용어와 일치해야 한다.

## 완료 기준

1. 사용자가 지적한 두 인증 문구가 삭제되거나 승인된 문구로 바뀐다.
2. 제품 UI에서 말투와 띄어쓰기 혼용이 사라진다.
3. `내 기술`, `스킬맵`, `필수·우대 미표기` 용어가 전 화면에서 일관된다.
4. 각 오류와 빈 상태가 다음 행동을 제시한다.
5. 필수 정책 사실은 유지되고 내부 구현 설명은 제품 화면에서 제거된다.
6. 사용자 작성 글과 기업 공고 원문은 바뀌지 않는다.
7. 단위 테스트, E2E, lint, 빌드와 핵심 화면 검증을 모두 통과한다.
