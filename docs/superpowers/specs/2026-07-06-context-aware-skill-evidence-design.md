# 문맥 기반 스킬 분류와 근거 추출 설계

작성일: 2026-07-06

## 1. 목표

현재의 사전 기반 스킬 추출을 확장하여 공고에 등장한 스킬을 `필수`, `우대`,
`미분류`로 구분하고 원문 근거를 함께 저장한다. `C`, `C++`, `Go`, `R`처럼 짧거나
일반 표현과 충돌하는 기술명도 빼지 않되, 대소문자·주변 표현·HTML 구역을 함께
검사하여 오탐을 억제한다.

이 기능은 다음 원칙을 따른다.

- LLM이나 외부 API를 사용하지 않는 결정적 규칙으로 구현한다.
- 같은 입력은 항상 같은 결과를 생성한다.
- 기존 `skills: string[]` API 계약은 유지한다.
- 확정하기 어려운 후보는 낮은 신뢰도로 보존하되 통계와 기본 화면에서 제외한다.
- 상세 화면에 노출하는 각 스킬에는 원문 근거와 판정 이유를 추적할 수 있어야 한다.

## 2. 현재 구현의 한계

`packages/backend/src/ejikfit/skills.py`는 제목과 평탄화된 본문을 소문자로 바꾼 뒤
모든 영문 별칭에 동일한 토큰 경계 규칙을 적용한다.

이 방식에는 다음 문제가 있다.

- `Go`와 일반 영단어 `go`, `R`과 일반 문자 `r`의 대소문자 차이를 잃는다.
- 단독 `C`와 `R`은 오탐 우려 때문에 사전에서 제외되어 있다.
- `C++17`처럼 기술명 바로 뒤에 버전이 붙은 표현을 놓칠 수 있다.
- `description_text` 생성 시 HTML 제목과 목록 경계가 공백으로 합쳐져
  `자격 요건`과 `우대 사항` 구역을 구분할 수 없다.
- 추출 결과에 근거, 필수·우대 구분, 신뢰도, 판정 이유가 없다.

## 3. 운영 데이터 재검증

2026-07-06에 공개 Vercel API를 읽기 전용으로 호출하여 10개 회사의 운영 중인
공고 130건 전체를 분석했다. Supabase 직접 연결은 현재 실행 환경의 PostgreSQL
포트 제한으로 실패했으나, 운영 API 상태는 정상이고 같은 공고 본문을 API로
검증했다.

### 3.1 짧은 기술명 후보

| 후보 | 등장 횟수 | 실제 기술 | 관찰된 오탐 |
| --- | ---: | ---: | --- |
| `C++` | 6 | 6 | 없음 |
| 단독 `C` | 5 | 2 | `C-level` 3회 |
| `Go` | 17 | 16 | `Go-to-Market` 1회 |
| `Golang` | 5 | 5 | 없음 |
| 단독 `R` | 9 | 3 | `R&D`, `Cortex-M/R` 6회 |
| `RStudio` | 0 | 0 | 없음 |

유효한 표현에는 `C/C++ 프로그래밍`, `Go/TypeScript/Solidity`,
`Python, R, SQL`, `Go 언어`, `Golang`이 있었다. 이 결과는 단순 단어 경계가
충분하지 않고 합성어 제외와 기술 문맥 확인이 필요함을 보여준다.

### 3.2 HTML 구역 구조

130개 공고에는 제목 태그가 1,193개 있었지만, 제목 태그가 아닌 짧은 문단으로
표시된 구역명도 있었다.

- `[자격 요건]`: 12회
- `[우대 사항]`: 12회
- `[일반 우대사항]` 및 직무별 우대사항
- `### 이런 분들을 찾고 있어요`
- `### 이런 분이면 더 좋아요`

따라서 `h1`~`h6`만 읽는 파서는 충분하지 않다. 짧은 `p`, `div`, `strong`,
`b` 요소도 알려진 구역명과 일치하면 의사 제목으로 처리해야 한다.

### 3.3 검증 결론

보정된 문맥 규칙은 이번 표본에서 관찰된 `C`, `C++`, `Go`, `Golang`, `R`
후보를 기술과 비기술 표현으로 모두 구별할 수 있다. 다만 아직 등장하지 않은
미래 표현의 재현율까지 보장하지는 않는다. 이를 위해 낮은 신뢰도 후보를
보존하고, 새 오탐·누락이 발견될 때 골든 테스트와 정책을 함께 갱신한다.

## 4. 검토한 접근법

### 4.1 정확한 문자열만 허용

명시적인 별칭만 일치시키면 구현이 단순하고 오탐이 적다. 그러나 `C`, `Go`,
`R`처럼 문맥 없이는 판정하기 어려운 기술을 계속 누락하므로 요구사항을
충족하지 못한다.

### 4.2 문맥 기반 결정 규칙

원문의 대소문자와 HTML 구조를 보존하고 스킬별 후보 규칙, 제외 규칙, 기술 문맥,
필수·우대 구역을 함께 평가한다. 재현 가능하고 운영 비용이 없으며 판정 근거를
설명할 수 있다. 이 설계의 채택안이다.

### 4.3 LLM 또는 학습 모델

표현 변화에는 유연하지만 비용, 응답 일관성, 재현성, 근거 추적 문제가 있다.
현재 130건 규모와 사전 기반 파이프라인에는 과도하므로 이번 범위에서 제외한다.

## 5. 추출 파이프라인

추출은 다음 순서로 실행한다.

```text
description_html 구역·블록 복원
  → 스킬 후보 탐지
  → 명시적 제외 규칙
  → 스킬별 문맥 검증
  → 필수·우대·미분류 판정
  → 동일 스킬 후보 병합
  → DB 동기화
```

제목도 별도의 `미분류` 블록으로 입력한다. HTML이 없거나 파싱 결과가 비어 있으면
`description_text`를 문장 단위로 나눈 블록에 폴백한다.

### 5.1 구역과 근거 블록 복원

HTML을 문서 순서로 순회하며 다음 요소를 구역 제목 후보로 본다.

- `h1`~`h6`
- 자식 블록이 없고 정규화한 길이가 80자 이하인 `p`, `div`, `strong`, `b`
- 대괄호, 마크다운 `#`, 이모지, 콜론을 제거한 뒤 알려진 구역명과 일치하는 요소

근거 블록은 `li`와 잎 `p`를 우선 사용한다. 목록이 아닌 평문은 문장 종결 부호와
줄바꿈으로 나눈다. 각 블록은 자신보다 앞선 가장 가까운 구역 제목을 상속한다.

구역 분류는 명시적인 표현 집합으로 관리한다.

- 필수: `자격 요건`, `필수 자격 요건`, `지원 자격`, `이런 분을 찾고 있어요`,
  `이런 분과 함께하고 싶어요`, `함께하기 위해 필요한 기본 요건`,
  `이런 기술이 필요해요`
- 우대: `우대 사항`, `일반 우대사항`, `이런 분이면 더 좋아요`,
  `이런 분이라면 더 좋아요`, `preferred`, `nice to have`
- 미분류: `사용중인 기술`, `이런 기술들을 사용하고 있어요`, 주요 업무,
  담당 업무, 팀 소개, 복리후생 및 그 밖의 구역

단순히 `필요`가 포함됐다는 이유만으로 필수 구역으로 보지 않는다. 새로운 구역명은
골든 테스트와 함께 명시적으로 추가한다.

### 5.2 필수·우대 판정

블록 안의 국소 표현이 구역보다 우선한다. 예를 들어 자격 요건 구역 안에서도
`AWS 경험자 우대`라고 쓰인 블록은 우대로 분류한다. 일반적인 `요구사항 분석`
같은 표현을 필수 조건으로 오해하지 않도록 단순 부분 문자열이 아니라 명시적인
조건 표현만 사용한다.

우선순위는 다음과 같다.

1. 블록 안의 명시적인 `필수`, `반드시`, `우대`, `선호`, `있으면 좋아요` 표현
2. 상속한 구역의 필수·우대 분류
3. 어느 쪽도 아니면 `unspecified`

같은 스킬이 여러 블록에 등장하면 `required`, `preferred`, `unspecified`
순으로 대표 관계를 선택한다. 같은 유형에서는 신뢰도가 높은 후보, 원문에서 먼저
등장한 후보 순으로 대표 근거를 선택한다.

## 6. 스킬 후보와 문맥 정책

현재 사전은 정규 기술명 52개와 별칭 68개를 포함하지만 전체 IT 기술 목록은
아니다. 이번 기능은 임의로 수백 개 이름을 추가하는 대신, 새 기술을 안전하게
추가할 수 있는 별칭별 정책 구조를 먼저 만든다. 단독 `C`와 `R`은 이 구조를
사용하는 첫 신규 기술로 추가한다.

같은 기술도 별칭마다 위험도가 다르므로 정규 기술명이 아닌 별칭 단위로 다음
정책을 지정한다.

| 정책 | 의미 | 예 |
| --- | --- | --- |
| `distinct` | 단어 경계만으로 충분히 구별됨 | `python`, `kubernetes`, `golang` |
| `contextual` | 일반 단어와 충돌하여 기술 구역이나 개발 문맥이 필요함 | `Spring`, `React`, `Swift`, `Rust`, `Ruby`, `Flask`, `Unity`, `Flutter`, `Gazebo` |
| `strict` | 짧은 토큰 또는 충돌이 많아 전용 양성·음성 규칙이 필요함 | `C`, `R`, `Go` |

별칭 정의는 다음 정보를 가질 수 있다.

- 원문 대소문자 보존 여부
- 토큰 또는 버전 패턴
- 위험도 정책
- 양성 문맥 단어
- 명시적 제외 패턴
- 판정 시 기록할 `match_reason`

예를 들어 `Golang`은 `distinct`, 같은 정규 기술의 `Go`는 `strict`다.
`Spring Boot`는 `distinct`, 단독 `Spring`은 `contextual`이다. 모든 별칭을
일괄 소문자 매칭하는 현재 방식은 제거한다.

문맥 충돌 가능성이 있는 현재 사전 항목에는 다음도 포함된다.

- 일반 영어 단어: `Spring`, `React`, `Swift`, `Rust`, `Ruby`, `Flask`,
  `Unity`, `Flutter`
- 일반 명사·고유명사: `Kafka`, `Gazebo`, `Android`
- 짧은 약어: `SQL`, `ROS`, `SLAM`, `iOS`
- 겹치는 기술군: `Java`/`JavaScript`, `C`/`C++`/`C#`,
  `SQL`/`PostgreSQL`, `React`/향후 `React Native`

`contextual`과 `strict` 별칭은 각각 최소 한 개의 양성 및 음성 골든 테스트가
없으면 사전에 등록할 수 없게 불변식 테스트로 강제한다.

### 6.1 일반 스킬

Python, Kubernetes처럼 충분히 구별되는 기존 별칭은 대소문자 비구분 토큰
경계로 탐지한다. 원문은 소문자로 변환하지 않고 비교용 값만 별도로 만든다.
Java와 JavaScript처럼 접두사가 겹치는 기술은 기존 경계 보호를 유지한다.

### 6.2 `C++`

다음을 확정 후보로 인정한다.

- `C++`
- `C++11`, `C++14`, `C++17`, `C++20`, `C++23`
- `C / C++`, `C/C++`

`C#`과는 별칭 및 경계를 분리한다.

### 6.3 단독 `C`

대문자 `C` 토큰에 다음 중 하나가 있을 때만 확정한다.

- `C 언어`, `C 프로그래밍`, `C 개발`
- `C/C++`, `C / C++`
- 필수·우대·기술 구역의 목록 항목에서 다른 프로그래밍 언어와 병렬로 등장
- 같은 근거 블록에 펌웨어, 임베디드, 컴파일러, RTOS와 같은 개발 문맥이 존재

`C-level`, `C level`, `Vitamin C`, 드라이브 문자, 등급 또는 단순 열거
`A/B/C`는 명시적으로 제외한다.

### 6.4 `Go`

`Golang`, `Go 언어`, `Go 1.x`, `go.mod`는 확정한다. 대문자 `Go`는 다음 중
하나가 있을 때 확정한다.

- 프로그래밍 언어 목록에서 다른 확정 스킬과 병렬로 등장
- 같은 블록에 개발, 백엔드, 서버, 프로그래밍, 언어, 프레임워크 문맥이 존재
- 필수·우대·기술 구역에 속한다.

`Go-to-Market`, `go to`, 일반 동사 `go`는 제외한다. 소문자 단독 `go`는
인정하지 않으며 `go.mod`, `go test`, `go build`처럼 Go 도구 문맥만 예외로
인정한다.

### 6.5 `R`

다음은 확정한다.

- `R 언어`, `RStudio`, `R Shiny`
- `Python/R`, `Python, R, SQL`처럼 다른 데이터 기술과 병렬로 등장
- 필수·우대·기술 구역에서 통계, 데이터 분석, 모델링, 시각화 문맥과 함께 등장

`R&D`, `Cortex-M/R`, 경로·등급·변수명으로만 등장한 `R`은 제외한다.
소문자 단독 `r`은 인정하지 않는다.

## 7. 신뢰도와 후보 보존

`confidence`는 확률이 아니라 어떤 규칙으로 확정했는지를 정렬하기 위한 결정적
점수다.

| 점수 | 조건 | 기본 노출 |
| ---: | --- | --- |
| `1.00` | 구별되는 별칭 또는 명시적 기술 표현 | 노출 |
| `0.95` | 짧은 기술명과 강한 개발·언어 문맥 | 노출 |
| `0.85` | 짧은 기술명이 기술 구역 또는 다른 스킬 목록에 등장 | 노출 |
| `0.50` | 토큰은 발견했지만 확정 문맥이 부족함 | 후보로만 보존 |

`0.80` 이상을 확정 결과로 본다. 명시적 제외 표현은 기술 후보가 아니므로 저장하지
않는다. 낮은 신뢰도 후보는 `posting_skills`에 남지만 기존 `skills`, 통계,
기본 상세 화면에서는 제외한다.

## 8. 데이터 모델

`posting_skills`에 다음 열을 추가한다.

| 열 | 타입 | 의미 |
| --- | --- | --- |
| `requirement_type` | `String(20)` | `required`, `preferred`, `unspecified` |
| `evidence_text` | `Text`, nullable | 대표 원문 근거 블록 |
| `confidence` | `Float` | 결정적 규칙 점수 |
| `match_reason` | `String(100)` | 대표 판정 규칙 식별자 |

기존 `(posting_id, skill)` 유일 제약은 유지한다. 마이그레이션은 기존 행을
`unspecified`, `0.50`, `legacy_backfill`로 안전하게 채우고 다음 재추출 때 실제
값으로 교체한다.

## 9. API 계약

기존 상세 응답의 `skills: string[]`는 확정 스킬 이름 목록으로 유지한다.
추가로 `skill_details`를 제공한다.

```json
{
  "skill": "Go",
  "category": "language",
  "requirement_type": "required",
  "evidence_text": "Go / C++ / Java 중 하나 이상을 사용한 백엔드 개발 경험",
  "confidence": 0.95,
  "match_reason": "ambiguous_alias_with_development_context"
}
```

`GET /api/skills/stats`의 기존 `skill`, `category`, `count`는 유지하고 다음 값을
추가한다.

- `required_count`
- `preferred_count`
- `unspecified_count`

통계와 기존 `skills` 배열은 `confidence >= 0.80`인 관계만 사용한다.

## 10. 웹 화면

공고 상세의 기존 단일 스킬 태그 목록을 다음 세 구역으로 나눈다.

- 필수 기술
- 우대 기술
- 공고에 언급된 기술

각 스킬에는 대표 근거 문장을 함께 표시한다. 근거가 없는 마이그레이션 직후의
레거시 행은 노출 기준을 충족하지 않으므로 표시하지 않는다. 낮은 신뢰도 후보는
내부 데이터로만 보존하며 사용자에게 확정 기술처럼 보여주지 않는다.

홈의 스킬 수요 순위는 기존 총 공고 수를 유지하고, 각 행에 필수·우대 건수를
보조 정보로 표시한다.

## 11. 동기화와 운영 반영

`sync_posting_skills`는 제목, `description_html`, `description_text`를 받아
스킬 관계 전체를 멱등 동기화한다.

- 사라진 스킬은 삭제한다.
- 새 스킬은 추가한다.
- 같은 스킬의 유형, 근거, 신뢰도, 판정 이유가 바뀌면 기존 행을 갱신한다.
- 같은 입력을 다시 처리하면 DB 내용이 바뀌지 않는다.

기존 `extract_skills(text) -> list[str]`는 하위 호환을 위해 확정된 이름만
반환한다. 새 구조화 함수는 제목과 HTML·평문을 입력받아 낮은 신뢰도 후보까지
포함한 스킬 매치 객체를 반환하고, DB 동기화가 이 구조화 함수를 사용한다.

기존 `ejikfit backfill-skills`는 같은 추출기를 사용한다. 배포 후 예약
`crawl-all`이 모든 공개 출처를 다시 처리하므로 운영 공고가 순차적으로
갱신된다. 즉시 반영이 필요하면 GitHub Actions 수집을 수동 실행한다.

## 12. 오류 처리

- 잘못된 HTML은 BeautifulSoup의 복구 가능한 파싱을 사용한다.
- HTML 블록을 얻지 못하면 평탄화 본문의 문장 분리로 폴백한다.
- 구역을 알 수 없더라도 스킬 자체가 명확하면 `unspecified`로 저장한다.
- 짧은 기술명의 문맥이 부족하면 확정하지 않고 `0.50` 후보로 저장한다.
- 추출 실패가 공고 수집 전체를 중단시키지 않도록 입력별 순수 함수로 분리하고
  골든 테스트에서 예외 없는 처리를 검증한다.

## 13. 테스트와 검증

모든 구현은 테스트 우선으로 진행한다.

### 13.1 골든 추출 테스트

다음 양성 표현을 포함한다.

- `C/C++ 프로그래밍 능력`
- `C++17 기반 엔진 개발`
- `Go/TypeScript/Solidity 중 1개 이상`
- `Python, Go, Node.js`
- `Golang 기반 백엔드`
- `Python, R, SQL을 사용한 통계 분석`
- `RStudio와 R Shiny 경험`

다음 음성 표현을 포함한다.

- `C-level 대상 제안`
- `Vitamin C`
- `A/B/C 등급`
- `Go-to-Market 전략`
- `go to the next step`
- `국가 R&D 과제`
- `ARM Cortex-M/R 시리즈`

추가로 현재 사전의 `contextual` 별칭에 대해 일반 의미와 기술 의미를 짝으로
검증한다. 예를 들면 계절을 뜻하는 `spring`과 `Spring Boot`, 일반 반응을
뜻하는 `react`와 `React 프론트엔드`, 부식을 뜻하는 `rust`와 `Rust 언어`를
각각 구분해야 한다.

### 13.2 사전 정책 불변식 테스트

- 모든 별칭에 `distinct`, `contextual`, `strict` 중 하나가 지정됨
- `contextual` 및 `strict` 별칭마다 양성·음성 골든 사례가 존재함
- 대소문자가 판정에 필요한 별칭이 비교 과정에서 소문자화되지 않음
- 겹치는 별칭이 더 긴 기술명 내부에서 중복 검출되지 않음

### 13.3 구역 분류 테스트

- 제목 태그 기반 `자격 요건`과 `우대 사항`
- 일반 문단 기반 `[자격 요건]`과 `[우대 사항]`
- 이모지가 붙은 `이런 분을 찾고 있어요`, `이런 분이면 더 좋아요`
- `사용중인 기술`은 필수가 아닌 `unspecified`
- 필수 구역 안의 `AWS 경험자 우대`는 국소 표현에 따라 `preferred`

### 13.4 저장·API·웹 테스트

- 스킬 관계의 추가, 갱신, 삭제와 재실행 멱등성
- 기존 스킬 문자열 배열의 하위 호환성
- 낮은 신뢰도 후보의 통계 제외
- 필수·우대·미분류 집계
- 상세 화면의 세 구역과 근거 문장
- 근거가 없는 빈 구역을 렌더링하지 않는 동작

### 13.5 운영 표본 감사

구현 후 공개 운영 API의 130개 공고 본문에 새 추출기를 읽기 전용으로 적용한다.
`C`, `C++`, `Go`, `Golang`, `R` 결과를 기존 수동 분류와 비교하여 관찰된
후보에서 오탐과 누락이 없는지 확인한다. 이 검증은 운영 DB를 변경하지 않는다.

## 14. 완료 기준

- 기존 스킬 추출 테스트와 새 골든 테스트가 모두 통과한다.
- 백엔드 전체 테스트, 웹 테스트, TypeScript 검사와 빌드가 통과한다.
- Alembic offline SQL 생성이 통과한다.
- 기존 API 필드가 제거되거나 타입 변경되지 않는다.
- 운영 130개 공고 읽기 전용 감사에서 관찰된 짧은 기술명 후보를 수동 기준과
  동일하게 분류한다.
- 동일 공고 재처리 시 `posting_skills` 내용이 변하지 않는다.

## 16. 사전 확장 원칙

새 기술은 운영 공고에서 실제 표현이 확인된 순서로 추가한다.

1. 원문 근거가 있는 미등록 기술 후보를 수집한다.
2. 정규 이름, 별칭, 범주를 정한다.
3. 별칭별 위험도와 양성·음성 문맥을 정의한다.
4. 실제 문장을 축약한 골든 테스트를 먼저 추가한다.
5. 운영 표본 읽기 전용 감사 후 사전에 반영한다.

이 절차는 사전 크기 자체보다 오탐과 근거 품질을 우선한다. 자동 발견 후보는
검토 대상으로만 다루며, 근거 없이 사용자 화면이나 통계에 확정 기술로 넣지 않는다.
