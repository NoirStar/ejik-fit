# 브라우저 전용 최근 커뮤니티 주제 설계

## 문제

홈 왼쪽 레일의 `최근 본 주제`는 현재 사용 이력과 무관한 고정 mock 태그를
표시한다. 커뮤니티 예시 자체는 허용되지만, 보지 않은 항목을 최근 기록처럼
표현하면 제품의 데이터 경계를 흐린다.

## 목표

- 사용자가 실제로 연 커뮤니티 상세의 대표 주제만 현재 브라우저에 기록한다.
- 최근에 본 항목부터 최대 4개를 홈 왼쪽 레일에 표시한다.
- 기록이 없으면 임의 추천을 채우지 않고 정직한 빈 상태를 표시한다.
- 같은 글을 다시 열면 중복하지 않고 가장 최근 위치로 이동한다.
- 같은 탭과 다른 탭의 기록 변경을 홈에 반영한다.
- 로컬 글을 삭제하거나 개인정보 화면에서 저장 데이터를 지우면 관련 기록도
  제거한다.

## 저장 경계

브라우저 `localStorage`의 `ejik-fit:recent-community-topics` 키에 아래 최소 정보만
저장한다.

- 안전하게 검증한 post ID
- 카드 제목
- 첫 번째 태그, 태그가 없으면 카테고리
- `mock | local` 출처
- 마지막으로 본 시각

본문, 작성자, 댓글, 반응 수는 복사하지 않는다. 서버 API나 metadata로 기록을
전송하지 않으며 다른 브라우저와 동기화하지 않는다. 손상된 값, 위험한 ID,
과도하게 긴 문자열과 잘못된 날짜는 읽을 때 제거한다.

## 표시와 이동

- 홈에는 대표 주제를 `# 주제`로, 아래에 원래 글 제목을 한 줄로 표시한다.
- 링크는 저장된 URL을 신뢰하지 않고 검증된 post ID로 `/posts/:id`를 다시 만든다.
- `이 브라우저` 라벨로 기록 범위를 텍스트로 설명한다.
- 최대 8개를 저장하고 홈에서는 최근 4개만 보여 향후 다른 화면에서도 재사용할
  여지를 둔다.
- 태블릿·모바일에서 기존 정책대로 왼쪽 레일이 숨겨지는 동작은 유지한다.

## 기록 시점

mock 상세는 서버 HTML을 유지하면서 작은 클라이언트 tracker가 hydration 뒤
기록한다. 로컬 상세는 브라우저 저장소에서 실제 글을 찾은 뒤에만 기록한다.
없는 로컬 ID, 삭제 완료 상태, 404는 기록하지 않는다.

## 오류와 삭제

- 저장소 읽기나 쓰기가 차단되면 상세 열람은 계속 가능해야 한다.
- 로컬 글 삭제는 글 삭제를 우선하며, 성공 뒤 해당 최근 기록을 best-effort로
  제거한다.
- 개인정보 화면의 전체 삭제는 새 키가 실제로 제거됐는지까지 확인한다.

## 검증 기준

- 저장 모델의 검증, 정렬, 중복 제거, 상한, 이벤트 구독이 단위 테스트를 통과한다.
- mock 및 로컬 상세를 열었을 때만 기록된다.
- 홈 빈 상태와 최근 순서, 새로고침 지속성이 정확하다.
- 삭제한 로컬 글 링크가 최근 목록에 남지 않는다.
- 데스크톱 브라우저 흐름에서 44px 링크, 오버플로 없음, 콘솔 오류 없음이 유지된다.
