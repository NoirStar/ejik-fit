# 스킬맵 표현과 홈 피드 안정화 설계

## 목표

스킬맵을 사용자가 설명 없이도 읽을 수 있는 시장 관계도로 만들고, 저사양 환경에서도
그래프가 불필요한 계산을 계속하지 않게 한다. 동시에 홈 피드에서 커뮤니티 글이 늦게
삽입되며 공고 카드의 위치가 바뀌는 현상과 태그 축약 표시의 형태 불일치를 제거한다.

이번 변경은 세 가지 독립 영역을 같은 사용자 체감 목표 아래 다룬다.

1. 스킬맵의 색, 크기, 선 굵기에 명확한 데이터 의미를 부여한다.
2. 그래프의 초기 배치 이후 지속되는 CPU 사용을 중단한다.
3. 홈 피드의 첫 렌더 순서와 태그 표현을 안정시킨다.

## 확인된 현재 상태

### 스킬맵 의미

- 기술 노드 색은 첫 번째 직무 분야를 나타내지만 그래프 안에 이를 설명하는 범례가 없다.
- 기술 노드 크기는 `demand_count`의 제곱근 보정값을 사용한다.
- 중심 기술은 실제 공고 수와 무관하게 고정값 `12`를 받아 항상 가장 크게 표시된다.
- 기술 간 선 굵기는 함께 등장한 원시 건수가 아니라 정규화된 `score`를 사용한다.
- 캔버스 툴팁은 영문 `postings` 표현을 사용하며 직접 연결 기술 수를 보여주지 않는다.

옵시디언 Graph view는 많이 참조된 노드를 크게 표시하고, 색은 그룹 구분에 사용하며,
호버 시 연결 관계를 강조한다. 이직핏은 이 절제된 원칙을 따르되 노드 크기의 주 지표를
메모 참조 수가 아니라 채용시장의 언급 공고 수로 둔다.

참고: <https://obsidian.md/help/plugins/graph>

### 그래프 성능

현재 Canvas 2D 렌더러는 기본 모션 상태에서 다음 설정을 사용한다.

- `cooldownTicks(Infinity)`
- `cooldownTime(Infinity)`
- `autoPauseRedraw(false)`
- 호버와 선택 시 `d3ReheatSimulation()` 호출

따라서 배치가 안정된 뒤에도 force simulation과 캔버스 redraw가 계속된다. 호버할 때는
전체 링크를 다시 순회해 이웃을 찾는다.

2026-07-24 로컬 production fixture를 Chromium에서 6배 CPU 제한으로 측정한 결과는
다음과 같다. 수치는 절대적인 사용자 기기 성능이 아니라 변경 전후 비교 기준이다.

| 조건 | 그래프 준비 | 준비 후 4초간 TaskDuration |
| --- | ---: | ---: |
| 기본 모션 | 6,090ms | 3,492ms |
| reduced motion | 5,904ms | 415ms |

기본 모션에서는 관찰 구간의 대부분을 메인 스레드 작업이 차지했다. reduced motion도
초기 엔진 준비 시간은 비슷하므로 지속 계산과 초기 로딩을 별도 문제로 다뤄야 한다.

### 홈 피드 첫 렌더

서버 페이지는 공고, 기술 통계, 스킬 그래프, 적합도만 가져온다. 커뮤니티 글은
`useCommunityFeed`가 hydration과 인증 확인 뒤 브라우저 Supabase 요청으로 가져온다.
그 결과 서버 HTML에는 공고와 시장 카드만 있고, 응답 후 커뮤니티 글이 배열 앞에
삽입된다.

자바스크립트를 끈 production fixture에서는 `Python Backend Engineer` 등 공고와 시장
카드만 렌더됐다. 자바스크립트를 켜면 `community_posts` 요청의 200 응답 후 실제 글이
첫 항목으로 추가되는 것을 재현했다.

### 태그 축약 표시

게시글은 최대 네 개의 태그를 저장할 수 있지만 피드에서는 세 개만 표시하고 나머지를
`+1`로 축약한다. 일반 태그는 44px 링크 영역 안쪽에 28px 높이의 시각 칩을 그린다.
`+1` 항목은 늘어난 44px 행 전체에 배경을 직접 채워 혼자 더 크고 네모나게 보인다.
`+1`은 버튼처럼 보이지만 실제로는 상호작용하지 않는 목록 항목이다.

## 제품 원칙

- 하나의 시각 속성에는 하나의 주 의미만 부여한다.
- 실제 공고 수, 동시 등장 수, 분야만 사용하고 합성 중요도 점수는 만들지 않는다.
- 색만으로 의미를 전달하지 않는다.
- 선택과 호버는 정보를 드러내는 상태이며 장식용 모션을 추가하지 않는다.
- 서버와 클라이언트의 첫 콘텐츠 순서를 같게 유지한다.
- 데이터 실패 시 가짜 글, 가짜 공고, 가짜 관계를 넣지 않는다.
- 저사양 장치를 추측하는 휴리스틱 대신 모든 장치에서 유한한 작업만 수행한다.

## 스킬맵 시각 설계

### 기본 상태

기술 노드는 단색 원을 유지하고 다음 의미를 사용한다.

| 시각 요소 | 의미 |
| --- | --- |
| 노드 색 | 기술의 주 직무 분야 |
| 노드 크기 | 해당 기술을 언급한 공개 공고 수 |
| 기술 간 선 굵기 | 두 기술이 같은 공고에 함께 등장한 횟수 |
| 기술 간 선 투명도 | 정규화된 관계 점수 |
| 작은 중립색 노드 | 관계의 근거가 된 공개 공고 |
| 선택 노드의 얇은 후광 | 현재 분석 대상 |

노드 기본값은 기존의 제한된 제곱근 보정을 유지한다.

```text
nodeValue = clamp(3.2 + sqrt(max(0, demand_count)) * 1.15, 3.2, 10)
```

중심 기술과 선택 기술도 같은 계산을 사용한다. 선택 여부가 노드 크기를 바꾸지 않으며,
선택 상태는 얇은 외곽선과 후광으로만 표시한다. 평상시 기술 노드에는 강한 그림자를
사용하지 않는다.

기술 간 선은 `cooccurrence_count`의 제곱근 보정값을 현재 표시 범위 안에서 제한해
굵기를 계산한다. 극단값 하나가 나머지 선을 모두 보이지 않게 만들지 않도록 최소·최대
폭을 둔다. `score`는 선의 투명도와 연결 정렬에 유지한다.

### 호버와 선택 상태

호버와 선택은 같은 관계 강조 모델을 사용한다. 터치 환경에서는 탭으로 선택한 상태가
데스크톱 호버 상태를 대신한다.

- 대상 노드와 직접 연결된 노드 및 선만 선명하게 유지한다.
- 관련 없는 노드와 선은 읽을 수는 있지만 시선 경쟁을 하지 않도록 낮은 투명도로 둔다.
- 관련 노드는 해당 대상과의 `cooccurrence_count`가 클수록 최대 22%까지만 커진다.
- 관련 선은 같은 동시 등장 수에 따라 굵어지며, 별도 발광 효과는 추가하지 않는다.
- 상태 전환은 단일 redraw로 반영하고 force simulation을 다시 시작하지 않는다.

관계 강조 비율은 현재 대상의 직접 연결 중 가장 큰 동시 등장 수를 기준으로 제곱근
정규화한다. 데이터 값은 바뀌지 않으며 시각 차이만 제한적으로 만든다.

```text
relationRatio = sqrt(cooccurrence_count / maxFocusedCooccurrence)
relatedNodeScale = 1 + relationRatio * 0.22
```

### 범례와 툴팁

그래프에는 짧은 상시 범례를 둔다.

```text
색: 분야  ·  크기: 언급 공고  ·  선: 함께 등장
```

데스크톱에서는 그래프의 조작 버튼과 겹치지 않는 상단 모서리에 배치한다. 모바일에서는
캔버스를 가리지 않도록 그래프 아래 상태 행에 배치한다. 분야별 실제 색은 기존 분야
필터와 분야 분포 목록에서 확인한다.

기술 툴팁은 자연스러운 한국어로 다음 정보를 표시한다.

```text
Kubernetes · 클라우드 · 언급 공고 39건 · 직접 연결 5개
```

다른 기술을 호버하거나 선택한 상태라면 `선택 기술과 함께 7건`을 추가한다. 공고 근거
노드는 회사명과 공고명을 표시한다. 모바일 선택 분석 패널에서도 언급 공고 수와 직접
연결 수를 읽을 수 있어야 한다.

### 시각 품질 제한

- 노드에 단계별 테두리, 배지, 중첩 링을 추가하지 않는다.
- 분야색의 채도 차이로 수요를 다시 표현하지 않는다.
- 선택 노드를 제외한 노드에는 큰 glow를 사용하지 않는다.
- 관계 수에 따라 계속 움직이거나 박동하는 애니메이션을 사용하지 않는다.
- 라벨이 노드와 겹치거나 과도하게 늘어나지 않도록 기존 zoom 임계값을 유지한다.

## 그래프 성능 설계

### 유한한 배치

- 데이터나 필터가 바뀔 때만 force layout을 실행한다.
- 초기 배치는 제한된 warmup과 최대 tick 또는 최대 시간 중 먼저 도달하는 시점에 멈춘다.
- 배치가 끝나면 `autoPauseRedraw(true)` 상태로 캔버스 redraw를 중단한다.
- `다시 배치`는 명시적인 사용자 요청으로만 짧은 유한 배치를 다시 실행한다.
- 호버, 선택, 툴팁 표시, 범례 표시 때문에 force simulation을 재가열하지 않는다.
- 선택 후 카메라 이동은 기존 transform tween만 사용하고 물리 배치와 분리한다.

정확한 tick 수는 구현 단계에서 실패 성능 테스트를 먼저 만들고 30개 기본 그래프와
대형 fixture의 안정 상태를 비교해 최소값으로 정한다. 어떤 값이 선택되더라도 무한
cooldown은 허용하지 않는다.

### 관계 조회 비용

그래프 데이터가 바뀔 때 다음 정보를 한 번 계산한다.

- 노드별 직접 이웃 ID
- 노드별 연결 edge ID
- 각 이웃과의 `cooccurrence_count`
- 직접 연결 수와 대상별 최대 동시 등장 수

호버와 선택은 이 adjacency index만 읽는다. 매 pointer 이동마다 전체 edge 배열을
순회하지 않는다. React state는 연속 포인터 좌표를 저장하지 않으며 캔버스에 필요한
강조 대상 ID만 ref로 유지한다.

### 생명주기와 저사양 대응

- `document.visibilityState`가 `hidden`이면 렌더링과 simulation을 중지한다.
- reduced motion에서는 초기 위치 계산 후 움직이는 전환 없이 바로 안정 상태를 그린다.
- 일반 모드도 안정 후 CPU 사용은 reduced motion과 같은 정적 상태에 가까워야 한다.
- 정적 fallback 그래프는 동적 import가 끝나기 전부터 기술 선택 기능을 제공한다.
- Canvas 2D 렌더러를 유지한다. 이번 범위에서 Sigma/WebGL 전환이나 새 그래프 엔진은
  도입하지 않는다.
- 평상시 노드 shadow blur를 제거하거나 최소화하고 선택·호버 상태에만 제한한다.

### 성능 검증 기준

기존 artifact 변환 benchmark와 별도로 실제 브라우저 렌더링을 측정한다.

- 기본 30개 기술 fixture와 확대된 fixture를 각각 측정한다.
- 4배 CPU 제한에서 3초 pan·zoom 평균 frame interval은 22ms 이하를 목표로 한다.
- 입력 중 200ms 이상의 long task가 없어야 한다.
- 선택 및 이웃 강조는 100ms 이내에 시각적으로 반영되어야 한다.
- 6배 CPU 제한에서 배치 종료 후 4초간 TaskDuration은 800ms 이하로 제한한다.
- 기본 모드와 reduced motion 모두 배치 종료 후 simulation이 멈췄음을 검증한다.
- 그래프 route의 초기 정적 fallback이 캔버스 준비 전에도 보이고 선택 가능해야 한다.

## 홈 피드 첫 렌더 설계

### 서버 데이터 흐름

홈 서버 페이지는 공개 커뮤니티 첫 페이지를 기존 네 가지 데이터와 병렬로 요청한다.

```text
Home server
  ├─ postings
  ├─ skill stats
  ├─ skill graph
  ├─ fit analysis
  └─ public community first page
          ↓
      HomeFeed initial props
          ↓ hydration
      viewer-specific state only
```

서버용 Supabase client와 기존 `createSupabaseCommunityStore`를 재사용한다. 첫 페이지의
글과 cursor를 직렬화 가능한 초기 상태로 `HomeFeed`와 `useCommunityFeed`에 전달한다.
홈은 이미 동적 렌더링이므로 요청마다 현재 공개 글을 가져온다.

서버와 클라이언트 사이에는 store 인스턴스를 넘기지 않고 다음과 같은 판별 가능한
초기 상태만 전달한다.

```text
{ status: "ready", page: { posts, cursor } }
{ status: "error", message }
```

초기 상태가 `ready`이면 hook은 해당 페이지를 자신의 첫 상태로 사용한다. `error`이면
공고와 시장 정보는 그대로 유지하고 커뮤니티 재시도만 클라이언트 store로 수행한다.

### hydration 규칙

- 서버와 첫 클라이언트 렌더는 같은 커뮤니티 글 배열을 사용한다.
- 추천 탭의 첫 순서는 처음부터 `커뮤니티 글 → 공고 → 시장 정보`로 고정한다.
- 초기 공개 페이지가 준비된 경우 mount 직후 같은 목록을 중복 요청하지 않는다.
- 로그인 사용자는 hydration 뒤 공감, 저장, 팔로우 membership만 추가로 가져온다.
- 팔로잉, 인기, 최신 탭 전환과 페이지네이션은 기존 클라이언트 store를 사용한다.
- 글 작성·삭제 후에는 현재 hook 상태와 cursor 계약을 유지한다.

공개 글 요청이 실패하면 공고와 시장 카드는 그대로 서버에서 렌더하고 커뮤니티 영역에
짧은 오류와 재시도 동작을 제공한다. 실패를 숨기거나 샘플 글로 채우지 않는다.

### 응답성과 안정성

커뮤니티 요청은 기존 데이터 요청과 병렬로 실행한다. 서버 응답 시간이 늘어나는지
production 브라우저 측정에 홈 TTFB와 첫 콘텐츠 시점을 포함한다. 첫 화면을 빠르게
보이게 하려고 순서가 바뀌는 불완전한 피드를 먼저 보여주지 않는다.

## 태그 표시 설계

게시글이 저장할 수 있는 태그는 최대 네 개이므로 피드에서도 네 개를 모두 표시한다.

- `slice(0, 3)`과 `+N` 축약 항목을 제거한다.
- 각 태그는 기존 검색 링크와 44px 상호작용 영역을 유지한다.
- 시각 칩 높이는 모두 동일하게 유지한다.
- 네 번째 태그는 필요한 경우 다음 줄로 자연스럽게 감긴다.
- 긴 태그는 피드 폭을 넘지 않도록 한 줄 말줄임과 최대 폭을 적용한다.
- 태그의 전체 텍스트와 검색 목적은 접근성 이름에서 유지한다.

비동작 `+1`을 실제 확장 버튼으로 바꾸는 방식은 최대 한 개만 숨기는 현재 데이터
계약에 비해 상호작용 비용이 크므로 사용하지 않는다.

## 오류 처리

- 그래프 데이터가 비면 기존 빈 상태와 필터 초기화 동작을 유지한다.
- 그래프 엔진 초기화 실패 시 정적 fallback과 DOM 분석 목록은 계속 사용할 수 있다.
- 서버 커뮤니티 요청 실패와 공고 API 실패는 서로 독립적으로 표시한다.
- 초기 서버 커뮤니티 상태가 실패해도 클라이언트 재시도는 가능해야 한다.
- 음수이거나 유한하지 않은 count가 들어와도 시각 크기 계산에는 0 미만의 값을 쓰지
  않는다. 데이터 계약 위반을 정상적인 수치처럼 표시하지 않고 테스트와 오류 경계에서
  드러낸다.

## 접근성

- 색은 분야 구분에만 사용하고 범례 텍스트와 선택 분석을 함께 제공한다.
- 공고 수와 관계 수는 툴팁뿐 아니라 선택 분석 및 관련 기술 목록에서도 읽을 수 있다.
- 모바일 탭과 데스크톱 호버가 같은 관계 강조 결과를 만든다.
- 기존 44px 터치 영역, 확대·이동 버튼, reduced motion, 포커스 표시를 유지한다.
- 범례는 캔버스를 가리지 않으며 스크린리더가 한 번만 읽을 수 있는 설명을 가진다.
- 서버와 클라이언트 렌더 순서가 같아 포커스 대상이 hydration 중 이동하지 않게 한다.

## 구현 경계

구현은 다음 단위로 분리한다.

1. `skill-graph-view`: 노드 크기, edge 원시 count, adjacency와 표시용 파생값
2. `skill-graph-force-canvas`: 유한 force, 단일 redraw 강조, 생명주기 정지
3. `skill-graph-experience`: 범례, 한국어 툴팁 데이터, 선택 분석의 연결 수
4. 홈 server loader: 공개 커뮤니티 첫 페이지를 안전하게 초기 props로 변환
5. `useCommunityFeed`: 초기 페이지 hydration과 viewer membership 후속 로드
6. `SocialCard`: 네 개 태그의 동일한 표시

각 단위는 입력과 출력이 명확해야 하며 그래프 계산, 캔버스 생명주기, 서버 데이터
조회, 카드 표현을 한 파일에 섞지 않는다.

구현은 `태그 표시 → 홈 SSR 초기 상태 → 그래프 파생값과 의미 → 캔버스 생명주기 →
브라우저 성능 검증` 순으로 진행한다. 앞의 두 영역은 그래프 변경과 독립된 커밋으로
검증할 수 있고, 그래프는 데이터 파생값이 확정된 뒤 렌더러를 바꿔 회귀 원인을 좁힌다.

## 테스트 전략

### 단위 및 컴포넌트 테스트

- 중심 기술도 실제 `demand_count`로 크기가 정해지는 실패 테스트
- `cooccurrence_count`가 큰 edge가 더 굵은 표시값을 갖는 실패 테스트
- adjacency index가 직접 연결 수와 대상별 동시 등장 수를 정확히 계산하는 테스트
- 범례와 한국어 툴팁 문자열 테스트
- 선택 분석에서 직접 연결 수를 확인하는 테스트
- 태그 네 개가 모두 검색 링크로 보이고 `+1`이 없는 테스트
- 초기 커뮤니티 페이지가 첫 렌더부터 글을 포함하는 테스트
- 초기 페이지가 있으면 공개 글 목록을 즉시 중복 요청하지 않는 hook 테스트
- 인증 완료 후 viewer membership만 병합되고 글 순서는 유지되는 테스트
- 커뮤니티 서버 실패 시 공고 카드가 유지되는 테스트

### 브라우저 테스트

- 자바스크립트를 끈 첫 HTML에서도 실제 커뮤니티 글이 공고보다 먼저 보인다.
- `domcontentloaded` 직후와 hydration 완료 후 피드 카드 제목 순서가 같다.
- 1440px와 390px에서 네 개 태그가 넘치지 않고 `+1`이 없다.
- 1440px, 820px, 390px에서 범례가 조작 버튼과 하단 내비게이션을 가리지 않는다.
- 마우스 호버와 모바일 탭에서 같은 강한 관계가 가장 선명하게 보인다.
- 4배·6배 CPU 제한 성능 기준과 long task 기준을 검증한다.
- reduced motion에서 그래프가 정적으로 표시되고 선택·확대·이동은 계속 동작한다.

모든 동작 변경은 테스트가 먼저 실패하는 것을 확인한 뒤 최소 구현으로 통과시킨다.

## 제외 범위

- 추천 기술 또는 적합도 알고리즘 변경
- 채용공고 수집기와 그래프 API 응답 계약 변경
- 데이터베이스 schema 또는 RLS 정책 변경
- 분야 분류 체계와 색상 팔레트 전면 교체
- Sigma/WebGL 전환 또는 5만 노드 렌더러 구현
- 홈 전체 정보 구조와 내비게이션 재설계
- 태그 입력 형식과 최대 네 개 제한 변경
- 임의의 종합 중요도, 합격 가능성, 시장 점수 추가

## 완료 조건

1. 사용자가 범례만 보고 색, 크기, 선의 의미를 설명할 수 있다.
2. 중심 기술이 실제 공고 수보다 과장되어 보이지 않는다.
3. 강한 동시 등장 관계는 호버와 선택 상태에서 장식 없이 분명히 구분된다.
4. 그래프 배치가 끝난 뒤 force와 redraw가 무한히 실행되지 않는다.
5. 저사양 성능 예산과 기존 데스크톱·모바일 상호작용 테스트를 통과한다.
6. 홈 첫 HTML부터 커뮤니티 글, 공고, 시장 정보 순서가 안정적이다.
7. hydration 전후 피드 카드가 갑자기 위아래로 이동하지 않는다.
8. 최대 네 개 태그가 모두 동일한 형태로 보이고 `+1` 축약 표시가 없다.
9. API 또는 Supabase 일부 실패 시 성공한 실제 데이터와 복구 동작이 유지된다.
