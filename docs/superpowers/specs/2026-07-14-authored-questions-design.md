# 브라우저 소유 `내 질문` 보관함 설계

## 문제

사용자는 홈에서 커뮤니티 글을 작성할 수 있지만, 저장 버튼을 누르지 않은 자신의 글을 다시 찾는 전용 경로가 없다. handoff의 왼쪽 빠른 메뉴에는 `내 질문`이 요구되어 있으므로, 작성부터 재방문·상세 확인·삭제까지 이어지는 정직한 브라우저 소유 흐름이 필요하다.

## 범위

- `/career/questions`에 이 브라우저에서 작성한 글만 표시한다.
- 서버 계정이나 동기화가 있는 것처럼 표현하지 않는다.
- 홈 왼쪽 레일과 사용자 메뉴에서 보관함으로 이동할 수 있게 한다.
- 새 글 작성, 상세 보기, 삭제를 제공한다.
- 삭제할 때 글에 연결된 로컬 반응과 최근 본 주제도 함께 정리한다.
- 다른 탭에서 글을 만들거나 삭제한 경우 목록을 갱신한다.

공고·기술·시장 API 계약은 변경하지 않는다. 커뮤니티 글은 사용자가 허용한 브라우저 로컬 데이터만 사용한다.

## 정보 구조

페이지 상단은 `내 커리어 / 브라우저 커뮤니티` 맥락, `내 질문` 제목, 로컬 저장 안내, `새 질문 작성` CTA로 구성한다. 목록은 최신 작성 순이며 각 항목에 다음을 표시한다.

- 작성 시각
- 제목과 본문 요약
- 태그
- 브라우저에 저장된 공감·댓글·저장 상태
- `상세 보기` 링크
- 2단계 삭제 확인

게시물이 없으면 사실에 맞는 빈 상태와 `첫 질문 작성` 링크를 보여준다. 초기 hydration 전에는 빈 상태를 단정하지 않고 로딩 상태를 사용한다.

## 삭제 계약

첫 번째 삭제 클릭은 확인 상태만 연다. 사용자는 취소하거나 `정말 삭제`를 선택할 수 있다. 실제 삭제는 기존 `deleteLocalCommunityPost`를 통해 글과 소셜 상호작용을 원자적으로 정리하고, 성공한 경우에만 `removeRecentCommunityTopic`을 호출한다. 저장 실패 시 글을 화면에서 제거하지 않고 오류 안내를 제공한다.

## 접근성·반응형

- 목록은 각 글 제목으로 이름 붙인 `article`을 사용한다.
- 삭제 확인은 해당 글 안에서 명시적 텍스트와 버튼으로 제공한다.
- 모든 행동은 44px 이상 터치 영역과 `focus-visible` 상태를 가진다.
- 모바일에서는 메타와 행동을 세로 배치하고 가로 넘침을 만들지 않는다.
- 삭제 성공·실패는 `aria-live` 상태로 알린다.

## 검증

- 단위/컴포넌트 테스트: 로딩, 빈 상태, 최신순 목록, 동기화, 반응 수, 2단계 삭제, 실패 보존.
- 페이지 테스트: 메타데이터와 보관함 렌더링.
- 브라우저 테스트: 홈에서 작성 → 내 질문 이동 → 새로고침 유지 → 상세 → 삭제 → 빈 상태, 모바일 터치 영역·가로 넘침·콘솔 오류.
- 전체 Vitest, TypeScript, audit, production build, 전체 Playwright 회귀.
