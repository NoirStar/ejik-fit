# 이직핏 브랜드 대시보드 리디자인 설계

## 목표

현재 홈과 기술 맵은 기능 구조는 잡혀 있지만, 브랜드 표현과 전체 화면 완성도가 아직 약하다. 이번 작업은 레퍼런스처럼 한눈에 읽히는 SaaS 대시보드 구조를 유지하되, 제품 이름과 메시지를 `이직핏` 중심으로 정리한다.

핵심 목표는 다음이다.

1. `기술 채용 인텔리전스`라는 임시 제품명을 `이직핏`으로 교체한다.
2. `/brandkit` 방향의 로고 시스템을 만들어 favicon, 사이드바 브랜드, 헤더, 앱 아이콘에 적용한다.
3. 홈은 공고 중심 대시보드로 유지하되, 첫 화면에는 기술 적합도 그래프와 다음 액션만 선명하게 남긴다.
4. `/skills/graph`는 어두운 그래프 캔버스가 주인공이 되는 `기술 맵` 화면으로 고급스럽게 다듬는다.
5. 라이트 모드와 다크 모드를 모두 지원하며, 두 모드 모두 같은 정보 구조와 브랜드 품질을 유지한다.
6. 실제 브라우저에서 확인한 뒤 UX가 어색한 부분을 재수정하는 루프를 반드시 거친다.
7. 모바일에서도 홈은 공고와 인사이트 우선, 기술 맵은 검색과 선택 주변 그래프 우선으로 읽히게 한다.

## 브랜드 방향

브랜드명은 `이직핏`이다. 영문 표기는 `EJIK FIT`을 사용한다.

추천 태그라인:

```txt
내 기술이 맞는 시장을 찾다
```

보조 문구:

```txt
기술 스택과 채용 신호를 연결해 다음 이직 판단을 돕습니다.
```

삭제 또는 교체할 문구:

- `기술 채용 인텔리전스`
- `Tech Hiring Intelligence`
- 임시 `ejik` 텍스트 로고 단독 표기

브랜드 성격은 다음처럼 잡는다.

- 데이터 기반이지만 차갑지 않다.
- 이직 준비자의 실전 판단을 돕는다.
- 채용 정보와 기술 스택을 하나의 fit 판단으로 연결한다.
- 과한 AI 느낌보다 정돈된 커리어 도구처럼 보인다.

## 로고와 favicon

`/brandkit` 생성 방향은 다음이다.

심볼 의미:

- 연결된 스킬 노드
- 다음 회사로 이동하는 경로
- fit을 확인하는 체크 또는 맞물림

형태 원칙:

- 작은 favicon에서도 읽히는 단순한 기하형
- 파란색을 주 accent로 사용하되, 보조로 녹색 또는 청록색 fit 신호를 작게 사용
- 글자 `ㅇ` 또는 `fit`을 직접적으로 우겨 넣지 않고, 이동 경로와 맞물림을 추상화한다.

필요 산출물:

- `apps/web/public/brand/ejikfit-mark.svg`
- `apps/web/public/brand/ejikfit-logo.svg`
- `apps/web/src/app/icon.svg` 또는 Next app icon 대응 파일
- `apps/web/src/app/apple-icon.svg` 또는 PNG 변환이 쉬운 원본 SVG

적용 위치:

- 브라우저 favicon
- DashboardShell 사이드바 브랜드
- SiteHeader 브랜드
- 빈 상태 또는 오류 상태의 작은 브랜드 마크

## 홈 대시보드 설계

홈은 실제 서비스 첫 화면이다. 랜딩 페이지나 디자인 컨셉 보드가 아니다.

상단 구조:

```txt
좌측 브랜드 사이드바
상단 검색과 필터
내 기술스택 기준 이번 주 요약 배너
신규 매칭, 80% 이상 Fit, 상승 기술, 마감 임박 KPI 4개
시장 변화 라인 차트와 이번 주 신규 공고 테이블
급상승 관련 기술 TOP 5와 변경 / 마감 임박 공고 테이블
```

첫 화면에서 보여야 하는 질문:

1. 오늘 볼 만한 맞춤 공고가 있는가
2. 내 스택과 관련된 시장 변화가 오르고 있는가
3. 이번 주 새로 뜬 공고는 무엇인가
4. 어떤 기술이 상승하고 어떤 공고가 마감 임박인가

홈은 첨부 레퍼런스처럼 주간 대시보드로 구성한다. `기술 채용 인텔리전스` 표기는 사용하지 않고 `이직핏` 브랜드만 남긴다. 중앙에는 `내 기술스택 기준 이번 주 요약`, 4개 KPI, 시장 변화 라인 차트, 신규 공고 테이블, 급상승 기술 TOP 5, 변경 / 마감 임박 공고를 배치한다. 기술 관계를 더 깊게 보려면 `/skills/graph`로 이동한다.

## 기술 맵 화면 설계

`/skills/graph`는 레퍼런스의 어두운 그래프 패널처럼 강한 시각 중심을 가진다.

구조:

```txt
좌측 브랜드 내비게이션
상단 검색과 선택 스택
중앙 어두운 graph canvas
좌측 또는 상단 접이식 그래프 설정
우측 선택 노드 인사이트
하단 범례와 선택된 공고 근거
```

기본 모드:

- `선택 주변`
- 기본 animation은 과하지 않게 낮춘다.
- 그래프 설정은 접이식으로 숨긴다.
- 사용자가 노드를 누르면 주변 노드, 관련 공고, 다음 준비가 같이 바뀐다.

그래프 시각:

- 배경은 deep navy 계열
- 노드는 domain별 색을 유지하지만 채도는 절제한다.
- 중심 노드와 owned skill은 더 강하게 표시한다.
- hover 또는 selected 상태에서는 주변 edge만 밝히고 나머지는 낮춘다.
- 라벨은 zoom과 중요도에 따라 제한한다.

## 시각 시스템

이직핏은 라이트 모드와 다크 모드를 모두 가진다. 라이트 모드는 기본 업무 대시보드처럼 깨끗하고 읽기 쉬워야 하며, 다크 모드는 레퍼런스의 두 번째 이미지처럼 graph canvas와 데이터 패널이 선명한 야간 분석 화면처럼 보여야 한다.

색상:

라이트 모드:

- Base: `#f6f8fc`, `#ffffff`
- Text: `#101827`
- Muted: `#66758e`
- Accent blue: `#1769ff`
- Fit green: `#12a875`
- Line: `#e5ebf4`
- Graph navy: `#07101c`
- Graph line: `rgba(125, 164, 255, 0.22)`

다크 모드:

- Base: `#050912`, `#07101c`
- Surface: `#0b1422`
- Text: `#f3f7ff`
- Muted: `#a4afc1`
- Accent blue: `#72a2ff`
- Fit green: `#37d099`
- Line: `rgba(183, 199, 229, 0.14)`
- Graph line: `rgba(125, 164, 255, 0.28)`

테마 동작:

- 기본은 시스템 설정인 `prefers-color-scheme`을 따른다.
- 라이트와 다크 모두 같은 브랜드 마크와 내비게이션 구조를 사용한다.
- 다크 모드는 단순 색 반전이 아니라 표면, 경계선, 그래프 glow, 칩 대비를 별도로 조정한다.
- 라이트 모드에서도 기술 맵 그래프 캔버스는 deep navy를 유지해 그래프가 화면의 중심 신호가 되게 한다.

타이포그래피:

- 기존 Geist를 유지한다.
- 숫자와 점수는 tabular number를 적용한다.
- H1은 브랜드명과 화면 목적을 짧게 보여준다.
- 카드 내부 제목은 과하게 키우지 않는다.

표면:

- 라이트 홈은 밝고 조용한 운영 UI
- 다크 홈은 deep navy 기반의 분석 대시보드 UI
- 그래프 패널은 두 테마 모두 어두운 고밀도 시각 영역
- 카드 radius는 8px에서 18px 사이로 제한한다.
- 그림자는 약하게 쓰고, graph canvas에서만 빛 효과를 허용한다.

## 검증 기준

정적 검증:

```txt
npm run lint
npm test -- --run
npm run build
git diff --check
```

브라우저 검증:

- `/` 첫 화면에서 `이직핏`과 최근 맞춤 공고가 보인다.
- `/skills/graph`에서 어두운 그래프 캔버스가 실제로 보인다.
- 320px, 768px, 1024px, 1440px에서 텍스트가 버튼이나 카드 밖으로 넘치지 않는다.
- 라이트 모드와 다크 모드에서 홈 대시보드, 사이드바, 기술 맵이 모두 읽힌다.
- 다크 그래프 패널과 각 테마의 홈 UI가 서로 충돌하지 않는다.
- favicon이 브라우저 탭에 적용된다.

UX 재수정 루프:

1. dev server를 띄운다.
2. 실제 브라우저에서 `/`와 `/skills/graph`를 연다.
3. 라이트 모드와 다크 모드를 각각 확인한다.
4. 320px, 768px, 1024px, 1440px viewport를 확인한다.
5. Playwright 또는 브라우저 스크린샷으로 레이아웃, graph canvas, 텍스트 overflow, 클릭 가능한 요소를 확인한다.
6. 발견한 UX 문제를 즉시 수정한다.
7. 수정 후 같은 화면을 다시 확인한다.
8. 최종 응답에는 수정 전후 확인한 항목과 남은 리스크를 적는다.
