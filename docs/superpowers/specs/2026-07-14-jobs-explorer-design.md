# 공고 탐색 개편 설계

## 목표

`/jobs`를 단순 행 목록에서 공식 원문과 추출 근거를 중심으로 한 공고 탐색 화면으로 개편한다. 검색 결과, 저장 상태, 내 기술과 겹치는 근거는 서로 다른 출처임을 명확히 보여주며 합격 가능성이나 임의 점수는 만들지 않는다.

## 제품 원칙

- 공고·기술·검증 시각은 실제 API 데이터만 사용한다.
- 커뮤니티 fixture나 추천 점수는 공고 화면에 섞지 않는다.
- `total`은 전체 시장 규모가 아니라 현재 API 응답 개수이므로 "현재 결과"로 표현한다.
- 마감일이 없는 공고를 마감 임박으로 분류하지 않는다.
- 저장한 공고와 내 기술은 브라우저에만 저장된 상태임을 표시한다.
- 기업 로고는 검증된 로컬 자산만 사용하고, 나머지는 일관된 이니셜 마크로 대체한다.

## 실제 데이터 계약

목록 응답에 다음 필드를 추가한다.

- `opens_at`, `closes_at`
- `required_skills`
- `preferred_skills`
- `unspecified_skills`

모든 기술 배열은 확정 임계값 이상의 추출 결과만 포함한다. 데이터베이스 목록 조회는 기술 관계를 한 번에 로드하며, 검색 인덱스 문서에도 같은 필드를 저장한다. 이전 검색 문서처럼 필드가 없는 응답은 빈 배열과 `null`로 안전하게 처리한다.

## 정보 구조

### 도입부

- 작은 출처 레이블
- `공고 탐색` 제목
- 공식 채용페이지, 검증 시각, 확정 기술 추출이라는 범위 설명
- 현재 응답 공고 수, 기업 수, 가장 최근 확인일을 한 줄 근거로 표시

### 검색 조건

- 기술·직무·기업 통합 검색
- 전체·신입·경력·신입경력 조건
- 검색과 초기화
- 공식 원문 재확인 안내 및 데이터 정책 링크

### 결과 보기

- `전체 공고`: 현재 API 결과
- `내 기술 겹침`: 브라우저에 저장한 기술과 추출 기술이 하나 이상 겹치는 결과
- `저장한 공고`: 브라우저에 저장한 ID가 현재 결과에 포함된 공고

보기 전환은 클라이언트 상태이며 검색 조건은 URL query가 기준이다. 내 기술이 없을 때는 빈 결과로 속이지 않고 `/career`에서 기술을 추가하도록 안내한다.

### 공고 항목

- 기업 마크, 기업명, 제목
- 경력 범위, 고용 형태, 근무지
- 실제 접수 종료일이 있을 때만 마감일
- 내 기술과 겹치는 이름
- 확정된 필수·우대·그 외 언급 기술
- 마지막 검증일, 상세 분석 링크, 공식 원문 링크
- 브라우저 저장 토글

## 상태

- API 오류: 내부 URL이나 상태 메시지를 노출하지 않는 안전한 재시도 상태
- 검색 결과 없음: 검색어와 경력 조건 조정 안내
- 내 기술 없음: 기술 저장 안내
- 기술 겹침 없음: 현재 불러온 결과 범위라는 점을 명시
- 저장 없음: 저장 버튼의 위치와 브라우저 저장 범위 안내
- 추출 기술 없음: 공고 사실 정보와 공식 원문은 계속 제공

## 반응형·접근성

- 1100px 이상: 17rem 검색 열 + 유연한 결과 열
- 1100px 미만: 검색 조건을 결과 위로 이동
- 680px 미만: 카드 사실과 하단 행동을 단일 열로 재배치
- 모든 저장·탭·링크는 최소 44px 목표 영역
- 필터는 label, 보기 전환은 접근 가능한 이름과 `aria-pressed`, 결과 변화는 live region 제공
- 올바른 제목 계층과 정의 목록을 유지하고, 색만으로 필수·우대·일치 상태를 구분하지 않는다.

## 검증 기준

- 목록 API가 확정 기술과 접수일 계약을 반환한다.
- 저장 상태가 새로고침 뒤 유지되고 오류 난 저장소에서도 화면이 중단되지 않는다.
- 내 스택 같은 탭 변경을 즉시 반영한다.
- 필터 query와 상세·공식·스킬맵 링크가 정확하다.
- 오류·빈 결과·내 기술 없음·저장 없음이 구분된다.
- 1440px, 820px, 390px에서 가로 넘침과 겹침이 없다.
- 전체 backend/web 테스트, TypeScript, Playwright, production build가 통과한다.
