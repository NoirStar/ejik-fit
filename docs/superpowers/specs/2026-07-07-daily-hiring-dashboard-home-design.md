# ejik Daily Hiring Dashboard Home 설계

## 목적

ejik의 첫 화면은 그래프 체험 페이지가 아니라 매일 확인하는 채용 브리핑이어야 한다.

사용자가 홈에서 해결해야 하는 첫 질문은 다음이다.

1. 오늘 새로 볼 만한 맞춤 공고가 있는가
2. 내 스택과 잘 맞는 공고는 무엇인가
3. 놓치면 안 되는 마감 임박 공고가 있는가
4. 최근 시장 변화 중 내가 알아야 할 신호가 있는가

그래프, 채용 캘린더, 트렌드 상세 분석은 중요하지만 홈의 주인공이 아니다. 홈은 빠르게 공고를 고르고 행동으로 이어지는 화면이어야 한다.

## 확정된 제품 방향

- A안: Daily Intelligence Dashboard 중심으로 간다.
- 홈 1순위는 공고다.
- 홈 기본 정렬은 `최신 공고 중 Fit 높은 것 우선`이다.
- 마감 임박 공고는 메인 정렬에 억지로 섞지 않고 별도 강조 영역에서 보여준다.
- 맞춤 공고가 있으면 맞춤 공고만 홈에 보여준다.
- 맞춤 공고가 부족하거나 없으면 전체 신규 공고를 보충하되, 그 상태를 사용자에게 명확히 알린다.
- 홈 공고 row는 하이브리드형으로 간다.
  - 데스크톱: 촘촘한 row
  - 모바일: 카드형
- 홈 row에는 부족 기술을 기본 노출하지 않는다.
- 부족 기술, 추천 이유, 원문 근거는 공고 선택 후 우측 상세 패널에서 보여준다.

## 최상위 정보 구조

```txt
대시보드
공고 탐색
트렌드
기술 맵
채용 캘린더
내 스택
```

각 탭의 역할은 다음과 같다.

- 대시보드: 매일 확인하는 맞춤 공고 브리핑
- 공고 탐색: 전체 공고 검색, 필터, 저장
- 트렌드: 기술, 직무, 기업 시장 신호 분석
- 기술 맵: Obsidian식 관계 그래프
- 채용 캘린더: 마감일, 저장 공고, 지원 일정
- 내 스택: 보유 기술, 관심 직무, 관심 기업, 지역 설정

## 홈 화면 레이아웃

홈은 많은 정보를 전부 펼치는 화면이 아니라 필요한 정보만 요약해서 들어오게 하는 화면이다.

```txt
전체 앱 셸
├─ 좌측 내비게이션
├─ 상단 검색과 필터
├─ 상단 요약 카드
│  ├─ 새 맞춤 공고
│  ├─ 높은 Fit 공고
│  ├─ 마감 임박
│  └─ 오늘 확인할 항목
├─ 메인 공고 리스트
│  └─ 최근 맞춤 공고
├─ 우측 패널
│  ├─ 기본: 오늘 확인할 항목
│  └─ 선택 시: 공고 상세 분석
└─ 하단 미니 신호
   ├─ 급상승 기술 3개
   └─ 내 스택과 자주 같이 나오는 기술 3개
```

## 홈 공고 리스트

홈의 메인 리스트는 `최근 맞춤 공고`다.

기본 노출 규칙:

```txt
1. 내 스택, 관심 직무, 관심 기업, 지역 기준으로 맞춤 공고 계산
2. 최근 등록 공고 중 Fit 높은 것 우선 노출
3. 맞춤 공고가 부족하면 전체 신규 공고로 보충
4. 마감 임박은 우측 패널과 요약 카드에서 별도 강조
```

홈 노출 점수는 다음 개념으로 계산한다.

```txt
홈 노출 점수 =
최신성 40%
내 스택 Fit 40%
필수 기술 매칭 15%
관심 기업 또는 관심 직무 5%
```

정확한 가중치는 이후 데이터 검증으로 조정할 수 있지만, 홈의 제품 의도는 이 비율을 따른다.

## 공고 row 정보량

홈 row는 빠르게 훑기 위한 정보만 가진다.

기본 노출:

```txt
회사 / 직무
Fit 점수
핵심 매칭 스킬 2-3개
등록일 또는 마감 상태
```

예시:

```txt
토스 Backend Engineer
Fit 84% · 오늘 등록 · D-12
Java · Spring · Kubernetes
```

기본 row에서 숨기는 정보:

- 부족 기술
- 추천 이유 전문
- 원문 근거
- 전체 요구 기술 목록
- 그래프 관계

이 정보는 공고 선택 후 우측 패널에서만 보여준다.

## 우측 패널

우측 패널은 빈 공간으로 두지 않는다.

아무 공고도 선택하지 않은 상태:

```txt
오늘 확인할 항목
마감 임박
저장한 공고
```

공고를 선택한 상태:

```txt
공고 상세 분석
왜 추천됨
맞는 기술
부족 기술
저장하기
캘린더 추가
원문 보기
```

우측 패널은 홈의 정보량을 줄이면서도, 필요한 사람은 바로 깊게 볼 수 있게 하는 역할이다.

## 그래프와 캘린더 분리

현재 구현은 그래프가 홈의 중심에 가깝다. 다음 구현에서는 역할을 분리한다.

- 홈 대시보드: 공고 브리핑 중심
- 기술 맵 탭: Obsidian식 그래프 중심
- 채용 캘린더 탭: 마감과 지원 일정 중심

홈에는 큰 그래프를 넣지 않는다. 그래프가 필요하면 `기술 맵` 탭으로 이동한다.

홈에는 큰 캘린더도 넣지 않는다. 마감 임박 요약만 보여주고, 전체 일정은 `채용 캘린더` 탭으로 이동한다.

## 빈 상태와 보충 상태

맞춤 공고가 있는 경우:

```txt
최근 맞춤 공고
```

맞춤 공고가 부족한 경우:

```txt
맞춤 공고가 아직 적어서 전체 신규 공고를 함께 보여줍니다.
```

맞춤 조건이 없는 경우:

```txt
내 스택을 입력하면 맞춤 공고가 정렬됩니다.
```

이 메시지는 홈의 정체성을 유지하기 위한 장치다. 사용자가 전체 공고 화면과 맞춤 홈을 혼동하지 않게 한다.

## 구현 시 삭제 또는 분리할 코드

다음 코드는 홈 구현에서 필요 없거나 역할이 바뀐다.

삭제 대상:

- 사용되지 않는 랜딩 페이지 컴포넌트
- 사용되지 않는 랜딩 모션 코드
- `.landing-*`, `.hero-*`, `.graph-product`, `.market-dashboard` 등 과거 실험 스타일 중 실제 참조가 없는 글로벌 CSS
- 홈에서 그래프를 주인공으로 만드는 레이아웃 코드

분리 대상:

- 현재 `SkillGraphExperience`는 `/skills/graph` 전용으로 남긴다.
- 홈은 별도 `DailyDashboardHome` 계열 컴포넌트로 만든다.
- 공고 row, 요약 카드, 우측 인스펙터, 미니 트렌드 신호를 작은 컴포넌트로 나눈다.

보존 대상:

- 그래프 데이터 API
- Fit 분석 API
- 보유 스킬 저장 로직
- 공고 상세 페이지
- 공고 스킬 evidence 렌더링
- Obsidian식 그래프 엔진 자체

## 컴포넌트 구조 제안

```txt
components/dashboard/
├─ app-shell.tsx
├─ daily-dashboard-home.tsx
├─ daily-summary-strip.tsx
├─ recent-fit-job-list.tsx
├─ fit-job-row.tsx
├─ job-inspector-panel.tsx
├─ deadline-brief.tsx
└─ mini-market-signals.tsx
```

그래프 화면은 기존 위치를 유지하되 이름과 역할을 명확히 한다.

```txt
components/skill-graph-experience.tsx
```

## 테스트 기준

홈 테스트:

- `/`가 랜딩 문구가 아니라 Daily Dashboard를 렌더링한다.
- 최근 맞춤 공고 영역이 보인다.
- 홈 row에 부족 기술이 기본 노출되지 않는다.
- 공고 선택 시 우측 패널에 부족 기술이 표시된다.
- 맞춤 공고가 없을 때 보충 메시지가 표시된다.

그래프 테스트:

- `/skills/graph`는 기존 Obsidian식 그래프 화면을 유지한다.
- 그래프 화면은 홈 대시보드용 공고 리스트를 주인공으로 만들지 않는다.

정적 검사:

- 사용되지 않는 랜딩 컴포넌트가 남아 있지 않다.
- 새 홈 TSX에 랜딩 카피가 남아 있지 않다.
- 불필요한 글로벌 CSS 클래스가 제거되어도 빌드가 통과한다.

## 이번 구현의 비목표

- 완성형 채용 캘린더 전체 구현
- 그래프 엔진을 WebGL로 교체
- 전체 공고 검색 페이지를 완성형으로 재설계
- 랜딩 페이지 복원
- 인증, 저장 동기화, 알림 구현

이번 범위는 홈을 공고 중심 Daily Dashboard로 재정의하고, 그래프와 캘린더를 별도 탭 역할로 분리하는 것이다.
