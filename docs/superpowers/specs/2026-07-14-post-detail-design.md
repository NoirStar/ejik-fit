# 이직핏 커뮤니티 글 상세 디자인

## 목표

홈 피드의 mock 커뮤니티 글과 면접 후기를 실제 읽기 흐름이 있는 상세 화면으로 연결한다. 커뮤니티 데이터는 예시임을 분명히 표시하고, 공식 공고나 시장 수치를 이 화면에서 임의 생성하지 않는다.

## 제품 원칙

- 페이지 상단과 댓글 영역에서 `커뮤니티 예시 콘텐츠`임을 명확히 표시한다.
- 실제 사용자, 실제 기업 후기처럼 보이는 표현은 쓰지 않는다.
- 홈과 상세의 공감·저장·브라우저 작성 댓글은 같은 localStorage 계약을 사용한다.
- 저장소 접근 실패 시 성공 상태를 표시하지 않는다.
- 모든 사용자 입력은 React text node로만 렌더하고 HTML을 주입하지 않는다.
- mock 원문 반응 수와 이 브라우저에서 추가한 반응을 구분해 합산한다.

## 정보 구조

1. 피드로 돌아가기
2. 글 유형, 작성 시각, 예시 데이터 표시
3. 작성자, 제목, 본문, 태그
4. 공감·댓글·저장 행동
5. 면접 후기인 경우 기업 유형·직무·단계 메타데이터
6. 대표 예시 댓글과 이 브라우저에서 작성한 댓글
7. 댓글 작성 폼
8. 같은 주제의 다른 예시 글

## 상태 계약

`ejik-fit:social-interactions` 한 키에 다음 상태를 저장한다.

- `reactedPostIds: string[]`
- `savedPostIds: string[]`
- `commentsByPostId: Record<string, LocalPostComment[]>`

댓글은 id, body, createdAt만 저장한다. 작성자는 항상 `나`, 출처는 `local`로 UI에서 결정한다. 읽을 때 타입과 길이를 검증하고 잘못된 값은 버린다. 게시물 id와 댓글 본문 길이를 제한해 비정상적인 저장값이 화면과 메모리를 과도하게 점유하지 않게 한다.

같은 탭에서는 custom event, 다른 탭에서는 storage event로 동기화한다.

## 화면 방향

- 기존 이직핏 글로벌 헤더와 모바일 하단 내비게이션을 그대로 사용한다.
- 최대 1160px의 2열 읽기 레이아웃을 사용한다.
- 본문 열은 약 720px, 보조 열은 약 280px로 제한한다.
- 900px 이하에서는 단일 열로 전환하고 보조 콘텐츠를 본문 뒤에 둔다.
- 680px 이하에서는 페이지 여백과 제목 크기를 줄이고 모든 주요 버튼을 최소 44px로 유지한다.
- 흰 표면, 중립 회색 배경, 얇은 경계, 현재 브랜드 보라색 한 가지를 사용한다.
- 자동 애니메이션은 추가하지 않고 hover, focus, active 피드백만 제공한다.

## 오류와 빈 상태

- 존재하지 않는 mock 글 id는 not-found로 보낸다.
- 저장소가 차단되면 기존 화면 상태를 유지한다.
- 빈 댓글은 제출하지 않고 인라인 오류를 표시한다.
- 600자를 넘는 댓글은 제출하지 않고 길이 오류를 표시한다.
- 예시 댓글이 없는 경우에도 로컬 댓글 작성 방법을 안내한다.

## 검증

- 저장 상태 정규화, 손상 JSON, 저장 실패, 같은 탭 및 탭 간 이벤트를 단위 테스트한다.
- 상세 컴포넌트에서 공감·저장·댓글 추가와 새로고침 유지 여부를 테스트한다.
- 커뮤니티 글과 면접 후기 두 타입의 서버 렌더링을 테스트한다.
- 1440px와 390px에서 가로 넘침, 44px 타깃, 모바일 내비게이션 여백을 Playwright로 확인한다.
