# 공고 탐색 구현 계획

## Task 1: 실제 목록 근거 확장

1. 실패 테스트로 목록 응답의 접수일과 필수·우대·그 외 기술 계약을 정의한다.
2. 확정 기술만 그룹화하는 목록 serializer와 eager loading을 구현한다.
3. 검색 인덱스 문서와 Python/TypeScript 응답 타입을 같은 계약으로 맞춘다.
4. 백엔드 집중 테스트와 전체 회귀 테스트를 실행한다.

## Task 2: 브라우저 저장 모델

1. 저장 ID 정규화, 읽기, 쓰기, 토글, 저장소 오류 경계를 실패 테스트로 정의한다.
2. 같은 탭과 다른 탭에서 동기화되는 작은 저장 모듈을 구현한다.
3. 직접 저장 상태와 서버 사실이 섞이지 않도록 타입을 분리한다.

## Task 3: 공고 탐색 모델과 UI

1. 내 기술 겹침, 보기별 필터링, 경력 범위, 날짜 표시를 순수 함수 테스트로 정의한다.
2. URL 기반 검색 조건을 유지하면서 `JobList`를 상호작용 가능한 탐색 화면으로 재구성한다.
3. 검증된 `CompanyMark`, 실제 기술 태그, 저장 토글, 상세·공식 링크를 연결한다.
4. 오류·검색 없음·기술 없음·겹침 없음·저장 없음 상태를 구현한다.

## Task 4: 반응형·검증·배포

1. 1440px, 820px, 390px 레이아웃과 44px 상호작용 영역을 브라우저에서 확인한다.
2. 실제 운영 API를 연결해 기술 근거, 필터, 저장, 내 스택 동기화를 검사한다.
3. 독립 코드 리뷰에서 Critical/Important를 수정한다.
4. 전체 테스트, TypeScript, Playwright, migration SQL, audit, Vercel production build를 실행한다.
5. main에 fast-forward 병합하고 푸시한다.
6. GitHub CI와 Vercel 웹·API 배포 후 운영 `/jobs`를 재검증한다.
